**Status:** research note, no design commitments. An observation matching an external language's type-system release against Rigor.
**Date:** 2026-06-04.
**Rigor version:** observed against the working tree (master @ `4eccc390`).
**Subject:** the type system of Elixir v1.20.0 (released 2026-06-03). The primary source is
[Elixir v1.20.0 released](https://elixir-lang.org/blog/2026/06/03/elixir-v1-20-0-released/),
with background in [Strong arrows: a new approach to gradual typing](https://elixir-lang.org/blog/2023/09/20/strong-arrows-gradual-typing/),
[Gradual set-theoretic types (HexDocs)](https://hexdocs.pm/elixir/main/gradual-set-theoretic-types.html), and
[The Design Principles of the Elixir Type System (Castagna, Duboc et al., arXiv:2306.06391)](https://arxiv.org/pdf/2306.06391).

**Why:** Elixir has been maturing toward running annotation-free, gradual set-theoretic types "over the whole program on every compile." Its goals (gradual, low FP, inference-first) are nearly isomorphic to Rigor's, yet its handling of soundness is the exact opposite — so holding the two up against each other lets us take stock of (a) confirmation that Rigor's direction aligns with the industry frontier, (b) improvements we could adopt without breaking the FP discipline, and (c) the difficulties they share. Subsequent improvement reviews / ADR filings will use this note as their basis.

**Reading order.** §1 is the survey of Elixir's surface, §2 maps the commonalities, §3 the points where the design decisions diverge, §4 the improvements Rigor might adopt, §5 the shared difficulties and the judgement on whether an ADR is needed.

## 1. Survey of the Elixir v1.20 surface

1. **Complete gradual typing without annotations.** Inference plus checking run over the whole program without type annotations written in the source. For backward compatibility it behaves "as if every argument were annotated with `dynamic()`." v1.18 completed pattern inference; v1.20 completed **full inference of patterns plus guards**.

2. **Set-theoretic types as the foundation.** Types are described and composed with the set operations union / intersection / negation (the design of Giuseppe Castagna, Guillaume Duboc, et al.).

3. **The `dynamic()` type — not `any()`.**
   - *Compatibility*: a violation is reported **only when the accepted type and the supplied type are disjoint (their intersection is empty)**. This prevents false positives in existing code.
   - *Narrowing*: `dynamic()` is narrowed along execution. Accessing `data.a` and `data.b` narrows `data` to `%{..., a: number(), b: number()}`.

4. **Guard / pattern-match analysis.**
   - `when is_list(x) and is_integer(y)` → infers union/negation
   - `def example({:ok, x} = y) when is_binary(x) or is_integer(x)` → `x` is binary|integer, `y` is a 2-element tuple
   - `when is_map_key(x, :foo)` → `%{..., foo: dynamic()}`, the negative side `foo: not_set()` (**retaining key absence as a positive fact**)
   - `when tuple_size(x) < 3` → tracks an upper bound on tuple length
   - narrows the type in each `case` clause; after a `nil ->` clause is passed, `value` is judged `binary()`. **Redundant clauses (i.e. dead code) are reported too.**

5. **Soundness via strong arrows.** The runtime type tests the VM performs are woven into the static types, "keeping gradual typing sound without adding runtime checks." It reports **only "verified bugs" that will certainly fail at runtime if executed**, with an extremely low false-positive rate.

6. **Deliberately deferred items.** Introducing type-signature (annotation) syntax is conditional on the following: (a) if v1.20's performance is satisfactory, (b) if recursive types can be implemented efficiently, (c) if parametric types can be implemented efficiently, (d) if the typing problem of traversing a map's key-value pairs as an enumerable can be solved.

## 2. Commonalities with Rigor

| Aspect | Elixir v1.20 | Rigor |
| --- | --- | --- |
| Thorough false-positive avoidance | "verified bug = only what certainly fails at runtime," "extremely low FP rate" | Top value "do not threaten working code" (`feedback_false_positive_discipline`) |
| Lenient dynamic type | `dynamic()`, a violation only when disjoint | `Dynamic[T]` algebra + gradual consistency + trinary certainty ([value-lattice.md](../../type-specification/value-lattice/), [relations-and-certainty.md](../../type-specification/relations-and-certainty/)) |
| Set-theoretic operations | union / intersection / negation as the foundation | type operators `~T` (negation) / `T - U` (difference) / value lattice ([type-operators.md](../../type-specification/type-operators/)) |
| Control-flow narrowing | `nil` elimination in `case` clauses, narrowing derived from guards | edge-aware narrowing / fact store ([control-flow-analysis.md](../../type-specification/control-flow-analysis/)) |
| Structural key tracking | tracks key presence/absence via `is_map_key` | inferred object shapes / structural interfaces / capability roles ([structural-interfaces-and-object-shapes.md](../../type-specification/structural-interfaces-and-object-shapes/)) |
| Inference-first over annotation | infers with zero annotations, annotations come later | `rigor sig-gen`-first, the philosophy of raising inference precision to make hand-written RBS unnecessary (`feedback_no_ai_generated_rbs`) |

Both share a strikingly close philosophy: "gradual, minimal false positives, catching what breaks at runtime with minimal friction." Rigor's direction is not an idiosyncratic path but aligns with an independent arrival point at the industry frontier.

## 3. Differing design decisions

1. **The stance on soundness (the biggest difference).**
   Elixir actively pursues "sound" with strong arrows, and by construction limits violations to verified bugs.
   Rigor is explicitly **unsound**, suppressing FPs with the heuristics of the robustness principle (the asymmetry of strict return values / lenient arguments, [robustness-principle.md](../../type-specification/robustness-principle/)) plus trinary certainty.
   **The goal is the same (low FP) but the mechanisms run in opposite directions** — Elixir achieves it with a theoretical guarantee, Rigor with certainty ranks plus budgets.

2. **Degree of embedding in the language foundation.**
   Elixir is compiler-integrated, with inference running on every compile, and as yet no annotation syntax.
   Rigor is an external checker for Ruby + RBS, with **an RBS superset plus extended annotations already complete**, extending frameworks via plugins. Their footings are contrasting.

3. **Use of runtime tests.**
   Elixir's strong arrows incorporate the BEAM VM's runtime type tests into the static types. Ruby/Rigor has no equivalent VM mechanism, so a direct port is impossible.

4. **The distinction between records and dictionaries.**
   Elixir distinguishes atom-keyed maps (record-like) from dictionaries at the type level. Rigor does the equivalent with object shapes plus capability roles, but the axis of the split differs.

## 4. Improvements Rigor might adopt

1. **Synthesizing structural shapes from use sites (demand-driven shape accretion).**
   Elixir observes the accesses `data.a` / `data.b` and narrows `dynamic()` to `%{a:, b:}`.
   Rigor's capability roles are close, but there is room to push the use-site-driven narrowing one step further — "accumulating the required structure from the members actually accessed in a method body." Standing up structure without annotations fits the sig-gen philosophy.

2. **Diagnosing dead code / unreachable clauses.**
   Elixir reports redundant clauses as dead code from exhaustive `case` narrowing.
   Rigor's fact store holds narrowing information, so there is a possibility of emitting "unreachable `when`/`case`/`if` branches" as a **high-signal, low-FP diagnostic class** — a kind of warning that is consistent with the FP discipline.
   (Connect this with the existing `20260603-dead-code-coverage-sweep.md`.)

3. **Retaining key absence as a positive fact.**
   Carrying **negative key facts** in narrowing, as in Elixir's `not is_map_key(x, :foo)` → `foo: not_set()`, would refine Hash-access narrowing (the branch after a `key?` guard).

4. **Tracking refinements of collection length / tuple length.**
   Elixir tracks `tuple_size(x) < 3` as a fact. Rigor already has refinements like `non-empty-string` / `positive-int` ([imported-built-in-types.md](../../type-specification/imported-built-in-types/)), so incorporating **array length / non-emptiness** as facts into narrowing is a natural extension.

5. **"Strong" typing of post-guard branches.**
   Rigor already narrows with `is_a?`, but if the strong-arrows idea — treating the branch after `is_a?` / `respond_to?` / `case ===` is passed as a strong type with `Dynamic` dropped — were carried through, it could **grow real-bug detection (reduce FNs) without increasing FPs**. A direction that makes Ruby's runtime guards the basis for static precision.

## 5. Shared difficulties and whether an ADR is needed

The remaining issues Elixir listed as prerequisites for introducing type signatures also overlap with Rigor's difficulties:

- **Efficiency of recursive / parametric types** — directly connected to Rigor's lightweight HKT (ADR-20, `project_lightweight_hkt`). It is worth getting ahead of this together with the performance budgets ([inference-budgets.md](../../type-specification/inference-budgets/), `project_inference_budgets_unwired`).
- **Typing the traversal of a map as an enumerable** — traversal typing of Hash/Enumerable is an unresolved area in Rigor too.

**Judgement on whether an ADR is needed.** This note alone makes no design commitments. The most readily adoptable items are §4-2 (unreachable-clause diagnostics) and §4-3/4-4 (the narrowing extensions for negative facts / length refinements), all of which grow real-bug detection without breaking the FP discipline. §4-2 should be considered for filing as an independent ADR built on the existing dead-code survey note; §4-3/4-4 are better treated as addenda to the control-flow-analysis spec. §3-1 (the soundness stance) conflicts with Rigor's foundational design (the robustness principle), so it will not be followed (recorded only).
