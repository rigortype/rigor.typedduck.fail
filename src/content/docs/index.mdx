Rigor runs `rigor check` over your Ruby code and reports real bugs — undefined methods on typed receivers, wrong argument counts, provable nil dereferences — with no type annotations required in source. Every type is inferred from what your code actually produces. Your team never writes a single annotation.

Point `rigor check` at ordinary Ruby — no annotations, no setup — and it reports real bugs:

```ruby
# lib/demo.rb
"hello".no_such_method        # typo'd method name
[1, 2, 3].rotate(1, 2)        # too many arguments
```

```text
lib/demo.rb:1:9: error: undefined method `no_such_method' for "hello" [call.undefined-method]
lib/demo.rb:2:11: error: wrong number of arguments to `rotate' on Array (given 2, expected 0..1) [call.wrong-arity]
```

Two paths below: one for **getting Rigor into your project**, one for **evaluating how it works and how it compares**.

**Get started in two commands:**

```sh
mise use ruby@4.0
mise use gem:rigortype
```

## Evaluating Rigor

For deciding whether Rigor fits — how it works and how it compares. Where Sorbet wants inline `sig`s and Steep wants hand-written RBS, Rigor wants neither — every type is inferred from the values your code actually produces.

- Coming from another type checker: [Sorbet](/reference/handbook/10-sorbet/), [Steep](/reference/handbook/appendix-steep/), [TypeProf](/reference/handbook/appendix-typeprof/), [TypeScript](/reference/handbook/appendix-typescript/) (mypy, Go, Rust and more in the handbook appendices).
