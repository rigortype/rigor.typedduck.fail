ステータス: 設計議論＋判断、2026-06-17にRigor v0.1.19（`[Unreleased]`）を対象として執筆。Rigorの部分実装されたPrismミューテーション機構が、内部の解析器機能として留まるべきか、それともRSpec／minitestを駆動する外部の型枝刈り型ミューテーションテストフレームワークへと育つべきか、という問いに対する両面評価（最大限拡張的vs. 厳しく批判的）を記録する。[ADR-69](../../adr/69-pluggable-mutation-substrate/)、[ADR-70](../../adr/70-fused-protection-coverage/)、[ADR-71](../../adr/71-type-guided-external-mutation-testing/)へ流れ込む。非規範的であり、拘束力を持つのはADRと仕様である。

根拠: [ADR-62](../../adr/62-mutation-testing-teeth-measurement/)（内部の歯のハーネス）、[ADR-63](../../adr/63-type-protection-coverage/)（プロダクト化されたファイル単位の有効性ティア）、[`20260613-mutation-teeth-harness.md`](../20260613-mutation-teeth-harness/)（生きたハーネストラッカー）、そして出荷済みのコード── `tool/mutation/mutate.rb`（開発専用のsweep／fuzz）、`lib/rigor/protection/mutator.rb`、`lib/rigor/protection/mutation_scanner.rb`。

## 問い

メンテナーが述べた2つの衝動:

1. ミューテーションフレームワークはRigorのスライス（ADR-62の歯のハーネス）として留まるべきか、それとも
2. RSpec／minitestと組み合わさって先進的なミューテーションテストフレームワークへと育つ**外部ライブラリ**になるべきか── Rigorの**クロスファイル依存グラフと型情報を使ってミューテーション空間を賢く枝刈りする**ことで、ミューテーションテストにありがちな「散弾的・遅い・低収率」という評判なしに潜在的な問題を見つけられるように？

## 今日実際に作られているもの（議論の土台）

機構全体は1つの問いのために最適化されている── **ミューテートされたバイトに対してRigorを再実行する── *新しいRigorの診断*が現れるか？** これが重要なのは、メンテナーのアイデアのうちどれだけが再利用可能かを決めるからだ。

- `kill`の定義は**Rigorの診断シグネチャ**（`mutation_scanner.rb`の`sig` = `(rule, path, line, column, message)`）である。テストスイートはループの**どこにも**いない。
- 知性── `Mutator#filter_by_type` ── は、アンカーで**Rigorが具体的な（非`Dynamic`の）型を保持している**箇所、つまりRigorが*噛める*箇所でのみミューテーションを残す。偽陽性（FP）に安全な方向（型が未解決⇒残す）。
- オペレータ（`nil_inject`／`type_swap`／`undefined_method`／`arity_extra`）は、テストスイートが捕まえる「実行可能だが振る舞いが異なる」ミュータントになるようにではなく、**Rigorの診断ルール**を引っ掛けるように設計されている。`undefined_method`は呼び出しを`foo__rigor_absent`にリネームする── 実行時の`NoMethodError`が保証され、それをカバーするテストならどれでも簡単に捕まえるが、情報量は低い（通常アサーションの前にクラッシュする）。

つまりADR-62が測るのは**解析器の偽陰性**（Rigorに歯があるか）である。メンテナーのアイデアが測るのは**テストスイートの偽陰性**（*あなたのテスト*に歯があるか）である。これらは別のプロダクトだ。

## フレーミング: 2つのプロダクト、共有は約20 %

| | **プロダクトA —— 既存のもの** | **プロダクトB —— 提案** |
| --- | --- | --- |
| ミューテーション対象 | 解析対象のコード | ユーザーのコード |
| `kill`オラクル | **Rigorが診断を出す** | **テストスイートがレッドになる** |
| 生存者が意味するのは | **Rigorの偽陰性**（エンジンのギャップ） | **テストギャップ** |
| ミュータントごとに走るもの | 1回のインプロセス再解析 | ユーザーの（関連する）テストスイート全体 |
| ステータス | 開発専用、ADR-50の凍結サーフェスの外 | 外部のエコシステムプロダクト |

唯一共有されるサーフェスは**Prismバイトスプライス機構**（`Mutation#apply`、ASTウォーク）── おおむね20 %である。残りの80 %（テストランナー駆動のキルオラクル、結果の帰属、flaky／timeout／順序／並列／Rails-DBの処理）は新規であり、まさに`mutant`が商用化する前に10年を費やした80 %そのものだ。

## スタンス1 —— 可能性を最大化する（拡張的）

1. **融合した静的×動的の「保護マップ」は真に新規だ**。Stryker／mutantは*テスト*保護を測り、Sorbet／Steepのメトリックは*型*保護を測る。**2つの軸を融合するツールは存在しない**。Rigorはすでに静的な側（ADR-63 Tier 1）を所有している。動的な側（テストがミュータントをキルするか）を重ね合わせれば、*ある行が無保護⟺型もテストもそれを守っていない*が得られ、しかも「型を足す」vs「テストを足す」のうち**より安い**方を指し示せる。
2. **型指向のオペレータは第1位の不満（等価ミュータント）を攻める**。Rigorは各サイトの型を知っているので、**well-typed（だから実行される）**かつ**型境界で証明可能に非等価**な`type_swap`を出せる── それがカバーするクラスにおいて等価ミュータントの無駄がほぼゼロになる。最も古いミューテーションの不満に対する、新鮮で型システム特有の切り口だ。
3. **「まず型チェッカーでキルする」── 漸進的（gradual）な筋書き**。Rigor*自身*が診断するミュータントはスイートの実行を一切必要としない（型 = 第一防衛線、テスト = 第二）。正直で安価なメトリックは*「型チェッカーを通過するミュータントのうち、あなたのスイートが捕まえる割合は？」*になる── 型のネットがすでに保持しているものすべてについて、高価な80 %（スイート実行）をスキップする。
4. **ADR-46によるインクリメンタル／CIミューテーションテスト── 未解決の夢**。ミューテーションテストはコストのせいでCIに乗らなかった。[ADR-46](../../adr/46-incremental-dependency-graph/)の依存グラフがあれば、差分のクロージャ（ΔF ∪ 依存先）だけをミューテートし、影響を受けるテストだけを走らせられる── *「すべてのPRで走らせられるミューテーションテスト」*。
5. **開かれたエコシステムの席＋Rigorが持つ最良のマーケティング**。Rubyのミューテーションテストはサービスが行き届いていない（`mutant`はライセンス制限あり、`mutest`はそのフォーク、どちらも遅い）。型認識でインクリメンタルなRSpec／minitestツールはその席を所有し、同時に旗印にもなりうる── 「Rigorは型をチェックするだけでなく、あなたのテストに歯があることを証明する」。「型誘導ミューテーションテスト」は際立った、出版に近い研究の切り口でもある。

## スタンス2 —— 厳しく見る（批判的）

1. **共有部分は簡単な20 %、難しい80 %は`mutant`の墓場だ**。テストランナーの駆動、結果の帰属、flaky／timeout／順序／並列／Rails-DB ── すべて新規で、すべて`mutant`が商用化した理由である。**契約をフリーズしよう**としている（[ADR-50](../../adr/50-release-engineering-and-stability-strategy/)）プレ1.0のスポンサーウェア型チェッカーが、第二のサポートサーフェスを持つ第二のプロダクトへと注意を分岐させるのは戦略的リスクだ。
2. **テスト*選択*については、静的グラフはランタイムカバレッジに負ける**。「依存グラフで枝刈りする」は問題を読み違えている── *どのテストがこのミュータントに触れるか*を決めるのは、ランタイム行／テストごとのカバレッジが最善であり、それは`mutant`／Stryker／PITがすでに使っている。静的グラフは**過大近似**なので、必要以上に*多くの*テストを走らせる── 速くなるどころか遅くなる。
3. **さらに悪いことに、テスト選択に使われた静的グラフは偽生存者（false survivor）を生む── これは偽陽性（FP）だ**。Rigorの依存グラフが記録するのは**型**の読み取りであって、テストが行使する**ランタイム**コールグラフではない。テストはメタプログラミング、動的ディスパッチ、`send`を叩く── まさにRigorが見られない`Dynamic`領域だ。グラフでテストを選択すると、**ミュータントをキルしたはずのテストをスキップし、存在しない「テストギャップ」を報告する**ことになる。これはミューテーションテストにおける偽陽性の類似物であり── 偽陽性の回避はこのプロジェクトの最上位の価値だ（`feedback_false_positive_discipline`）。選択にグラフを使うことはRigor自身の精神に背く。
4. **型枝刈りは価値が低い箇所で役立ち、価値が高い箇所で盲目だ**。型フィルタが噛むのは**型付き**サイト── そこにはすでに静的なネットがあるので、テストの限界的価値が*最も低い*。テストが最も必要なのは**`Dynamic`**サイト（静的なネットがない）であり、そこではフィルタは枝刈りも非等価の証明もできない。ツールの知性は、テストギャップの価値がない場所にちょうど集中してしまう。
5. **等価ミュータントの検出は型の形をしたわずかな断片しか見えない**。Rigorが等価性を証明するのはその束が捉える狭いクラスだけだ（定数畳み込み、デッドな節、型不変なスワップ）。支配的な等価ミュータントのクラスは**意味的**なもの（決して当たらない境界の`<=`対`<`、可換な並べ替え、マスクされたオフバイワン）── 型には見えない。「型が等価ミュータントを解決する」は角をひとつ削るだけだ。
6. **高価な部分は縮まない**。ミューテーションのコストは*スイートの実行*であって、ミュータント生成ではない。型フィルタは走らせるミュータントを減らす── が（4）のとおり高価値の`Dynamic`サイトのものを取り除き、安い側を最適化して高くつく側を残す。
7. **プロジェクトはすでに熟議し、意図的に内部に留めた**。ADR-62 WD4（凍結サーフェスの外、未出荷）、ADR-63が出荷するのは*狭く厳選した部分集合*だけで、荷重を担う**「常に有効性、決して生の生存ではない」**というフレーミングを伴う── 生の生存は動いているコードを怖がらせるからだ。外部プロダクトはそのリスクを、Rigorの管理の外で、他人のスイートに対して再度走らせることになる。

## 判断（統合）

切り分けると、批判の砲火は**外部の汎用形態**に当たるが、**真に価値のある核**は逃れる。だから二者択一ではなく順序付けだ。

1. **今は外部プロダクトをフォークしない**。80 %のテストランナーの墓場を、v1.0の契約フリーズの直前に引き受けるのはタイミングが悪い。
2. **安く、ミッションに沿い、新規な一手は融合した保護マップ── ADR-63の拡張として**であって、汎用フレームワークとしてではない。`coverage --protection`に「*生存者*に対してスイートを1回走らせ、静的∪動的の保護を重ね合わせる」薄い層を足す。新サーフェスは最小限、誰も出荷していない新しいアーティファクトだ。→ **ADR-70**。
3. **オプション性を安く買う── シームをきれいにする**。オラクルが*「新しいRigorの診断」*（今日）でも*「スイートがレッドになった」*（将来）でも再アーキテクチャなしで切り替えられるよう、**キルオラクル**と**オペレータ**をプラガブルにする。`lib/rigor/protection/mutator.rb`はすでにlibと`tool/mutation/`の間で共有されている。これがシームを仕上げる。→ **ADR-69**。
4. **外部ツールは需要 × ADR-46が現実になることをゲートにし、生き残るウェッジ（wedge、楔）を正直にマーケティングする**。批判的なケースを生き延びるウェッジは**インクリメンタル／CIミューテーションテスト**（すべてのPRで走らせる）であって、**「もっと賢く枝刈りする」**ではない（選択ではカバレッジに負ける）。各レバーは健全な箇所でのみ使う── **ランタイムカバレッジ**がテストを選択し（健全）、**依存グラフ**が差分で*どのファイルをミューテートするか*をスコープし（「何が変わったか」については健全）、**型情報**が生成＋層別レポートを行う（テスト選択には決して使わない── （3）の偽生存者を避ける）。正直な資産は2つ、（a）融合した保護メトリックと（b）差分スコープの実行だ。→ **ADR-71**（評価、需要ゲート付き）。

一行の要点: **「型がミューテーション空間を賢く枝刈りする→普通のミューテーションテストより良い」は過大評価されている**（選択ではカバレッジに負ける、枝刈りは価値が最も低い箇所で噛む）が、**「静的と動的の保護を1つのマップに融合する」と「ミューテーションテストを差分スコープしてCIで実用化する」は本物で新規だ**。最初のものを今Rigorの内部で取り、シームを準備し、残りは需要ゲートにせよ。

## ADRへの分解

- **ADR-69 —— プラガブルなミューテーション基盤（キルオラクル＋オペレータのシーム）**。機械的／アーキテクチャ。これを可能にするリファクタ。今すぐ着手可能。
- **ADR-70 —— 融合した静的∪動的の保護カバレッジ**。熟議的。ADR-63を拡張する。ADR-69の最初の利用者。今すぐ着手可能（薄いスライス）。
- **ADR-71 —— 型誘導の外部インクリメンタルミューテーションテスト**。評価／提案。ウェッジ、却下した過剰な売り込み、需要 × ADR-46のゲートを記録する。先送り。
