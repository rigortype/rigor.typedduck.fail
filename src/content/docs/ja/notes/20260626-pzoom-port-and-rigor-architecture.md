執筆。Matt Brownの記事[*From Psalm to Pzoom*](https://mattbrown.dev/articles/from-psalm-to-pzoom)
を読み、その中心的な発見がRigorの既存の設計判断に何を示唆するかを記録する。
非規範的；ADRと仕様が拘束力を持つ。これは戦略的コンテキストのノートであり、
変更提案ではない。

裏付け（Rigor側）: [ADR-17](../../adr/17-monkey-patch-pre-evaluation/)（`pre_eval:`）、
[ADR-39](../../adr/39-plugin-target-library-invocation/)（プラグインは対象ライブラリの
実際のメソッドを呼び出す）、[ADR-45](../../adr/45-unchanged-project-fast-path/)
（「プラグインは解析中にプロジェクトファイルを読む」というPunditの発見が、解析前
フィンガープリントを不健全にした）、[ADR-52](../../adr/52-compiled-plugin-contribution-dispatch/)
（コンパイル済み貢献ディスパッチ）、そしてパフォーマンスの軌跡である
[ADR-44](../../adr/44-dispatch-allocation-churn/) /

[ADR-46](../../adr/46-incremental-dependency-graph/) /

## 記事の要約（一段落）

Matt Brown（PHP静的解析器Psalmの作者）はPsalmをRustで書き直し、**Pzoom**を生み出した。
彼はこの約10万行のポートを、主にLLM — 歴代のClaudeモデル、最終的にFable 5 — を使い、
Psalmの元のPHPロジック、彼自身の以前のHack向けRustポート（Hakana）、そしてMagoの
Rust製PHPパーサーを素材として、Psalmの5,000＋1,500のテストを等価性オラクルとして
駆動しながら行った。コスト: 約6か月で約100時間、約2,000ドルのトークン代。結果:
**10倍高速、Psalmのテストの99.9%が通過**。しかし彼自身の結論は、**ほとんどの人は
これを採用すべきでない**というものである。理由 — LLMの物語ではなく — こそが
この記事の本題である。

## 核心の発見、そしてそれがRigorに関わる理由

真の主張は「LLMで移植できる」ではない。より高速な解析器が採用されないという
構造的な理由である。

> "since there's no scan-time execution of PHP scripts allowed there are limits to what it
> can do automatically."
>
> "That PHP 'magic' resolves at runtime (which a static pass can't observe)."

その連鎖は次のとおり。

1. 現代のPHPフレームワークは実行時メタプログラミング（マジックメソッド、DSL）に
   大きく依存している。
2. 純粋な静的パスはそれを観測できないため、プロジェクトは**PHPで書かれた
   プラグイン**でそのギャップを埋める。
3. PHPで**書かれている**Psalmは、**scan時にそれらのプラグインを実行できる**。
4. **コンパイル済みRustバイナリ**であるPzoomは**scan時にPHPを実行できない** →
   プラグイン資産が動かない → 10倍高速であることは乗り換えの理由にならない。

これはRigorがまさに直面した分岐点であり、Rigorは意図的に**Psalmと同じ側**を
選んだ。

| ツール | 解析器の実装言語 | scan時の対象言語実行 | プラグイン |
|-------|-------------------|----------------------------------------|--------------------|
| Psalm | PHP               | あり | PHP、実行可能 |
| Pzoom | Rust              | **なし** | 実行不可 |
| Rigor | **Ruby** | **あり** | **Ruby、実行可能** |

Rigorの設計は、まさにこの記事がコンパイル済みツールに欠けていると述べる能力に
依拠している。

- **ADR-17** — プロジェクト側monkey-patchの事前評価（`pre_eval:`）。
- **ADR-39** — プラグインは*実際の*対象ライブラリの安全なメソッドを呼び出す
  （`ActiveSupport::Inflector`を再モデル化するのではなく、実物を実行する）。
- **ADR-45** — Punditプラグインが*解析中にプロジェクトファイルを読む*という
  健全性の発見。これは文字どおりの意味での「scan時実行」であり、解析前
  フィンガープリントを不健全にしたものそのものである。Pzoomの限界全体は、
  まさにこれの不在である。
- **ADR-52** — 実行1回あたり1度コンパイルされるが、それでも候補ヒットに対して
  プラグインコードを呼び出す、実行時プラグイン貢献ディスパッチ。

**論点1。** Pzoomは事実上、「Rigorをコンパイル型言語で書き直したら何を失うか」の
対照実験である。失うのはscan時実行 — つまりプラグインエコシステムである。
この記事は、プラグインの多い動的エコシステムに対しては解析器を*ホスト言語で
実装する*ことが正しい戦略であるという外部の証拠であり、将来の「速度のために
RigorをRust/Crystalで書き直す」提案に対する定番の回答となる。

## 「速いが採用されない」 vs. Rigorのパフォーマンス哲学

> "Unless developers really need the speedup _and_ they're comfortable maintaining it
> themselves, they shouldn't take it on as a dependency."

教訓は、保守負担と実用性がベンチマーク数値に勝るということである。これは
Rigorのパフォーマンス関連ADRを次のように再構成する。

- Rigorはその速度を**実装内最適化**から得ている — アロケーション衛生
  （ADR-44）、記録・検証キャッシュ（ADR-45）、キャッシュのスリム化（ADR-54）、
  進行中のインクリメンタル依存グラフ（ADR-46） — **プラグイン実行を捨てることなく**。
  Pzoomは実行を捨てることで10倍を買った。Rigorはそれを保持したままインクリメンタルに
  改善する。
- この記事の警告は「言語ポートによる一発の10倍は罠である」というものである。
  Rigorは**ポートではなくインクリメンタルな最適化**を選ぶことで、構造的に
  この罠を回避している。ADR-15（Ractor並行性）とADR-46（インクリメンタル）は、
  正しい種類の「次なる大きな速度源」である。それは言語境界を越えるのではなく、
  実装の内側にある。

**論点2。** Rigorの「1つの実装の中で、FP規律を守りながら時間を削る」という
哲学こそが、Pzoomの結末が示す答えである。「プログラムが動く」ことと「プラグインが
動く」ことをベンチマーク数値より優先するという価値の順序は、Rigorの偽陽性の
規律と同じ順序である。

## LLMポート手法をRigorの既存の規律に写像する

Brownのプロセスから再利用できる点が2つある。

- **テストは目標ではなく等価性オラクルとして扱う。** 初期の試みは「テスト通過に
  過剰適合していた」。修正は、テストスイートを満たすべき目標としてではなく
  *等価性の証明*として扱うことである。Rigorはすでにこれのより強い版を行って
  いる。そのリファクタは**実コーパスに対するバイト同一の診断出力**をゲートと
  している（ADR-52 / ADR-53 / ADR-54はすべてMastodon / GitLab / haml / kramdownで
  同一出力を要求する）。それは「テストを等価性オラクルとする」教訓が「実コーパス
  出力の同一性」へと硬化したものである。
- **「LLMで9割まで到達できるが、最後の1割を出荷するにはドメイン専門知識が
  必要」。**

  > "deep expertise is no longer necessary to get 90% of the way there — but it is still
  > necessary to ship."

  これはRigorの繰り返し現れる**「裁定せよ、思い込むな」**プロトコル
  （ADR-57 / ADR-62）と同じ形をしている。発火も生存も機械的に信用しない —
  人間が裁定する。LLMが提案した変更をコーパスと手動プローブ形状に照らして
  裁定するRigorのワークフローは、「出荷に必要な最後の1割の専門知識」の
  制度化された形である。

**論点3。** Rigorの検証規律（バイト同一のコーパスゲート＋裁定プロトコル）は
すでにBrownが経験的に到達した安全弁を織り込んでいる。その帰結として、
**RigorはLLM駆動の内部リファクタを安全に実行するのに適した立場にある** —
例えばADR-53 Track Bのshadow-harnessでゲートされたwalk一本化は、Pzoom手法
（LLM＋等価性オラクル）をRigor内部に適用する理想的な場所であり、過剰適合という
失敗モードを避けるため合否判定を常にコーパス出力の同一性に固定する。

## Rigorへの具体的な持ち帰り

1. **アーキテクチャ上の正当化を文書化する。** 「なぜRigorはRuby実装のままなのか
   （＝scan時に対象言語を実行できるから）」は現在ADR-17/39/45/52に分散して
   いる。Pzoomという公開の反例を得て、これは設計境界（ADR-0近辺）としての
   一行への集約に値する。「コンパイル済みポートはscan時実行＝プラグイン
   エコシステムを失う（Pzoom、2026年）」。これは「なぜRustで書き直さないのか」
   に対する定番の回答になる。
2. **パフォーマンスの語り方。** 高速化を宣伝するときは、「N倍」単独ではなく
   「プラグイン実行を保ったままN倍」と表現する — それがこの記事の教訓が
   示す差別化である。
3. **LLM駆動リファクタをどこに適用するか。** すでにバイト同一ゲートを持つ領域
   （walk一本化、キャッシュ層、カタログのインポート）はPzoom手法を安全に
   取り入れられる。テスト過剰適合を避けるため、合否判定はコーパス出力の
   等価性に固定する。
4. **採用障壁の自己点検。** 速度は採用を保証しない。保守性と実用性が勝る。
   Rigorの採用ドライバーはFP規律＋プラグインエコシステム＋DX（ADR-73
   skills、ADR-74オフラインドキュメント）である。投資はこの順序を保つべき
   である。

## 一行まとめ

Pzoomは、動的言語の解析器をコンパイル型言語へ移植すると速度を得る代わりに
scan時実行 — すなわちプラグインエコシステム — を失い、そのために採用されない
ことを経験的に実証している。それは、Ruby実装のままで強力なプラグイン実行
モデルを維持し、実装内のインクリメンタル最適化を通じて速度を追求するという
Rigorの選択を外部から裏付ける。Brownの「テストを等価性オラクルとする＋
最後の1割は専門家の裁定」というポート手法は、Rigorがすでにバイト同一の
コーパスゲートと「裁定せよ、思い込むな」プロトコルとして実践している規律と
一致する。
