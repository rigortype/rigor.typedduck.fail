**ステータス:** Proposed — 設計。全体実行キャッシュ（[ADR-45](../45-unchanged-project-fast-path/)）は粗粒度（解析済みファイルが1つでも変わると全体を再実行）。このADRはその後継として、ファイル単位のインクリメンタル解析（incremental analysis）を設計する。葉のコントローラーを編集した場合はそのファイルだけを再チェック、モデルを編集した場合はそのモデルと実際に依存しているファイルだけを再チェックする。

ADR-45は*変更なし*のプロジェクトを高速化した（全体実行のキャッシュを記録・検証する手法で、GitLab上では約42倍）。ただし意図的に粗粒度にしている。変更ファイルが1つあればエントリ全体が無効になるため、1ファイルを編集するだけで全体の再スキャンが必要になる（GitLab上では約113秒）。このADRの背景にある直観はまさにユーザーの感覚と一致する。MVCアプリにおいて具体的な`PostsController`は依存グラフ（dependency graph）の**葉**であり、そのアクションを通じて何かが型推論されることはない。したがってその編集はそのファイルだけの再チェックで済むべきだ。一方`Post`モデルを編集した場合は、そのメソッドを呼び出しているサービスやコントローラーのほんの一部だけを再チェックすればよく、2,630ファイル全体を対象にする必要はない。

この設計は、解析済みの各ファイルが*実際に他のファイルから何を読み取ったか*を記録し、それを反転させてdependentsインデックスを構築する。編集時には影響を受ける推移閉包（closure）だけを再解析し、それ以外のすべてのファイルの診断結果はキャッシュから提供する。健全性（soundness、古い診断結果を提供しない）がこの設計の核心課題である。

根拠: [ADR-45](../45-unchanged-project-fast-path/)（全体実行キャッシュと「プラグインが解析中にファイルを読み込む」という問題）および[ADR-24](../24-self-method-call-resolution/)（ファイル横断のself呼び出し解決）。

## あるファイルの診断結果が実際に依存するもの

AのASTと、Aの解析が消費したすべてのファイル横断（cross-file）ファクトを純粋関数として受け取り、Aの診断結果が得られる。エンジンに照らしてファイル横断ファクトの種類を列挙すると次のとおりだ:

1. **クラス・メソッド・定数の宣言**。Aは`Post`、`Post#publish`、`Admin::Report`をプロジェクト全体の`discovered_classes` / `discovered_def_nodes` / `discovered_superclasses` / `discovered_includes`インデックス（`seed_project_scope`によって各ファイルの`Scope`に注入される）に対して解決する。エンジン内の66箇所のファイル横断読み取りは、`user_def_for`、`user_def_site_for`、`superclass_of`、`includes_of`、`top_level_def_for`、及びdiscovered-classルックアップという少数の`Scope`アクセサーに集約される。これがこの設計が計装する**チョークポイント（choke point）**だ。
2. **推論された戻り値のボディ（深いエッジ）**。`infer_user_method_return`は`Post#publish`の`DefNode`を解決し、呼び出しの戻り型を推論するために**そのボディを評価する**。したがってAは`Post#publish`の存在だけでなく、その**ボディ**にも依存する。推論された戻り型を変える本体編集は、Aの診断結果を変える。
3. **クラス状態の推論**。プロジェクト全体の`class_ivars` / `class_cvars` / `program_globals` / `in_source_constants`は、あるファイルで代入され別のファイルで読み取られる型を持つ。
4. **プラグインの寄与**。プラグインは解析の途中でプロジェクトファイルを読み込み（`rigor-pundit`ポリシー）、診断結果や戻り型を提供することがある。各プラグインの`io_boundary.cache_descriptor`（ADR-45）を通じてファイルダイジェストとして既に捕捉されている。
5. **グローバル入力**。RBS環境（gems + `sig/`）、解決済みの設定、`Rigor::VERSION` — ADR-45のフィンガープリント（fingerprint）。

主要な種類についてはソースの帰属（attribution）が既に存在する。`discovered_def_sources`は`(class, method) → "path:line"`をマッピングするため、メソッド宣言や推論戻り値の依存関係は現時点でもファイルに帰属させられる。クラス・スーパークラス・includeの帰属はまだ記録されていないが、追加コストは低い。プリパスでは各ファイルをマージしながら`path`を保持しており（`merge_discovered_defs`はすでにメソッドに対してこれを行っている）。

## 設計

### 1. アクセサーのチョークポイントでファイル単位の依存関係を記録する

軽量な**依存関係レコーダー**を`analyze_file(A)`に通す。Aの解析が`Scope`アクセサーを通じてファイル横断シンボルを解決するたびに、レコーダーは解決されたエントリの**ソースファイル**（`discovered_def_sources`および今後追加するクラス・スーパークラス・includeのソースマップ経由）を記録する。また**ネガティブ**ルックアップ — Aが問い合わせたが解決*しなかった*シンボル — も記録する。後でそのシンボルが追加された場合にAを再チェックしなければならないためだ（§4参照）。

結果として`deps[A] = { Aが宣言またはボディを読み取ったファイル }`、Aのダイジェスト、グローバルフィンガープリントが得られる。プラグインのファイル読み取り（種類4）はio_boundaryのディスクリプターから折り込まれる。

### 2. dependentsインデックスへの反転

`dependents[X] = { A : X ∈ deps[A] }`。ファイル単位の診断キャッシュと並べて永続化する。これがインクリメンタルステップが辿る逆辺である。

### 3. ファイル単位の診断キャッシュ

`(A.digest, {dep.digest for dep in deps[A]}, global fingerprint)`をキーに**各ファイル**の診断結果をキャッシュする（全体実行ではなく） — ADR-45の記録・検証機構（`fetch_or_validate` + `Descriptor#fresh?`）をファイル単位に適用する。AとAが依存しているすべてのファイルが変更されていなければAはキャッシュヒットとなる。

### 4. インクリメンタルステップ: 宣言構造ティアとボディティア

実行時に、前回実行で記録されたファイルダイジェストとプロジェクトを差分比較し、変更セットΔFを求める。そして:

- **宣言構造フィンガープリント**。プロジェクトの*宣言のシェイプ* — すべてのclass / module / def / constの*名前* + そのソースファイル + 各クラスのスーパークラス・include名 — をダイジェストする。ただしメソッドのボディは含まない。ボディ編集ではこれは変わらず、クラス・メソッド・constの追加・削除・リネーム・移動では変わる。
  - **フィンガープリント不変（よくある開発編集 — メソッドボディ）:**シンボルの生成・削除がないため、ネガティブ依存関係は安定している。`ΔF ∪ ⋃_{X∈ΔF} dependents[X]`（変更ファイルとその推移的dependents）だけを正確に再解析し、その他すべてのファイルはファイル単位キャッシュから提供する。
  - **フィンガープリント変化（構造的編集）:**シンボルが出現・消滅・移動した。保守的に範囲を広げる: ΔF ∪ そのdependents ∪ 宣言デルタ内の名前に対して**ネガティブ**依存関係を持つすべてのファイル（`Post#publish`を参照して見つからなかったが今なら見つかるファイル）を再解析する。最初のカットでは構造的変更が起きると全体再解析にフォールバック（fallback）する場合もある。ネガティブ依存関係の絞り込みは後のスライスで行う。

### 具体例

- **`PostsController#create`のボディを編集**。宣言フィンガープリント不変。`dependents[posts_controller]`は空（コントローラーアクションを通じて型推論するファイルはなく、ルートはrails-routesプラグイン経由で*文字列*名として参照し、`routes.rb`を読み込むためコントローラーの戻り型ではない）。**1ファイル**を再解析。✔「葉」の直観。
- **`Post#publish`のボディを編集**（12個のサービス・コントローラーがPostを使用）。宣言フィンガープリント不変。`Post` + `dependents[post]` = **約13ファイル**を再解析。2,630ファイルではない。✔「関連ファイルのみ」の直観。
- **`ApplicationController`を編集**（すべてのコントローラーの基底クラス）。サブクラスはそれに依存しているため（継承されたdef・推論戻り値）、`dependents`はコントローラーツリー全体 — 正しく再チェックされる。
- **`Post#archive`メソッドを追加**。宣言フィンガープリント変化 → 構造的ティア: `Post` + そのdependents + `archive`をネガティブにルックアップしたファイルを再解析する。

## 健全性 — 難しい部分

問題となる唯一の失敗モード: **依存関係の記録漏れ**があると古い診断結果が提供される。型チェッカーにおいては意図せず偽陽性・偽陰性を生み出すことになる — このプロジェクトが速度のために妥協しない最上位の規律だ。防御策:

- **1つのチョークポイント、監査済み**。ファイル横断の読み取りは少数の`Scope`アクセサーを通じて行われる。66箇所のコールサイトではなく*それらのアクセサー*を計装することで、宣言・ボディ・状態の辺を構造的に捕捉できる。アクセサーの集合は完全性の監査に十分なほど小さく、ファイル横断アクセサーが記録なしで追加された場合に失敗するテストでガードできる。
- **精度より保守的なフォールバック**。不確実性があれば再解析の範囲は広がるのみで、狭まることはない。構造的変更、読み取りを記述できないプラグイン、キャッシュスキーマのバンプ、設定・RBS・バージョン変更 → 全体再解析。過剰解析は遅いが健全であり、不足解析は非健全だ。
- **`--verify-incremental`クロスチェック**。インクリメンタル解析と`--no-cache`の全体解析を両方実行し、バイト単位で同一の診断結果であることを確認するモード（およびCIジョブ）。これがインクリメンタルコンパイラが信頼を得る方法であり、潜在的な健全性バグを、サイレントな古い結果ではなく大きなテスト失敗へと変換する。すべてのスライスの受け入れゲートだ。
- **ADR-45の不変条件を引き継ぐ**。記録されたファイルを再ダイジェストして検証し、検証ゲートは`--no-cache`で実行し、キャッシュエラーが実行を壊すことはない。

## ステージング

1. **帰属 + 記録**。クラス・スーパークラス・includeのソースマップを追加し、`Scope`アクセサーに依存関係レコーダーを通し、`deps` / `dependents`とファイル単位の診断エントリを永続化する。`--verify-incremental`がMastodon + GitLabでグリーンの状態で、デフォルトオフのフラグの後ろにランディングする。
   - **スライス1aランディング済み** — `Analysis::DependencyRecorder`（スレッドローカルアキュムレーター + 無効化時のファストパスが単純な整数読み取りになるようモジュールレベルの有効化カウント、計装済みアクセサーはディスパッチのホットパス上にあるため）は、ファイルごとに解析がメソッド・ボディを読み取ったソースファイルと、未解決（ネガティブ）のクロスクラスメソッドルックアップを記録する。`Scope#user_def_for` — メソッド解決・`infer_user_method_return`のチョークポイント — が計装され、既存の`discovered_def_sources`を通じて帰属する。`Runner.new(record_dependencies: true)`でオプトイン、`runner.file_dependencies`として公開。デフォルトはオフ（診断結果はバイト同一、`make verify`グリーン）。
   - **スライス1bランディング済み** —— 祖先エッジ。ファイルごとの`class_sources`マップ（`class_name → Set<宣言ファイル>`、ある名前に`def` / スーパークラス / `include` / 素の宣言を寄与するすべてのファイルから`accumulate_project_index`で構築）が記録スコープにシードされ、`Scope#superclass_of` / `#includes_of`が祖先エッジを解決するときに宣言ファイルの完全な集合を記録する —— つまり複数ファイルにまたがって再オープンされたクラスは、その祖先を読む消費者を*すべての*再オープンに依存させる（設計上の過剰記録、保守的な方向）。これを修正する過程で潜在的なスライス1aのギャップが表面化した: ADR-44の単一アロケーションボディスコープ（`ExpressionTyper#build_user_method_body_scope`、`StatementEvaluator#build_fresh_body_scope`）が`discovered_def_sources`を落としていたため、メソッドボディエッジ（`user_def_for` → `def_sources`）が暗黙的selfの祖先解決呼び出しに対してサイレントに*記録されていなかった*（スライス1aのスペックは明示的なレシーバーのみをカバーしていた）。両ボディスコープは現在`def_sources` + 新しい`class_sources`を持つ;デフォルトはオフ、`make verify`グリーン、診断結果はバイト同一。このスライスの残り: 永続化（`deps` / `dependents` + ファイル単位の診断エントリ）と`--verify-incremental`。
2. **ボディティア**。宣言フィンガープリント不変のパス（ΔF ∪ dependentsを再解析）を繋ぎ込む。これだけでMVCの恩恵が得られる。
3. **構造的ティア**。ネガティブ依存関係の追跡により、シンボルの追加時に全体へフォールバックするのではなく、そのシンボルの候補リゾルバーだけを再チェックするようにする。
4. **シンボル粒度（オプション）**。ファイルレベルの依存関係を`(file, symbol)`に精緻化し、1つのモデルメソッドを編集したときにモデルの全呼び出し元ではなく、*そのメソッド*の呼び出し元だけを再チェックするようにする。

## 却下・延期された代替案

- **全体実行キャッシュのみ（ADR-45）**。粗粒度の前身。このADRはその置き換えではなく改良版であり、変更なしのケースでは引き続きADR-45が1回のルックアップで勝る。
- **純粋なタイムスタンプ・mtime無効化**。ファイル横断辺にわたって非健全（mtimeが変わらないボディ編集、またはディスク上で触れられていないdependentを見逃す）であり、推論戻り値の依存関係も無視する。
- **Salsaスタイルのファクトレベルメモ化**（個々のファクトの値を記録し、ファクト単位で再検証）。精密な理想だが、クエリレイヤーを*エンジン全体*（66箇所すべての読み取り点 + ディスパッチャーティア）に通す必要があり、はるかに大規模な書き直しになる。アクセサーチョークポイントでのファイルレベル依存関係は、少ないサーフェスでほとんどの恩恵を得られる。シンボル粒度（ステージ4）がそこへの漸進的なステップだ。
- **宣言ティアなしで、全体の逆到達可能セットを編集ごとに再解析**。健全ではあるが、推論戻り値を変えないボディ編集でdependentsを不必要に再チェックする。宣言ティアと（後の）戻り型サマリーがそれを剪定する。

## リスク

- **記録の完全性は正確さの不変条件であり、パフォーマンスのノブではない**。`--verify-incremental`ゲートは必須であり、オプションではない。
- **推論戻り値の揮発性**。メソッドの推論戻り値を変えるボディ編集はすべての呼び出し元に波及する。深いコールグラフは「影響を受けるクロージャー」を大きくしうる。戻り型**サマリー**（依存先の推論サマリーが変わった場合のみdependentを再チェックする）がこれを抑えるが、それ自体が独立したスライスだ。
- **永続化コスト**。`deps` / `dependents` / ファイル単位エントリはプロジェクトとともに肥大する。これらはADR-6の退避なしバックエンドを共有しており、サイズに関する方針が必要だ（ファイル単位エントリは安定したパスをキーとしているため、ADR-45のスロットと同様に蓄積ではなく上書きになる）。
