ステータス: **Proposed、2026-06-05**。Rigorをv0.1.xの機能開発ラインからエンタープライズ用途に耐える安定製品へ移行させる戦略を記録する。**v0.2.0はリリースエンジニアリングの試行マイルストーン**（契約フリーズではない）、**v1.0.0がハードな契約フリーズ**となる。判断軸はエンタープライズ採用であり、**リリースライン内でのBreaking Changeなし・パフォーマンスリグレッションなし**を、ユーザーが固定して信頼できる品質で保証することが目標だ。全体を通じての参照モデルは**PHPStan**——Rigorはすでにそのプラグイン/型アーキテクチャ（ADR-0/1/2）を踏襲しており、リリースエンジニアリング（`bleedingEdge`オプトイン、`N.x`サポートブランチモデル）を踏襲することは一貫した続きとなる。実装は段階的（perfゲート、bleeding-edgeフラグ、サポートラインポリシー、公開サーフェスドキュメント）に行われ、このADRはコードではなく決定を固定する。

以下の作業決定のいくつかは、*具体的な形状*（bleeding-edgeオーバーレイ、perfしきい値、バックポート/Rubyの相互作用）を提案するものであり、調整の余地がある——該当箇所には**(proposed shape)**と明示している。このADR自体も試行後フリーズモデルに従い、**v0.2.0ライン中は`Proposed`のまま**（最終形状を知らせるリハーサルとして機能）、**v1.0.0で批准**される（契約が固定されるとき）。

## コンテキスト

### エンタープライズ軸

エンタープライズユーザーはCIでRigorのバージョンを固定し、マイナーアップグレードにまたがって2つの保証を期待する。（a）通過していたコードがAPIの変更によっても、それまで発火しなかった診断によっても新たにビルド失敗しないこと、（b）解析が著しく遅くなったり、メモリを多く消費したりしないこと。RigorはCIクリティカルパス上の正確性ツールであり、どちらの次元でのリグレッションも、見た目だけの問題ではなく信頼を損なうイベントとなる。

### 部品はすでに揃っている——ADR-50がそれらを契約に組み立てる

安定リリース戦略が必要とするメカニズムはほぼ構築済みだ。

- **診断チャーン吸収**——ベースライン（baseline）メカニズム＋severityプロファイル（[ADR-22](../22-baseline-and-project-onboarding/)、[ADR-8](../8-steep-inspired-improvements/)）、新規・強化されたルールは`:off`または`:info`で出荷し、コーパスFPゲートを経て昇格するという確立された慣例（[ADR-24](../24-self-method-call-resolution/)、[ADR-47](../47-narrowing-driven-clause-reachability/) WD4）。
- **コーパスバリデーション**——OSSコーパスFPスウィープ（[ADR-47](../47-narrowing-driven-clause-reachability/) WD4で16件をスウィープ）と[`rigor-regression-sweep`](https://github.com/rigortype/rigor/blob/master/.claude/skills/rigor-regression-sweep/SKILL.md)スキル（マルチバージョンのベースラインドリフト）、セルフチェックゲート（`make check` / `check-plugins` / `check-incremental`）。
- **パフォーマンス計測**——Mastodon / Redmine / GitLabでのプロファイリングハーネスとコーパスラン（[ADR-44](../44-dispatch-allocation-churn/)、[ADR-45](../45-unchanged-project-fast-path/)、[ADR-46](../46-incremental-dependency-graph/)）。
- **配布/サプライチェーン**——latest-Ruby専用、`Gemfile`外（[ADR-27](../27-tool-distribution-model/)）、コントリビューションポリシー（[ADR-31](../31-contribution-and-supply-chain-policy/)）。

欠けているのは、これらをリリースに結びつける**契約**だ——公開サーフェスの明示、約束されたperfゲート、非推奨（deprecation）/bleeding-edgeポリシー、サポートラインモデル。それがADR-50のスコープとなる。

### フリーズタイミングの不整合を解消する

3つのADRがプラグイン契約のフリーズ時期について食い違っている。[ADR-25](../25-plugin-contributed-rbs/) WD6と[ADR-32](../32-rbs-inline-comment-ingestion/) WD8は「契約サーフェスは**v0.2.0**でフリーズする」と述べ、[ADR-37](../37-plugin-interface-segregation/)は「フックシグネチャは**1.0**で公開契約としてフリーズする」と述べている。**ADR-50が権威を持ち、ADR-37を支持して解決する**。ハードフリーズは**v1.0.0**だ。v0.2.0はサーフェスの*列挙を開始*し、マイナーノンブレークを軌道として約束するが、フリーズはしない——推論エンジンはまだ進化中であり、早期フリーズは製品の核（深い推論精度）を拙速な保証と引き換えることになる。ADR-25/32の文言は「サーフェスはv1.0.0でフリーズし、v0.2.0はその開始点」と読むべきだ（フォローアップでそれらのテキストを修正するかもしれない）。

## 決定

戦略は4本の柱からなり、それぞれPHPStanを型とする。

### 1. v0.2.0は試行、v1.0.0がフリーズ

**v0.2.0**は*リリースエンジニアリング機構*（perfゲート、サポートポリシー、bleeding-edgeフラグ、公開サーフェスドキュメントの草案）を出荷し、列挙された公開サーフェスに対する**マイナーノンブレーク**を運用上の<ruby>規律<rp>（</rp><rt>discipline</rt><rp>）</rp></ruby>として約束する——v1.0.0契約のリハーサルであり、エンジンがまだ進化しながら実行される。**v1.0.0**がハードフリーズだ。公開サーフェス（§ WD1）は以下の互換性ポリシーの下で拘束力を持つ。

### 2. 互換性: 公開サーフェスはフリーズ、エンジン内部は自由

**Breaking Change**とは、列挙された公開サーフェス（§ WD1）を、準拠するユーザーの設定・プラグイン・抑制を無効化する形で変更することだ。エンジン内部（`lib/rigor/inference/*`、`Scope`、`Type::*`キャリア、ディスパッチティア）は**明示的に非公開**であり、どのリリースでも変更しうる——ADR-19がすでにLSP向けに依拠しているのと同じ内部/公開の分割だ。

### 3. 診断: 出力は非契約、*新しい規律*はゲートが必要

発火する診断のセットは互換性契約の一部では**ない**——以前見逃していた本物のエラーを報告するようにルールを強化することは**マイナー**で許可され、ベースライン（[ADR-22](../22-baseline-and-project-onboarding/)）が常設の吸収装置となる。**しかし**、ユーザーが**それまで求められていなかった新しい規律に従うことを要求する**変更はBreaking Changeであり、次のメジャーでデフォルトオンになるまで**PHPStan方式のbleeding-edgeオプトインの背後でデフォルトオフで出荷される**。（§ WD2 / WD3がその境界線を引く。）診断サーフェスで**契約に含まれる**のは**安定した語彙**だ——ルール識別子、抑制マーカー（`# rigor:disable <id>`）、ベースライン形式、`severity_overrides:`キー——つまりユーザーの抑制とベースラインは、発火が変化しても、アップグレードを通じて機能し続ける。

### 4. パフォーマンス: 約束されたベンチマークゲート

「perfリグレッションなし」は、確立されたOSSコーパスに対してCIゲートとして実行されるベンチマーク（benchmark）ハーネスによって**機械的に**強制され、許容帯域内でコミットされたベースラインと比較する。帯域を超えたリグレッションはビルドを失敗させる（§ WD4）。

### 5. サポート: latest + previous minor、その後は`1.x`ブランチモデル

1.0以前: **最新マイナー＋直前のマイナー**がサポートされる（セキュリティ＋リグレッションバックポート）。1.0以降: PHPStanのモデルに切り替え——`1.x`ブランチがデフォルト開発ラインとなり、サポートラインポリシーがそれに従う（§ WD5）。

## 作業決定

### WD1 — 列挙された公開サーフェス（契約vs内部）

v1.0.0の契約（列挙済み、v0.2.0で草案作成）は以下をカバーする。

| サーフェス | v1.0.0での契約 |
| --- | --- |
| CLIコマンド＋フラグ（`check`、`triage`、`baseline`、`sig-gen`、`lsp`、`mcp`、…） | あり |
| `.rigor.yml`のキー＋値の文法 | あり |
| プラグイン契約——`Plugin::Base`フック＋マニフェストフィールド（ADR-37の`node_rule` / `dynamic_return` / `type_specifier`＋宣言的フィールド） | あり（ADR-37のナロープロトコル、WD3による非推奨のfatフック） |
| 診断**識別子**＋抑制マーカー＋`severity_overrides:`キー | あり（語彙であり、発火セットではない——§ Decision 3） |
| ベースラインファイル形式＋キャッシュスキーマバージョン | あり（形式はあり、スキーマバンプは無効化するのみで誤読しない） |
| `RBS::Extended`アノテーション文法（`%a{rigor:v1:…}`） | あり |
| エンジン内部（`Inference::*`、`Scope`、`Type::*`、ディスパッチティア） | **なし——内部、どのリリースでも変更可能** |
| 特定ファイルで発火する診断 | **なし——§ Decision 3** |
| `rigor sig-gen`の*出力精度*（推論されたシェイプ） | **なし——精度は自由に向上する** |

ドリフトスペックはすでにこの多くを固定している（`public_api_drift_spec.rb`）。WD1はリリースドキュメントが公開する、人間が読めるコンパニオンだ。

### WD2 — inspectable・granularなオーバーレイとしてのbleeding-edge（proposed shape）

bleeding-edgeは**Rigorがメンテナンスするデフォルトオーバーライドのオーバーレイ**だ——次のメジャーのキューに入った変更（severityマップの昇格＋新規規律ルールの有効化）のセットであり、gemとともにバージョン管理される。ユーザーが提供するconfig-fileパスでは**ない**。PHPStanの「差分が明示的」という利点（PHPStanの`bleedingEdge`は*読める*デフォルトオーバーライドincludeだ）を保ちながら、`.neon`includeではなくRigorのYAML＋プラグインサーフェスに合わせる。3つの特性がある。

- **Inspectable**。`rigor show-bleedingedge`（仮称）は、オーバーレイを現在のデフォルトに対する明示的なdiffとして表示する——変更するすべてのルール/severity/規律を、それぞれのfeature idとともに。これはPHPStanが読めるincludeファイルから得る透明性を、パスではなくコマンドとして提供するものだ。
- **Granular＋明示的オプトイン**。`.rigor.yml`の`bleeding_edge:`は`true`（オーバーレイ全体を採用）、**feature idのリスト**（それらのみ採用）、または`{ all: true, except: [...] }`（指定したもの以外をすべて採用）を受け入れる——ユーザーはオールオアナッシングのスイッチではなく、望む高度な機能だけを取り込める。`--bleeding-edge[=ids]`がCLIでそれをミラーする。
- **`severity_profile:`と直交する**。プロファイル＝*今日の*ルールの音量、bleeding-edge＝*次のメジャーの*キューに入った変更のうちどれを早期採用するか。2つは組み合わせられる。

各bleeding-edgeフィーチャーには**安定したfeature id**が付与され、それはWD1の契約語彙の一部となる。`show`サブコマンド、設定ファイル、最終的なCHANGELOGマイグレーションノートはすべて同じidを参照し、フィーチャーはオーバーレイから削除されることで次のメジャーでデフォルトオンに卒業する。これにより新しい規律に実際のランディングパスが生まれる——マイナーNでオーバーレイに出荷し（アーリーアダプタとRigor自身のCIがidで試用）、そのidを鍵にしたマイグレーションノートとともにメジャーでデフォルトにする。

（WD2の審議は、単一のbooleanではなくinspectableなオーバーレイ＋フィーチャー単位のgranularityへと解決した——実際のincludeパスなしにPHPStanの明示的diff透明性を保ち、高度な機能を一度に一つずつ採用できる。棄却された代替案——bleeding-edgeを4番目の`severity_profile`値に折り込む——は依然として成立する。それは音量とdisciplineセットを混同し、granularityと明示的diffの両方を失うことになる。）

**基盤が着地（v0.1.19）**。WD2の*サーフェス*はエンドツーエンドで配線されたが、オーバーレイは意図的に空のままだ（まだメジャー向けにキューに入った規律はない）。`Rigor::BleedingEdge`がメンテナンスされる`Feature`レジストリ（安定id＋サマリー＋`severity_overrides`マップ）であり、`.rigor.yml`の`bleeding_edge:`は`true` / feature-idのリスト / `{ all:, except: }`を受け入れ、正規化されて`Configuration`に公開される。`rigor show-bleedingedge`はオーバーレイ＋プロジェクトが採用するものを表示する（`--format text|json`）。そして`Configuration::SeverityProfile.resolve`は、ユーザー自身の`severity_overrides:`（完全一致またはファミリー）よりも*下*、かつプロファイルテーブルよりも*上*で合成される`bleeding_edge_overrides:`マップを得た。レジストリが空なら合成マップは`{}`なので、解決はビットレベルで変わらないまま——最初のキューに入った規律は、それ以上のエンジン配線なしに単一の`FEATURES`エントリーとして着地する。

**CLIミラーが着地（Unreleased）**。`rigor check --bleeding-edge[=ids]` / `--no-bleeding-edge`は、設定された`bleeding_edge:`の選択を単一の実行に対して上書きする（`--workers` / `--no-cache`と同じCLI-over-configの優先順位）。並行する上書きをすべての`Runner`構築サイト（およびワーカー境界越し）に通す代わりに、CLIは新しい`Configuration#with_bleeding_edge`——`bleeding_edge_severity_overrides`を再導出する凍結された`dup`——を介してロード済みの`Configuration`を再構築する。これにより2つの`SeverityProfile.resolve`サイトとワーカーパスは実行の選択を変わらず見ることになり、`Configuration`が単一の真実の源であり続ける。フラグはOptionParserの`=[LIST]`（attached-value-only）形式を使うので、裸の`--bleeding-edge lib`はオーバーレイを採用しつつパスを消費せず`lib`をチェックする。リスト形式を表面化したことで`coerce_bleeding_edge`のセレクタidも凍結され、config-fileのリストパスに対する`Ractor.shareable?`が復元された。**まだ出荷されていないもの:** 専用のbleeding-edge CHANGELOGセクション（まだ運ぶエントリーはない——最初のキューに入ったフィーチャーとともに着地する）。

### WD3 — Breaking診断変更の定義（規律テスト）

「許可されたマイナー強化」と「BC-breakingな新規規律」の境界線。

- **マイナーで許可（非契約出力）:**そのルールの対象範囲内にすでにあったコードで本物のエラーに発火するルール（見逃したnil-receiver、実際のundefined-method）、または`:off` / `:info`で出荷するあらゆる新ルール。ベースラインで吸収され、新たな記述負担なし。
- **BC——bleeding-edgeの背後にゲート:** **デフォルト**プロファイル下で、以前はcleanだった*慣用的な*コードを、新たな記述上の規律を要求することで（例: アノテーション、より狭い慣用形、それまで要求されていなかった構造変更）失敗させる変更。これらはデフォルトオフで`bleeding_edge:`の背後に出荷され、メジャーでのみオンになる。

コーパスFPスウィープ（[ADR-47](../47-narrowing-driven-clause-reachability/) WD4）が候補を分類する手段だ。慣用コーパスでの発火ゼロ → 新規規律ではなく強化。

### WD4 — パフォーマンスゲート（proposed shape）

- **ハーネス:**確立されたコーパスの固定サブセット（Mastodon `app/models`、Redmine `app`、GitLab `app/{controllers,services,…}`サブセット——ADR-44/45/46がすでにプロファイルするツリー）に対してアナライザーを実行する`make bench-perf`ターゲット、`--no-cache`、固定ワーカー数。
- **メトリクス:**ウォールタイム、総アロケーション（`ObjectSpace`）、ピークRSS、診断カウント（*カウント*変化は意図しない挙動のシフトをフラグする——ADR-44/45/46がすでに使用しているバイト同一診断チェック）。
- **ベースライン＋しきい値はコミットされた、チューナブルな成果物**。コミットされた`bench/baseline.json`（コーパスごと、メトリクスごと）と、コミットされた許容帯域（`bench/thresholds.yml`または同じ場所）——**どちらも意図的にリフレッシュ可能であり、サイレントに変更されない**。帯域はエンジンコードにハードコードされていない。まさにランナー/コーパスの現実が変化するにつれてチューニングできるよう。変更はスナップショットリフレッシュと同様、レビューされるコミットとなる。
- **ゲート:**コミットされた許容帯域を超えてメトリクスがリグレッションした場合、CIジョブが失敗する。**初期**帯域（上記のとおりチューナブル）: ウォール+10%、アロケーション+5%、RSS+10%——ランナーノイズに対してウォールは広く、決定論的シグナルとしてアロケーションは狭い。帯域を超えた*改善*はベースラインのリフレッシュを促すが、失敗にはならない。
- ローカルの`make verify`高速パスには含まない（遅すぎる）。`check-incremental`（ADR-46）のような専用CIジョブ。

### WD5 — サポートラインとlatest-Ruby専用の相互作用

- **1.0以前:**最新マイナー＋直前のマイナーはセキュリティ＋リグレッション修正バックポートを受ける。それより古いものはpin-and-upgrade。2ライン、バックポートコスト有界。
- **1.0以降:** `1.x`ブランチがデフォルト開発ライン（PHPStanのモデル）。サポートウィンドウがそれに従う。bleeding-edgeの規律（WD3）がデフォルトになり、フリーズされたサーフェスのブレーク（WD1）が許可されるのはメジャーバージョンのケイデンス（cadence）で。
- **Rubyのフロア（[ADR-27](../27-tool-distribution-model/)との整合）:**バックポートは**そのラインがすでに出荷されたRubyバージョン**を対象とする——Rubyのフロアを広げず、Rigorは*開発ライン*においてlatest-Ruby専用のままとなる。サポートされた以前のマイナーラインは自身のRubyのピンを保つ。新しいRubyには後付けしない。こうして、latest-Ruby専用と2ラインサポートウィンドウは矛盾なく共存する。

### WD6 — リリース受け入れゲート（リリースをカットするために緑でなければならないもの）

リリース候補は、`make verify`（`test`/`lint`/`check`/`check-plugins`）に加え、以下を通過しなければならない。

1. `make check-incremental`（ADR-46の健全性ゲート）。
2. コーパス**FPスウィープ**——前回リリースと比較して、フリーズされたベースラインコーパスで*新たな*診断ゼロ（`rigor-regression-sweep`の形状）。
3. **perfゲート**（WD4）——許容帯域内。
4. **公開サーフェスドリフトスペック**グリーン（意図しない契約変更なし）。

これは[`rigor-release-prep`](https://github.com/rigortype/rigor/blob/master/.claude/skills/rigor-release-prep/SKILL.md)（バージョンバンプ＋CHANGELOG＋ビルド）と組み合わせる——そのスキルはこのゲートを実行するpre-flightステップを取得する。バージョンバンプの規律（自律的なバンプなし、一桁コンポーネント、ランディング時のリリーススタイルCHANGELOG）は変更なし。

**メカニズム（着地済み、advisory）**。ゲートは`release/x.y.z`ブランチ＋`release-gate.yml`ワークフローとして運用化される。ブランチをpushすると、ベースのCIゲート（`ci.yml`、現在は`release/**`でもトリガー）＋包括的な追加——`make bench-perf` perfゲート（WD4）、gem-buildバリデーション、OSSコーパススウィープ（現在はMastodon、データドリブンなのでRedmine / GitLabはしきい値ファイルの追加で対応）が実行される。試行後フリーズの基本方針に従い、包括的ゲートは**advisory**として出荷される（レポートするのみ、ブロックしない——required-check配線なし、perf/sweepステージは`continue-on-error`で実行）。ベースライン（`bench/baseline.json`、スウィープしきい値）がCI数値に対してキャリブレーションされた後、requiredに昇格する。

### WD7 — 非推奨＋卒業のケイデンス

bleeding-edgeフィーチャー（WD2）は、オーバーレイから削除されることによって、**semver**のメジャーリリースでデフォルトオンに卒業する。このとき、ガイドラインの最低基準として**約4週間のソーク**期間が必要だ。ソークは下限であり、スケジュールではない。すぐにデフォルトにすべきフィーチャーは早いメジャーをトリガーできるが、緊急性のないものは次のメジャーがカットされるまで待つだけだ。卒業は**バンドル化される、フィーチャーごとにソロではない**——メジャーはソークをクリアした*すべての*オーバーレイフィーチャーをオンにし、メジャーは重要な規律を卒業させるため*か*、何らかの他のBC要因（フリーズされたサーフェスの変更、WD1）がすでに強制しているためにカットされる。メジャーは**BC-readiness駆動であり、カレンダー駆動ではない**。そのケイデンスは、ソーク済み規律＋サーフェスブレークのキューが必要とするものだ。

これはPHPStanのモデルに近い——PHPStanは各*マイナー*（`x.y → x.y+1`）でbleedingEdgeセット全体をデフォルトに昇格させる——が、Rigorの**メジャー**に結びついている。Rigorは新たに要求される規律をBreaking Change（§ Decision 3）に分類し、Breaking Changeは1.0以降はメジャー専用だからだ。

**1.0以前（v0.2.0試行）はこれをマイナーバンプでリハーサルする**。`v0.2.x → v0.3.0`ステップでbleeding-edgeフィーチャーが卒業しうる。0.xラインはまだフリーズを持たないからだ。ケイデンスはv1.0.0でメジャー専用に固定される——ADR全体と同じ試行後フリーズの基本方針だ。

**CHANGELOG**。bleeding-edgeの変更は**専用の独立したCHANGELOGセクション**を取得する（PHPStanは`bleedingEdge`を分離して保持）。読者はオプトイン-今/デフォルト-次メジャーのセットをWD2のfeature idをキーにして一目で確認できる。正確な形式は後回し——テンプレートではなく方向として記録する。

## 棄却・延期された代替案

| 候補 | ステータス | 理由 |
| --- | --- | --- |
| v0.2.0でのフルSemVer 1.0フリーズ | 棄却 | 推論エンジンはまだ進化中。今フリーズすると製品の核（深い推論精度）を拙速な保証と引き換えることになる。v0.2.0はリハーサル、v1.0.0でフリーズ。 |
| 診断出力をハードな契約とする（マイナーで新たな発火なし） | 棄却 | 精度/バグ検出の改善をフリーズすることになり——それがRigorの価値の全部だ。bleeding-edgeゲート（WD2/3）は*新しい規律*からエンタープライズを守りながら、本物の強化を流れさせる。 |
| perfを情報的なテレメトリとして扱い、ハードゲートなし | 棄却 | no-regression軸を満たさない。リグレッションは後から気づかれるのではなく、ビルドを失敗させなければならない。 |
| bleeding-edgeを4番目の`severity_profile`値として扱う | 棄却（WD2） | 音量（プロファイル）とdisciplineセット（bleeding-edge）を混同し、PHPStanとの対称性と、両方のノブを組み合わせ可能にする直交性を失う。 |
| 長いバックポートウィンドウを持つLTS | 延期 | 2ライン（latest + previous-minor）ウィンドウが有界コストの出発点。より長いLTSは1.0以降にエンタープライズ需要とメンテナー容量が正当化するなら再検討。 |
| 単一の`rigor check --strict-everything`エンタープライズモード | 棄却 | `severity_profile` + `bleeding_edge` + ベースラインの組み合わせがすべてのエンタープライズポスチャーをすでに表現している。モノリシックなモードはそれらを複製するだけだ。 |
| `rigor upgrade`マイグレーションアシストコマンド（非推奨設定/古いベースライン/キューに入った卒業を検出し、バージョンステップを案内） | 延期（計画済み） | 望ましく、アップグレードUXの自然な場所。しかし投機的にではなく、*具体的な*最初のBreaking Changeに対して設計するのが最善。最初の実際のBCが来たときにspecするよう計画済みの将来の作業として。 |

## 結果

ポジティブ:

- エンタープライズはマイナーを固定し、ラインの中でアップグレードしても、APIや*必須の*診断規律によるビルド破壊のサプライズを受けない。本物のバグ検出は引き続き流れ、ベースラインで吸収される。
- 「perfリグレッションなし」は期待ではなく、機械的なCIのファクト（fact）になる。
- PHPStan方式の`bleeding_edge:`は新しい規律にデフォルトラインを破らない実際のランディングパスを与え——アーリーアダプタに次のメジャーを事前テストする方法を与える。
- フリーズタイミングの不整合（ADR-25/32 vs ADR-37）が単一の権威によって解決される。

ネガティブ/コスト:

- 2つ目のサポートラインは実際のバックポート作業だ（1.0以前は直前のマイナー1つに有界）。
- perfゲートはCIの時間と維持されるベースライン成果物を追加する（意図的にリフレッシュするものが1つ増える）。
- `bleeding_edge:`フラグは、文書化して安定させておくべき新しい公開サーフェスノブだ。

## 他のADRとの関係

- **ADR-37 / ADR-25 / ADR-32**——ADR-50はそれらのフリーズタイミングの文言を解決する（ハードフリーズ = v1.0.0、v0.2.0は列挙の開始）。
- **ADR-22 / ADR-8**——この戦略が診断チャーン吸収のために依拠するベースライン＋severityプロファイル機構。`bleeding_edge:`が直交する第3のノブ。
- **ADR-27**——配布モデル。WD5がlatest-Ruby専用と2ラインサポートウィンドウを整合させる。
- **ADR-44 / ADR-45 / ADR-46**——WD4 perfゲートがコミットされたCIゲートへと変えるプロファイリング/測定作業。
- **ADR-47**——そのコーパスFPスウィープ手法はWD3が「強化vs新規規律」の分類に使い、WD6がリリースゲートとして使う手段だ。
- **ADR-31**——サプライチェーン/コントリビューションポリシー。リリースエンジニアリングはその中で動作する。
- **`rigor-release-prep` / `rigor-regression-sweep`スキル**——WD6受け入れゲートはリリース準備に組み込まれ、スウィープスキルがFPゲートの手段となる。
