ステータス: **Accepted — ゲートは2026-06-12にオープン（スライス1〜3）**。
裁定のアークは完了した。ゲートオープンで発火するすべてのクラスを分類し、
アーティファクトを根本で修正し（スライス1〜3）、残余をgenuine-or-winまで
減らし、WD2に従ってゲートを恒久的にオープンした。
`ExpressionTyper#adoptable_self_call_result?`は削除された。解決済みの
ユーザーメソッド呼び出しは、呼び出し先の推論された戻り値を無条件に採用する。
アーキタイプ: 評価提案（evaluation-proposal）。ステークス: 高（このゲートは
計測された偽陽性（false positive、FP）リグレッションが理由で存在していた。
偽陽性の規律という価値が拘束する）。

根拠: 下記のゲートオープン実験（2026-06-12）と
[2026-06-12 Dynamic-fall調査](../../notes/20260612-dynamic-fall-pattern-survey/)
（バケットB2、再診断: 調査の「非定数パラメータ」という枠組みは誤りだった
── 非定数引数を伴うトップレベル呼び出しは既に精密に型付けされる。実際の残余は
このゲートである）。

## Context

ADR-24 WD3は、解決済みの暗黙self呼び出しの戻り値型をメソッド本体内で採用する
ことをゲートする。採用されるのは`Bot`のみ（およびADR-55以降、アンロール中の
値固定された結果）で、それ以外はすべて`Dynamic[top]`のままとなる
（`ExpressionTyper#adoptable_self_call_result?`）。このゲートが設置されたのは、
無条件採用が`rigor check lib`を16件の診断ぶんリグレッションさせた（ADR-55/56
以前の計測）からである。

これは今や、素朴な手続き型Rubyにとって**最大の残存Dynamic源**である:

```ruby
def helper(x) = x * 2
def outer = helper(3)        # outer → Dynamic[top] (helper resolves; adoption gated)
class S; def run = helper(3); end   # same
def fib(n) = n < 2 ? n : fib(n-1) + fib(n-2)
fib(x)                       # 1 | Dynamic[top] — the gate also blocks ADR-55's
                             # fixpoint summary from reaching in-body call sites
```

隣接する、別途計測された事項: モジュールシングルトン呼び出し
（`def self.triple`に対する`Util.triple(x)`）は、異なる解決経路
（モジュール定数上のシングルトンディスパッチ）を通じて`Dynamic[top]`に型付け
される ── ゲートとは独立してスライス化できる。

## WD1 — ゲートオープン計測（2026-06-12）

ADR-55/56適用後のエンジンでゲートを無条件にオープンする（`true`を返す）と、
**`rigor check lib`で+25件の発火**が生じる（ベースライン: ゼロ）。分布:

- `always-truthy/falsey-condition`警告が約17件 ── 採用された戻り値
  （しばしば定数または非nilの名前的型）が条件を畳み込んだもの。混在母集団:
  一部は真に到達不能な分岐、一部はアーティファクト（`Dynamic`が覆い隠していた
  ヘルパー戻り値の不精密さ ── ADR-55のクランプとADR-56の書き戻しが見つけ続けて
  いたまさにその失敗モード）。
- `Configuration.load`系の経路における`argument-type mismatch`エラーが4件 ──
  採用された戻り値が、`String? | false`形のオプショナリティを呼び出し地点へ
  運ぶようになったもの（サイトごとにgenuine-or-artifact）。
- `possible nil receiver`エラー1件、`return-type mismatch`警告1件、その他2件。

歴史的な「16」は今や「25」になった ── エンジンの精密さが動くにつれ母集団が
変化するのであり、これ自体がゲートを恒久的なものとして扱うのではなくエンジン
世代ごとに再計測する論拠である。

## スライス1の結果 — 裁定＋最初のアーティファクト修正（2026-06-12）

ゲートオープン時のセルフチェック発火25件と3コーパスのデルタを裁定した。
完全な項目別テーブルは
[`docs/notes/20260612-adr57-adjudication.md`](../../notes/20260612-adr57-adjudication/)
にある。8つの異なるメカニズムが見つかり、genuineとアーティファクトに分類された:

- **メカニズム1（セルフチェック発火15件）── 修正済み**。末尾のみの本体
  評価器が明示的な`return value`ノードを取りこぼしていた（`type_of_jump → Bot`）
  ため、述語ヘルパー`return false unless c; …; true`が`Constant[true]`と推論され、
  `if helper`をalways-truthyに畳み込んでいた。修正
  （`StatementEvaluator#eval_return`＋スレッドローカルなreturnシンク、
  `ExpressionTyper#infer_user_method_return`で結合）により、到達可能なあらゆる
  明示的returnが ── Rubyでは囲んでいるメソッドを抜けるブロック内returnも含めて
  ── 推論された戻り値に寄与する。ネストした`def`/lambdaはバリアであり、フローで
  刈り取られた到達不能分岐は寄与しない。これはゲートクローズ時に精密さを加える
  のみ（`lib`、プラグインセルフチェック、Mastodon/haml/kramdownで診断変化ゼロ ──
  すべてバイト単位で同一）。修正後にゲートをオープンすると、25件から減って
  セルフチェック発火は**10**件になる。
- **まだ修正していない残余アーティファクト（独自のスライス）:**ブロックで
  キャプチャされたHash要素／String変更が不可視（メカニズム2／8 ── ADR-56の
  キャプチャ変更ファミリーを、エスケープするブロック内の`h[k]=v`と`s << x`に
  拡張したもの。triage:35、diff:48、kramdown html.rb ×3）。多値`return a, b, c`が
  Tupleに寄与しない（メカニズム7。haml parser.rb ×2 ── メカニズム1の直接的拡張で、
  スライス1のコレクターは単一／裸のreturnのみを扱う）。
- **残余genuine（アーティファクトではない）:** `Configuration.load`のRBSは
  `?String`だが、その本体は`path || discover`とnilチェックなので、5件の
  `argument-type`発火は厳しすぎる*自作*RBSに対して正当に発火している
  （`?String?`が正しい）。同様に`MethodCatalog#reset!`の`() -> nil`というRBSは
  誤り（`@catalog = …`代入はHashを返す）。両方とも真のシグネチャ訂正であって
  エンジンのバグではない。1つのスタブ（`string_unary_blow_up?`は常に`false`）は
  真に無害な到達不能ガードである。`loader.rb:76`は健全な`Array#find`の保守性で、
  Kahnの`in_degree.zero?`不変条件が排除するもの ── 出荷するのはFPリスクがあり、
  フローナローイングが必要。

WD2に従い、残余はすべてgenuineではないため、スライス1の後もゲートはクローズの
ままとなる。次のスライス: メカニズム2/7/8を修正してから再裁定する。RBS経由の
genuine発火は、2つの自作シグネチャを広げることで独立して対処できる。

## スライス2〜3＋ゲートオープン（2026-06-12）

スライス2は、スライス1の裁定で先送りした2つのコーパス／セルフチェック
アーティファクトファミリーを修正した。多値`return a, b, c`がTupleに寄与する
ようになり（メカニズム7）、キャプチャした外側のローカルを内容変更する
エスケープブロックが、そのローカルを裸のコレクションへフロアリングする
（メカニズム2／8）。これにより`OptionParser#on { options[:k] = v }`コールバックが
`options`を不健全に精密な空シードのまま残すことがなくなった。RBS経由の
genuine発火2件は、厳しすぎる`Configuration.load`のパラメータと
`MethodCatalog#reset!`の戻り値シグネチャを訂正することで解決した。

スライス3は残余を解消した:

- **エスケープ内容フロア ── レシーバーチェイン＋境界越えのバリアント**。
  スライス2のフロアは、文レベルの呼び出しに直接付いたブロックに対してのみ
  発火していた。実際のCLIの形は変更ブロックを1ホップ上に隠す ──
  `OptionParser.new do |opts| opts.on { o[:k] = v } end .parse!(argv)`
  （レシーバーチェイン内のブロック）と`build_option_parser(options).parse!(argv)`
  （`options`をパラメータとして受け取った*呼び出し先*の内側に保持された
  ブロック）。エスケープハンドラはレシーバーチェインを辿るようになり、
  パラメータがエスケープ変更されるユーザー定義に解決するself呼び出し
  （メモ化されたdefごとの本体スキャン）は、対応する呼び出し側の引数を
  フロアリングする。両方とも健全（精密さの損失のみ）で、ゲートクローズ時に
  バイト単位で同一。
- **FPセーフなオプショナルタプルスロット分解**。`X | nil`としてフロー型付け
  された分解タプルスロットは、その非`nil`部分へとソフト化される。nil可能な
  スロットは、フローが証明できない相関した不変条件をまたいでほぼ常に
  オプショナルにされる（hamlの`parse_tag`の9タプル。`node, @parent = @parent,
  @parent.parent`）。スロットごとに`possible nil receiver`を製造することは
  動作しているコードを怯えさせる。これはhaml `parser.rb:546`で*ゲートクローズ時の
  偽陽性を2件除去*した ── 裁定された勝利。
- **生きたStringの畳み込みガード**。`string_unary_blow_up?`は常にfalseの
  スタブで、そのガードが採用下で到達不能に畳み込まれていた。今は実際の
  バイトサイズチェックを行う（現在のあらゆる畳み込みに対して振る舞いは同一）。

これらにより、ゲートオープンのデルタは次のとおり: セルフチェック＋
プラグイン契約の発火は**ゼロ**。haml／kramdownコーパスは（今や改善された）
ゲートクローズ時のベースラインと同一。Mastodonは1件発火するが ── これは
既存の同じ`compact_blank!`エラーで、そのメッセージ内のレシーバー型が*より
精密*になったもの（勝利であり、新規発火ではない）。残余はすべて
genuine-or-winなので、WD2に従ってゲートは恒久的にオープンした:
`adoptable_self_call_result?`は削除され、`try_local_def_dispatch`／
`try_user_method_inference`が推論された戻り値を直接採用する。歴史的な
`self_type.nil?`／`Bot`／不動点サマリー／アンロールの特殊ケースはすべて
包摂された。ADR-55 WD1の`clamp_unroll_result`バックストップは独立して保持される。

コスト: コールドな`rigor check --no-cache lib`の実時間が約12%上昇する
（≈26.7秒 → ≈29.9秒）── 以前は`Dynamic`へショートサーキットしていた、解決済み
呼び出し先の本体を再型付けする本質的コスト。明白な呼び出し地点ごとの戻り値結果
メモは先送りした: ADR-52 WD5は呼び出しノードごとの結果キャッシュをスコープ感応性
／FPの理由で却下し、ADR-24 WD5は再帰ガードを引数型でキーイングすることを禁じる。
健全な結果メモはキューイングされたパフォーマンスのフォローアップである。ウォームな
キャッシュ経路（ADR-45／ADR-46）は影響を受けない。

## WD2 — 決定基準

> ゲートは**裁定された発火クラスごとにオープンし、一括ではない**。ゲートオープン
> デルタ内の各発火（セルフチェック＋コーパス）は、*genuine*（採用された型は
> 正しく診断は正当）または*アーティファクト*（採用された型は誤り ── 評価器の
> 盲点）に分類される。アーティファクトはその根本で修正される（各々が見つける
> 価値のあるエンジンのバグである ── ADR-55/56のアークはこの方法で5つを修正した）。
> 残余デルタがすべてgenuineになったとき、ゲートはオープンし、genuine発火は勝利
> として着地する。それまではゲートは留まる。タイムラインを無理に進めるために
> どの発火クラスも抑圧しない。

ティアの順序（それぞれが独自のコーパスゲート付きスライス）:

1. **25件を裁定**（セルフチェック）＋コーパスデルタ（Mastodon／haml／kramdownの
   ゲートオープン実行）: 項目別のgenuine/アーティファクトテーブル。
2. **アーティファクトクラスを修正する**。ADR-55/56のアークから知られている容疑:
   ブロック内`return`がメソッドの戻り値型に寄与しない（末尾のみの本体評価器）、
   早期returnでのオプショナリティの過剰結合、定数が通過すべきでない経路を生き
   延びる。
3. **ゲートをオープンする**（段階化の可能性あり: まずDynamicフリーの結果、次に
   すべて）、再計測し、genuine発火をCHANGELOGに項目別で記載して着地させる。
4. **モジュールシングルトン解決**（モジュール定数レシーバー経由の`def self.x`）を
   独立したスライスとして ── 同じ裁定プロトコル。
   **2026-06-12に着地（LANDED）**。モジュール／クラス定数上のユーザー側
   シングルトンメソッド（`def self.x`／`def Foo.x`／`class << self`／
   `module_function`）への呼び出しは、いまや呼び出し先の本体を`Singleton[X]`
   レシーバーに対して再型付けし、インスタンス側の祖先ウォークを鏡映する:
   新しい`discovered_singleton_def_nodes`インデックス
   （`discovered_def_nodes`のシングルトン版で、`ScopeIndexer`が投入し、
   ファイルごと＋ファイル横断でシードされる）、`Scope#singleton_def_for`
   アクセサ、精密推論ティアが`untyped`ショートサーキットの代わりに走るように
   する`MethodDispatcher#try_discovered_method`の辞退、そして本体スコープが
   同じ`Singleton[X]`のself型を運ぶ
   `ExpressionTyper#try_singleton_method_inference`ティア（これにより、別の
   シングルトンヘルパーへの暗黙selfの呼び出しが同じテーブルに対して解決する）。
   ADR-55の再帰機構と戻り値メモはそのまま適用される（どちらもレシーバー
   キャリアをキーにしており、`Singleton`はそれを満たす）。自クラスのみ。
   シングルトン祖先チェイン（`extend`／継承されたクラスメソッド）は将来の
   スライスである。セルフチェック＋プラグインセルフチェックは発火フリー。
   Mastodonの`app/models`／hamlの`lib`／kramdownの`lib`はバイト同一。CRubyの
   `references/ruby/lib`調査は+16件の発火を表面化させたが、すべてゲート
   オープンプロトコルが予言した「精密化が潜在的なstrictnessを顕在化させる」
   クラスである（いまや解決されるシングルトン戻り値が、消費側のRBS
   シグネチャより精密になる ── 例: `Bundler.root → Pathname`が
   `Pathname#expand_path`のString限定の`dir`パラメータに当たる、いまや
   nilableなシングルトン戻り値がrubygemsの`setup_command`で本物のnil安全性
   ギャップを表面化させる）。これらは読み取り専用のベンダードコーパスの
   シグナルであり、`make verify`のゲートではない。既存のエンジン欠陥が1件、
   新しい入口を通じて表面化した（入口が導入したものではない ── 素の
   インスタンスメソッドでも再現可能）: ADR-55の相互再帰フィックスポイントが、
   2メソッドの再帰（`even?`／`odd?`）を不健全な片側定数に畳み込んでいた。
   この欠陥は同じセッションで修正された（絡み合ったフィックスポイントの
   `untyped`フロアへの降格）ため、`recursive_fixpoint_summary`フィクスチャ内の
   いまや解決される`Parity.even?`シングルトン呼び出しは健全な`Dynamic[top]`に
   型付けされる ── このスライスはその経路を行使する（そして追補が修正する）
   のであって、不健全な畳み込みを祝福するものではない。

ADR-50のbleeding-edgeオーバーレイが先に出荷されれば、オープンしたゲートは自然な
最初の`bleeding_edge:`機能になる。そうでなければWD2基準の下で通常のエンジン精密化
変更として着地する。

## 却下／先送りした代替案

- **今すぐ一括でオープンし、ベースラインで吸収する**。却下 ── 25件のうち4件は
  *エラー*である。FPの規律は、動作しているコード（rigor自身のCLIは動作する）に
  対して裁定されていないエラーを出荷することを禁じる。
- **ゲートを恒久的に維持する**。立場として却下 ── ゲートのコストはエンジンの
  残りがより精密になるにつれ増大する（今やADR-55のサマリーをブロックしている）。
  WD1はブロッカー母集団が有限で列挙可能であることを示す。
- **値固定されていない名前的型のみ採用する（定数をスキップ）**。ティア段階化
  として検討。それ自体は基準ではない ── 価値の大半（そしてアーティファクトリスクの
  大半）は定数採用にある。

## 他のADRとの関係

- **ADR-24** ── これは独自の方法でWD3を再訪する: ゲートをクローズした計測は
  エンジン世代ごとに再実行され、ゲートは主張ではなく裁定によってオープンする。
- **ADR-55／ADR-56** ── 盲点修正の先例（5つのアーティファクトクラスを発見・修正）と、
  本ADRが採用する裁定ゲートプロトコル（ADR-56 WD4）を供給した。ゲートをオープンする
  ことが、それらの精密さをメソッド本体内の呼び出し地点へ届かせる。
- **ADR-50** ── bleeding-edgeオーバーレイが先に出荷されれば、自然な段階化の媒体となる。
