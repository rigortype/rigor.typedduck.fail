**Status:** research note, no design commitments. 外部言語の型システムリリースをRigorと突き合わせた観察。
**Date:** 2026-06-04.
**Rigor version:** working tree（master @ `4eccc390`）に対する観察。
**対象:** Elixir v1.20.0（2026-06-03リリース）の型システム。一次資料は
[Elixir v1.20.0 released](https://elixir-lang.org/blog/2026/06/03/elixir-v1-20-0-released/)、
背景は[Strong arrows: a new approach to gradual typing](https://elixir-lang.org/blog/2023/09/20/strong-arrows-gradual-typing/)、
[Gradual set-theoretic types (HexDocs)](https://hexdocs.pm/elixir/main/gradual-set-theoretic-types.html)、
[The Design Principles of the Elixir Type System (Castagna, Duboc et al., arXiv:2306.06391)](https://arxiv.org/pdf/2306.06391)。

**Why:** Elixirは注釈ゼロの勾配的集合論型を「毎コンパイルで全プログラムに」走らせる方向で完成度を上げてきた。
Rigorとゴール（勾配・低FP・推論優先）がほぼ同型でありながら、健全性（soundness）の扱いだけが正反対なので、
両者を突き合わせて（a）Rigorの方針が業界最前線と一致している確証、（b）FP規律を崩さず取り込める改善、（c）共通の難所、を棚卸しする。
後続の改善検討／ADR起票はこのノートを根拠にする。

**読み順.** §1がElixir側のサーフェス棚卸し、§2が共通点マッピング、§3が設計判断の分岐、§4がRigorに取り込めそうな改善、§5が共通の難所とADR要否の判断。

## 1. Elixir v1.20側のサーフェス棚卸し

1. **アノテーション不要の完全な勾配型付け**。ソースに型注釈を書かずに全プログラムへ推論＋検査を走らせる。
   後方互換のため「すべての引数が`dynamic()`で注釈されているかのように」振る舞う。
   v1.18でパターン推論、v1.20で**パターン＋ガードの完全推論**が完成。

2. **集合論的型（set-theoretic types）が土台**。型をunion / intersection / negationの集合演算で記述・合成する
   （Giuseppe Castagna・Guillaume Dubocらの設計）。

3. **`dynamic()`型 — `any()`ではない**。
   - *Compatibility*: 受理型と供給型が**disjoint（交わりが空）のときだけ**違反として報告する。既存コードでの誤検知を防ぐ。
   - *Narrowing*: 実行に沿って`dynamic()`が絞り込まれる。`data.a`と`data.b`にアクセスすると
     `data`は`%{..., a: number(), b: number()}`にnarrowされる。

4. **ガード／パターンマッチ解析**。
   - `when is_list(x) and is_integer(y)` → union/negationを推論
   - `def example({:ok, x} = y) when is_binary(x) or is_integer(x)` → `x`はbinary|integer、`y`は2要素タプル
   - `when is_map_key(x, :foo)` → `%{..., foo: dynamic()}`、否定側は`foo: not_set()`（**キー不在を正の事実として保持**）
   - `when tuple_size(x) < 3` → タプル長の上限を追跡
   - `case`各節で型を絞り、`nil ->`節通過後の`value`は`binary()`と判定。**冗長節＝デッドコードも報告**。

5. **Strong arrowsによる健全性**。 VMが行う実行時型テストを静的型に織り込み、
   「ランタイムチェックを追加せずに勾配型付けを健全に保つ」。報告するのは
   **「実行されれば実行時に必ず失敗するverified bug」だけ**で、誤検知率は極めて低い。

6. **意図的に先送りした項目**。型シグネチャ（注釈）構文の導入は次の条件付き — (a) v1.20の性能が満足なら、
   （b）再帰型を効率的に実装できれば、（c）パラメトリック型を効率的に実装できれば、
   (d) mapのキー値ペアをenumerableとして走査する型付け問題が解ければ。

## 2. Rigorとの共通点

| 観点 | Elixir v1.20 | Rigor |
| --- | --- | --- |
| 誤検知の徹底回避 | 「verified bug＝実行時に必ず失敗するものだけ」「極めて低いFP率」 | 最上位価値「動いているコードを脅かさない」（`feedback_false_positive_discipline`） |
| 寛容な動的型 | `dynamic()`、disjointのときだけ違反 | `Dynamic[T]`代数＋勾配的整合性＋trinary certainty（[value-lattice.md](../../type-specification/value-lattice/), [relations-and-certainty.md](../../type-specification/relations-and-certainty/)） |
| 集合論的演算 | union / intersection / negationが土台 | 型演算子`~T`（否定）・`T - U`（差）・value lattice（[type-operators.md](../../type-specification/type-operators/)） |
| 制御フローnarrowing | `case`節での`nil`除去、ガード由来の絞り込み | edge-aware narrowing・fact store（[control-flow-analysis.md](../../type-specification/control-flow-analysis/)） |
| 構造的なキー追跡 | `is_map_key`でキー有無を追跡 | 推論オブジェクトシェイプ・構造的インターフェース・capability roles（[structural-interfaces-and-object-shapes.md](../../type-specification/structural-interfaces-and-object-shapes/)） |
| 注釈より推論を志向 | 注釈ゼロで推論、注釈は後回し | `rigor sig-gen`優先、推論精度を上げ手書きRBSを不要にする思想（`feedback_no_ai_generated_rbs`） |

両者とも「勾配的・誤検知最小・実行時に壊れるものを最小限の摩擦で捕まえる」という哲学が驚くほど近い。
Rigorの方針は独自路線ではなく、業界最前線の独立した到達点と一致している。

## 3. 異なる設計判断

1. **健全性のスタンス（最大の差）**。
   Elixirはstrong arrowsで「健全（sound）」を積極追求し、違反は構成上verified bugに限る。
   Rigorは明示的に**非健全**で、robustness principle（戻り値は厳格／引数は寛容という非対称、
   [robustness-principle.md](../../type-specification/robustness-principle/)）＋trinary certaintyというヒューリスティックでFPを抑える。
   **ゴールは同じ（低FP）だが機構が逆方向** — Elixirは理論的保証で、Rigorは確実性ランク＋予算で達成する。

2. **言語基盤への埋め込み度**。
   Elixirはコンパイラ統合で毎コンパイル時に推論が走り、注釈構文はまだ無い。
   RigorはRuby+RBSの外付けチェッカで、**RBSスーパーセット＋拡張注釈が既に完備**、プラグインでフレームワークを拡張する。立ち位置が対照的。

3. **実行時テストの活用**。
   Elixirのstrong arrowsはBEAM VMの実行時型テストを静的型に取り込む。Ruby/Rigorには等価なVM機構が無く、直接移植はできない。

4. **recordsとdictionariesの区別**。
   Elixirはatomキーのmap（record的）とdictionaryを型レベルで区別。Rigorはobject shape＋capability roleで等価のことをするが、切り分けの軸が違う。

## 4. Rigorが取り込めそうな改善

1. **使用箇所からの構造シェイプ合成（demand-driven shape accretion）**。
   Elixirは`data.a` / `data.b`のアクセスを観測して`dynamic()`を`%{a:, b:}`に絞る。
   Rigorのcapability roleは近いが、「メソッド本体で実際にアクセスされたメンバから必須構造を累積する」
   use-site駆動の絞り込みをもう一段進められる余地がある。注釈なしで構造を立ち上げるのはsig-gen思想と合致。

2. **デッドコード／到達不能節の診断**。
   Elixirはexhaustiveな`case` narrowingから冗長節をデッドコードとして報告する。
   Rigorのfact storeはnarrowing情報を持つので、「到達不能な`when`/`case`/`if`分岐」を
   **高シグナル・低FPの診断クラス**として出せる可能性がある。FP規律と整合する種類の警告。
   （既存の`20260603-dead-code-coverage-sweep.md`と接続すること。）

3. **キー不在を正の事実として保持**。
   Elixirの`not is_map_key(x, :foo)` → `foo: not_set()`のように、**ネガティブなキー事実**をnarrowingに乗せると、
   Hashアクセスの絞り込み（`key?`ガード後の分岐）が精緻化する。

4. **コレクション長／タプル長のリファインメント追跡**。
   Elixirは`tuple_size(x) < 3`を事実として追う。Rigorは既に`non-empty-string` / `positive-int`のリファインメント
   （[imported-built-in-types.md](../../type-specification/imported-built-in-types/)）を持つので、
   **配列長・非空性**をfactとしてnarrowingに組み込むのは自然な延長線。

5. **ガード後ブランチの「強い」型化**。
   Rigorは既に`is_a?`でnarrowするが、strong arrowsの発想 — `is_a?` / `respond_to?` / `case ===`通過後の分岐を
   `Dynamic`を落とした強い型として扱う — を徹底すれば、**FPを増やさず実バグ検出（FN削減）を伸ばせる**。
   Rubyのランタイムガードを静的精度の根拠にする方向。

## 5. 共通の難所とADR要否

Elixirが型シグネチャ導入の前提に挙げた残課題は、Rigorの難所とも重なる:

- **再帰型・パラメトリック型の効率** — Rigorの軽量HKT（ADR-20, `project_lightweight_hkt`）と直結。性能予算
  （[inference-budgets.md](../../type-specification/inference-budgets/), `project_inference_budgets_unwired`）と併せて先回りする価値がある。
- **mapをenumerableとして走査する型付け** — Hash/Enumerableの走査型付けはRigorでも未解決領域。

**ADR要否の判断.**このノート単体では設計コミットしない。最も取り込みやすいのは §4-2（到達不能節診断）と §4-3/4-4
（ネガティブ事実・長さリファインメントのnarrowing拡張）で、いずれもFP規律を崩さず実バグ検出を増やせる。
§4-2は既存のデッドコード調査ノートを足場に独立ADR化を検討、§4-3/4-4はcontrol-flow-analysis仕様の追補として扱うのが妥当。
§3-1（健全性スタンス）はRigorの基本設計（robustness principle）と相反するため、追従はしない（記録のみ）。
