ほとんどの品質指標は、あなたが何かをどれだけ*多く*持っているか——カバーされた行数、型付けされた式——を教えてくれます。それらは、あなたが本当に気にかけている問いにめったに答えません。**バグを混入させたら、何かがそれを捕まえるだろうか？**

Rigorの型保護カバレッジ（type-protection coverage）は、まさにそれに答えます。そして、他のどのツールもしないことをします。あなたの**型とテストを1枚のセーフティネットとして**計測するのです。呼び出しサイトが安全なのは、型チェッカーが誤った呼び出しを却下する*か*、テストがレッドになる*かの*いずれかのときであり、本当に無防備なのは**どちらも**起きないときに限られます。そのユニオン（union、合併型とも）こそ、このコマンドが描く絵であり、あらゆるギャップで、より安価に欠けている半分が名指しされます。

## 精度は保護ではない

`rigor coverage`単体は**型精度（type precision）**——Rigorが精密な（`Dynamic`でない）型を与える式の割合——を報告します:

```sh
rigor coverage [paths]
```

それはRigorがどれだけ推論しているかを教えてくれて有用ですが、精度は保護ではありません。決して誤った呼び方をしない、精密に型付けされた式は何の得にもなりませんし、テストが叩きまくる型のない式は完全に安全です。*保護*を——バグが捕まるかどうかを——計測するには、`--protection`を加えます。これは3つのティアからなり、安価な順です。

## ティア1 — ここでバグを捕まえられるか？（静的、即座に）

```sh
rigor coverage --protection [paths]
```

ティア1は、あらゆる**ディスパッチサイト（dispatch site）**（レシーバーを持つメソッド呼び出し）を、レシーバーが具象型を持つかどうか——Rigorのルールが誤ったメソッドや引数を*捕まえられる*サイトかどうか——で分類します。これは1回の解析パスで、対話的にもCIでも実行できるほど速く、**健全な上界**です。具象的なレシーバーは診断が発火するための必要条件ですが、十分条件ではありません。

レポートはまず保護された比率を示し、続いてランク付けされた**「ここに型を足せ」**リスト——`Dynamic`レシーバーで最も多く呼ばれているメソッド、すなわち型アノテーションが最も多くの捕捉力を生む箇所——を示します。`--threshold=RATIO`はそれをCIゲートに変え（比率を下回ると`1`で終了）、`--format=json`は構造化されたフィールドを運びます。

これが日常的に見る数字です。その背後にある真実が欲しくなったら、ティア2へ進みます。

## ティア2 — バグは*実際に*捕まるか？（ミューテーション）

```sh
rigor coverage --protection --mutation [paths]
```

ティア1はサイトが保護され*うる*と言い、ティア2はそれが実際に保護されている*かどうか*を証明します。各サイトに型から見える破壊を導入し——呼び出し引数を`nil`に落とす、その型を入れ替える、呼び出しを存在しないメソッドへ改名する——ミューテーションされたコードを再解析し、Rigorがどれだけ**捕まえる**か（キルレート）を報告します。Rigorが見逃す破壊は本物の「ここに型を足せ」サイトであり、当て推量なしで表面化されます。

多数の解析を実行するので、デフォルトでは**gitで変更された**`.rb`ファイルを対象とし（広げるには明示的なパスを渡します——プロジェクト全体は数分かかります）、対話的なチェックではなくオプトインのCI深掘りです。フレーミングは常に*有効性／どこに型を足すか*であって、決して「あなたのコードは壊れている」ではありません。`Dynamic`サイトで生き残った破壊は、単に型のネットが届かない箇所です。

ここで当然の問いが浮かびます——では何が*届く*のか？ あなたのテストです。

## 融合ビュー — 型**と**テスト（`--with-tests`）

```sh
rigor coverage --protection --mutation --with-tests \
  --test-command "bundle exec rspec" [paths]
```

これがこの機能の核心です。型チェッカーが**捕まえない**破壊ごとに、Rigorはあなたのテストスイートを実行し、**テスト**がそれを捕まえるかを問います。各ディスパッチサイトは3つのバケットのいずれかに入ります:

| 分類 | 意味 |
| --- | --- |
| **type-protected** | 型チェッカーがバグを却下する |
| **test-protected** | 型は見逃すが、テストがレッドになる |
| **unprotected** | どちらも——本物の、無防備なディスパッチサイト |

レポートはあらゆるギャップで**より安価な欠落軸**を名指しします。`Dynamic`レシーバーの穴は*型を足せ*と言い、型付きだがテストされていない穴は*テストを足せ*と言います。サイトが無保護と報告されるのは**両方**の半分が外れたときに限られ——そここそ本物のリスクが実際に潜む場所であり、型だけ・テストだけのツールには見せられないものです。

コストは穴に比例し続けます。型チェッカーがすでにキルする破壊は**スイートに到達しません**（漸進的短絡）ので、高価なテスト実行は静的なネットに隙間がある箇所にのみ費やされます。誠実な見出しはこうなります——*「自分の型が通してしまうバグのうち、自分のテストはどれだけ捕まえるか？」*

`--format=json`は`mode`（`protection-fused`）、`type_killed`、`test_killed`、`unprotected`、`protected_ratio`、ファイルごとの行、そして`add_protection_here`を運びます;`--threshold`は融合比率でゲートします。

### test-commandフック

`--test-command=CMD`は、Rigorがあなたのスイートを実行する方法です（デフォルトは`bundle exec rake`）。知っておくべきことが2つあります:

- **スイートはまずクリーンなコードでパスしなければなりません**。さもなければ「破壊が生き残った」は無意味になります——実行は明確なメッセージとともに中断します。コマンドは素のパス／フェイルのランナーへ向けてください;それ以外はグリーンな単一ファイル実行でも非ゼロ終了するカバレッジフロアがこれに引っかかります。
- **コマンドはシェルなしで実行されます**（argvに分割されて直接実行される）。そしてRigorは自身のBundler環境を取り除くので、`bundle exec`コマンドは*あなたの*プロジェクトのbundleを解決します。したがって`env`ラッパーは不要です——ただしシェル構文は、インラインの`BUNDLE_GEMFILE=… `プレフィックスを含め、解釈されません。デフォルトでないGemfileには、`bundle config set --local gemfile PATH`を使うか、コマンドを`bash -c '…'`で包んでください。

## `--include-dynamic` — 型のないコードをカバーする

デフォルトでは、融合ビューはRigorが型チェックできるサイトだけをミューテーションします。しかし動的なRubyではそれは少数派です——ほとんどの呼び出しのレシーバーは`Dynamic`であり、そこでは**テストが唯一可能な保護**です。`--include-dynamic`はそれらのサイトもミューテーションします:

```sh
rigor coverage --protection --mutation --with-tests --include-dynamic \
  --test-command "bundle exec rspec" --limit 40 [paths]
```

これはマップを、型付きのものだけでなく*すべての*ディスパッチサイトへと完成させ、融合ビューがその真価を発揮する場所です——あなたの型のないコードのうち、どれがテストに支えられ、どれが何にも支えられていないかを示します。そのようなサイトはどれも、定義からして型チェッカーが捕まえられないものなので、スイートをはるかに多く実行します;したがって明示的なオプトインです。

`--limit=N`（`--seed=N`付き）は計測をファイルごとの`N`個のミューテーションの決定的なサンプルに上限し、大きなファイルでのコストを抑えます;ファイルごとの比率はその後推定値となり、`--format=json`のstdoutがクリーンに保たれるようstderrに注記されます。

## レポートを読む — 「ここに型**または**テストを足せ」リスト

無保護のサイトがペイロードです。実際には、それらはいくつかの見分けやすい種類に分かれ、それぞれに自然な修正があります:

- **テストされていないメソッド本体** — スイートが決して行使しないヘルパー。*テストを足す*か、テストコマンドを広げてカバーします。
- **到達されない分岐** — エラー経路（`rescue`内の`raise`）、バージョンディスパッチのアーム。重要なら、その分岐に*テストを足し*ます。
- **`Dynamic`レシーバーのコラボレータ** — 外部gemオブジェクトへの呼び出し、フレームワークのファサード、ダックタイピングされたパラメータ、あるいはメタプログラミングDSL。*型を足し*ます——多くは[`rigor sig-gen`](../02-cli-reference/#rigor-sig-gen)からの1行——そうすれば静的なネットがそれを捕まえ始めます。

完全性についての注意: `--test-command`をあなたのテストの*部分集合*にスコープすると、レポートは`unprotected`を過大報告します（*別の*テストが捕まえる破壊がギャップとして現れます）。正確なマップのためには、計測対象のファイルを行使するすべてのテストにそのコマンドを走らせてください——時間と完全性を引き換えにします。

## コストとスコープ

融合ティアは本物の解析と本物のテストスイートを実行するので、キーストロークごとのチェックではなく深掘りとして扱ってください:

- **スコープを絞る**。変更ファイルのデフォルト（パスを与えない）は、実行を差分に比例させ続けます;意図的に広げるには明示的なパスを渡します。
- **スイートを速く保つ**。コストは`（計測される破壊数） × （スイートの実行時間）`です。速く、よくスコープされたテストコマンドが最大のレバーです。
- **`--limit`で上限を設ける** — `--include-dynamic`や大きなファイルで。

## CIでは

ティア1（`--protection`）はすべての実行をゲートできるほど安価です;ミューテーションと融合のティアは、スケジュールされた、またはラベルでトリガーされる深掘りとして適しています。すべてのティアは`--threshold`（融合比率または有効性比率のゲート）と`--format=json`を尊重するので、それらをパイプラインに配線することは`rigor check`と同じ機構を再利用します。[CIでのRigor実行](../11-ci/)を参照してください。
