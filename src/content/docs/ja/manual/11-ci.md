RigorはRuby 4.0で動作します（[Rigorのインストール](../01-installation/)を参照）。CIではこれが最初に述べておく価値のある1つの帰結をもたらします。このページの残りはそこから導かれます。

## 独立したジョブでRigorを実行する

テストスイートとは**別のCIジョブ**でRigorを実行してください——できればワークフローファイルも別にします。理由は具体的です: `ruby/setup-ruby`は*ジョブの*アクティブなRubyを設定します。テストジョブはプロジェクトが実行するRuby（多くの場合3.xバージョン、またはそのマトリックス）をプロビジョニングします。RigorはRuby 4.0が必要です。2番目の`setup-ruby`呼び出しが最初のものを上書きするため、2つを同じジョブで共有することはできません。

独立したジョブはRigorにRuby 4.0のプロビジョニングが何とも競合しないクリーンなランナーを提供します。独立したワークフローファイルを使用すると、さらに独自のトリガー、並行性グループ、ステータスバッジを持ち——テストワークフローからアナライザーを分離できます。これはどちらにせよ良い実践です。

## 最小限のGitHub Actionsワークフロー

```yaml
# .github/workflows/rigor.yml
name: rigor
on: [push, pull_request]
jobs:
  rigor:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: ruby/setup-ruby@v1
        with:
          ruby-version: "4.0"
      - run: gem install rigortype
      - run: rigor check
```

これだけです: プロジェクトをチェックアウトし、Ruby 4.0をプロビジョニングし、Rigorをインストールして実行します。そのワークフローは診断をジョブログ内にのみ表示します。それらを**変更されたコード上にインライン**で — PRアノテーション、code scanningアラート、マージリクエストウィジェットとして — 表示するには、以下のCIネイティブな出力フォーマットのいずれかを使います。

## GitHub Actionsではそのまま動く（自動検出）

`rigor check`は自分が実行されているCIを検出し、デフォルト出力では**`--format`なしで**プラットフォームネイティブな方法で診断を表示します — PHPStanのCI自動検出をモデルにしています。そのため、このページ冒頭の最小限のワークフローはGitHub Actions上ですでにインラインのPRアノテーションを生成します: 人間向けのログはそのまま残り、その上に`::error`アノテーションが追加されます。

- **GitHub Actions**と**TeamCity**（ファーストクラス）: ネイティブなアノテーションがテキスト出力の上に自動で発行されます。
- **GitLab CI**（ネイティブだがアーティファクトベース）: 1行のヒントが`--format gitlab`を実行して`codequality`アーティファクトを公開するよう促します（後述）。
- **その他のCI**（CircleCI、Jenkins、Travis、Azure、Bitbucket、Buildkite、Drone …）: 1行のヒントがreviewdogまたは`--format junit`の経路を案内します。

自動検出はデフォルトの人間向け出力を補強するだけ — 明示的な`--format`には決して触れません。`--no-ci-detect`または`RIGOR_CI_DETECT=0`でオフにできます。以下のセクションは、自動のデフォルトではなく*特定の*表示（code scanning、GitLabウィジェット、レビューコメント）が欲しい場合のためのものです。

## PR／MR上に診断をインライン表示する

`rigor check --format`は、診断をジョブログに残す代わりに、CIプラットフォームが読み取って差分に直接注釈を付けるフォーマットで同じ診断をレンダリングします（[ADR-51](../adr/51-ci-diagnostic-output-formats/)）。これらは汎用の`--format json`ストリームの上に重なります;新しい診断を追加するのではなく、プラットフォームネイティブなレンダリングだけを加えます。コピーしてすぐ使えるテンプレートファイルは[`ci-templates/`](ci-templates/)にあります。

### GitHub — インラインアノテーション（デフォルト）

`--format github`はGitHub Actionsのワークフローコマンド（`::error file=…,line=…::`）を発行し、ランナーがそれをPR上のインラインアノテーションに変換します — アップロード手順も、code scanningのセットアップも、追加の権限も不要です。**あらゆる**リポジトリ（公開でも非公開でも、有料機能なしで）で動くため、GitHubでの推奨デフォルトです:

```yaml
      - run: gem install rigortype
      - run: rigor check --format github
```

### GitHub — code scanning（SARIF）

セキュリティタブ、重複排除された永続的なアラート、よりリッチなPRアラートが欲しい場合はSARIFにアップグレードします — **code scanningが利用できるとき**: 公開リポジトリ（無料）、またはGitHub Advanced Security / Code Securityを有効にした非公開リポジトリ。（それがないと、`upload-sarif`は*「GitHub Advanced Security must be enabled for this repository」*で失敗します — 上記のアノテーション形式を使ってください。）checkの手順は`continue-on-error`を使い（アップロードは`if: always()`で走り）、Rigorが非ゼロ終了してもレポートは公開されます:

```yaml
# .github/workflows/rigor.yml
name: rigor
on: [push, pull_request]
permissions:
  contents: read
  security-events: write   # required by upload-sarif
jobs:
  rigor:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: ruby/setup-ruby@v1
        with:
          ruby-version: "4.0"
      - run: gem install rigortype
      - run: rigor check --format sarif > rigor.sarif
        continue-on-error: true
      - uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: rigor.sarif
```

SARIFはクロスプラットフォームな選択肢です: 同じレポートがGitHubだけでなく、あらゆるSARIF対応ツールで利用されます。

### GitLab — Code Qualityウィジェット

`--format gitlab`はGitLab Code Quality（CodeClimateのサブセット）レポートを発行します。これを`codequality`レポートアーティファクトとして公開すると、GitLabがマージリクエストウィジェットに検出結果をレンダリングします:

```yaml
# .gitlab-ci.yml
rigor:
  image: ruby:4.0
  script:
    - gem install rigortype
    - rigor check --format gitlab > gl-code-quality-report.json
  artifacts:
    reports:
      codequality: gl-code-quality-report.json
    when: always
```

### あらゆるプラットフォーム — reviewdog

[reviewdog](https://github.com/reviewdog/reviewdog)はGitHub、GitLab、Gerrit、Bitbucket、Giteaで検出結果をPR/MRのレビューコメントとして投稿します。Rigorの`sarif`または`checkstyle`出力を読むため、これらのフォーマットのいずれかでそのすべてのレポーターに届きます。[`reviewdog/action-setup`](https://github.com/reviewdog/action-setup)でインストールし、Rigorをそこにパイプします:

```yaml
      - uses: reviewdog/action-setup@v1
      - run: rigor check --format checkstyle | reviewdog -f=checkstyle -reporter=github-pr-review -fail-level=error
        env:
          REVIEWDOG_GITHUB_API_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

`github-pr-review`には`permissions: pull-requests: write`が必要です。`rigor-ci-setup`スキル（`rigor skill print rigor-ci-setup`）がレポーターと`-fail-level`の選択を案内します。

### その他のテストレポートCI — JUnit

`--format junit`はJUnit XMLを発行し、GitHubのテストレポート、GitLab、CircleCI、Jenkinsがレンダリングします。すべての診断は、その深刻度で型付けされた`failure`を持つ`testcase`になります:

```sh
rigor check --format junit > rigor-junit.xml
```

### 深刻度のマッピング

Rigorの深刻度はフォーマットごとに次のようにマッピングされます（[ADR-51](../adr/51-ci-diagnostic-output-formats/) WD2）:

| Rigor | SARIF | GitHub | GitLab | Checkstyle | JUnit | TeamCity |
| --- | --- | --- | --- | --- | --- | --- |
| error | `error` | `::error` | `major` | `error` | `error` | `ERROR` |
| warning | `warning` | `::warning` | `minor` | `warning` | `warning` | `WARNING` |
| info | `note` | `::notice` | `info` | `info` | `info` | `INFO` |

終了コードは`--format`では変わりません — エラーがなければ`0`、それ以外は`1` — ので、ジョブは引き続きパイプラインをゲートします。`--format json`は、生の診断ストリームが欲しいほかのツールのために引き続き利用できます。

## Rigorバージョンのピン留め

上記のワークフローは実行時の最新の`rigortype`をインストールします。バージョンをピン留め——CIを再現可能に保つ——には、以下のいずれかを選択します。`0.1.x`系がプレビュー中の間はサーフェスがまだ落ち着いていないため、ピン留めを推奨します;Rigorが（どのリリースから）安定して保つことを約束するものは[`docs/compatibility.md`](../compatibility/)に列挙されています。

### CI専用`Gemfile`（推奨）

2行の`.github/rigor/Gemfile`をコミットします:

```ruby
source "https://rubygems.org"
gem "rigortype", "~> 0.1"
```

その`Gemfile.lock`とともにコミットし、`BUNDLE_GEMFILE`を通じてRigorジョブに指定します:

```yaml
jobs:
  rigor:
    runs-on: ubuntu-latest
    env:
      BUNDLE_GEMFILE: .github/rigor/Gemfile
    steps:
      - uses: actions/checkout@v4
      - uses: ruby/setup-ruby@v1
        with:
          ruby-version: "4.0"
          bundler-cache: true
      - run: bundle exec rigor check
```

この`Gemfile`はRigorジョブのみが読み込みます——アプリケーションの依存関係解決には決して入り込まず、コミットされたlockfileがRigorとその依存関係を再現可能な実行のためにピン留めします。通常のBundler `Gemfile`なのでDependabotが最新に保てます: `.github/dependabot.yml`でそのディレクトリにスコープされた`bundler`エントリーを追加します:

```yaml
version: 2
updates:
  - package-ecosystem: bundler
    directory: /.github/rigor
    schedule:
      interval: weekly
```

### ピン留めした`gem install`

ワークフローで`gem install rigortype -v "0.1.15"`を実行します。余分なファイルなしでよりシンプルですが、Dependabotは`run:`ステップ内のバージョンを認識しないため、ピンの更新は手動になります。

## コンテナイメージ

Ruby 4.0とRigorが組み込まれたスタンドアロンイメージがGHCRに公開されています。Rubyツールチェーンのないランナーに適しています——プロジェクトを`/src`にマウントします:

```sh
docker run --rm -v "$PWD:/src" ghcr.io/rigortype/rigor check
```

`rigor`がイメージのエントリーポイントなので、サブコマンドとフラグはイメージ名の後に続きます。明示的なタグでバージョンをピン留めします——`ghcr.io/rigortype/rigor:0.1.15`。

## Nix

すでにNixを実行しているCIでは、flakeがRuby 4.0をクロージャに含むパッケージとしてRigorを公開しています——ランナーに他のものが何もない完全にハーメティックな実行です:

```sh
nix run github:rigortype/rigor#rigor -- check
```
