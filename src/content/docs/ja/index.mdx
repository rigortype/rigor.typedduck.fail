Rigorはソースに型注釈を一切書かずに`rigor check`でRubyコードを解析し、型付きレシーバーへの未定義メソッド呼び出し、引数個数の誤り、証明可能なnilデリファレンスなどの実際のバグを報告します。型はすべてコードが実際に生成する値から推論されます。チームに型注釈を書いてもらう必要はありません。

普通のRubyに`rigor check`を向けるだけ——型注釈も設定も不要で、実際のバグを報告します:

```ruby
# lib/demo.rb
"hello".no_such_method        # typo'd method name
[1, 2, 3].rotate(1, 2)        # too many arguments
```

```text
lib/demo.rb:1:9: error: undefined method `no_such_method' for "hello" [call.undefined-method]
lib/demo.rb:2:11: error: wrong number of arguments to `rotate' on Array (given 2, expected 0..1) [call.wrong-arity]
```

以下、目的別に2つの導線があります——プロジェクトに**導入する**ための道筋と、仕組みや他ツールとの違いを**評価・調査する**ための道筋です。

**2つのコマンドで始められます:**

```sh
mise use ruby@4.0
mise use gem:rigortype
```

## Rigorを導入する

プロジェクトにRigorを組み込み、日々のワークフローに乗せるための導線です。

- [インストール](/ja/manual/01-installation/): mise / asdf / `gem install` / devコンテナ。上の2コマンドが最短経路です。
- [はじめての解析](/ja/handbook/01-getting-started/): 最初の`rigor check`まで。
- [設定](/ja/manual/03-configuration/): 重大度プロファイル、対象範囲、プラグイン。
- [ベースライン](/ja/manual/06-baseline/): 既存コードベースへ段階的に導入する。
- [CI](/ja/manual/11-ci/)・[エディター統合](/ja/manual/09-editor-integration/): 日常のワークフローへ組み込む。
- [Railsクイックスタート](/ja/manual/14-rails-quickstart/): Railsアプリに最短で乗せる。

## Rigorを評価・調査する

導入を決める前に、Rigorの仕組みと他ツールとの違いを確かめるための導線です。Sorbetがインラインの`sig`を、Steepが手書きのRBSを求めるのに対し、Rigorはどちらも要求しません——型はすべてコードが実際に生成する値から推論されます。

- 他の型チェッカーから来たなら: [Sorbet](/ja/handbook/10-sorbet/)・[Steep](/ja/handbook/appendix-steep/)・[TypeProf](/ja/handbook/appendix-typeprof/)・[TypeScript](/ja/handbook/appendix-typescript/)（mypy・Go・Rustほかの付録もハンドブックにあります）。
- [型仕様の概要](/ja/type-specification/): 何を型とみなし、何を保証するか。注釈なしで推論する設計の核心です。
- [ナローイング](/ja/handbook/03-narrowing/): フローセンシティブな解析が何を実現するか。
- [型理論との接続](/ja/handbook/appendix-type-theory/): 設計を支える理論的背景。
- [互換性](/ja/compatibility/)・[ロードマップ](/ja/roadmap/): 対応Rubyバージョンと今後の方針。
