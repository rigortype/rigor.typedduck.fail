ステータス: **Accepted（強形は棄却）、2026-06-12**。 RSpecのアサーションを*実装に対する*型推論のソースとして使えるのではないか — `expect(foo(1)).to be_a(String)` ⇒ `foo: (Integer) -> String`を採用する — という創設期のアイデアは、先送りではなくポリシーとして棄却する。偽陽性の<ruby>規律<rp>（</rp><rt>discipline</rt><rp>）</rp></ruby>と両立する3つの弱形（矛盾診断、`Dynamic`内部のヒント、ランタイムトレース→sig-gen）は、認可された将来の道筋としてここに記録する。いずれも需要ゲート付きである。READMEのコミットメント#2は、公開されている語り口が実際に出荷されたものと一致するよう、同じ変更で文言を改める。

## コンテキスト

「あなたのスペックが型である」はREADMEの3つの設計コミットメントの1つである。その下で出荷されたもの（rigor-rspec / rigor-factorybot、「柱2」）は実在するが、流れは一方向のみである。マッチャーのアサーションは*スペック本体の中で*ローカル変数をナローイングし（`plugins/rigor-rspec/lib/rigor/plugin/rspec/matcher_analyzer.rb`、`post_return_facts`）、`let`／`subject`の束縛は*実装側で*推論された型を`it`本体へ運び、ファクトリー定義はADR-9の`:factory_index`ファクトを公開する。スペックのアサーションを`lib/`メソッドの推論済みシグネチャへ還流させるチャネルは存在せず、もともとの強形のアイデアは、いまや精度に関わるあらゆる変更を統べる偽陽性（FP）の規律より前に生まれたものである。本ADRはこのアイデアを明示的に引退させ、未解決の問いとして再浮上するのを止めるとともに、その代わりに出荷してよいものを記録する。

## 決定

スペックのアサーションに由来する実装シグネチャを棄却する。判別基準は次のとおり。

> テストのアサーションは入力ごとの*証人（witness）*であって、普遍的な主張ではない。証人はシグネチャを両端とも間違った側から押さえるため、スペック由来の型主張が着地してよいのは、間違っていても診断を発火させ得ない場所 — 漸進的（gradual）な内部（`Dynamic`の`static_facet`）、人間によるレビューのチャネル（`rigor sig-gen`の出力）、あるいは推論に*対する*検査として — に限られ、採用済みのシグネチャとなることは決してない。

## 作業上の決定

- **WD1 — 強形が、労力とは無関係にRigorにとって不健全である理由**。観測された戻り値は*下界*である。アサートされた戻り値型のユニオンは真の戻り値型の部分集合であり、これを宣言された戻り値として採用するとシグネチャが狭まり、スイートが演習しなかった呼び出し箇所で偽陽性が発火する。スペック中の呼び出し例も同様に、受理されるパラメータの下界であり、これを採用すると他の現役の呼び出し元を拒絶する。ロバストネス原則（ADR-5: 戻り値に厳格、パラメータに寛容）は、テストが下界を与えるまさにその両端で上界の情報を必要とする。この境界の非対称性に加えて、アサーションはたまたま実行された願望である。`pending`／`xit`／タグフィルタがあるため、CIがグリーンでも所与のexampleが実行されたとは限らず、解析器はスイートがグリーンであること自体を静的に知り得ず、古びたアサーションは古びた注釈とまったく同じようにドリフトする — これは「型は願望ではなくファクトである」（READMEコミットメント#1）に矛盾する。

- **WD2 — 障壁はアーキテクチャではなくポリシーである**。仕組みとしては実現可能だろう。rigor-factorybotの`producer :factory_index`がすでにこのパターンを実証しており（ファイル単位の推論パスの外でプロジェクトファイルを`io_boundary`としてAST走査し、ADR-9ファクトとして公開する）、類似の`spec_return_claims`インデックスに必要なのはリテラルなマッチャー認識だけで、完全な推論は要らない。フォークプール下での順序付けとADR-46のスペック→lib依存エッジはコストではあるが、壁ではない。したがってこの棄却は、アーキテクチャが変わっても失効しない — 失効するのはFPの規律が失効したときだけである。

- **WD3 — 認可された弱形（将来の作業、需要ゲート付き）**。いずれも構成上、上記の基準を満たす。
  1. **矛盾診断** — 推論が`expect(foo(1)).to be_a(String)`の成立不可能性を証明したとき（subjectへのマッチャーによるナローイングが`bot`に落ちたとき）、`spec.impossible-assertion`診断を発火する。これはスペックを推論に*対して*検査するものであり、推論そのものには触れない。最も安価な道筋で、`MatcherAnalyzer`とADR-47のbot検出ファミリーを再利用する。
  2. **`Dynamic`内部のヒント** — 推論が`Dynamic[top]`に落ちた場所（メタプログラミング、未解析のgem）では、スペックの主張が`static_facet`を`Dynamic[T]`へ精緻化してよい（`lib/rigor/type/dynamic.rb`）。`Dynamic`は決して診断を発火しないので、誤った主張のコストはゼロであり、下流の精度・`type-of`・sig-genヒントは改善する。機構: WD2の主張インデックスを、推論結果が`Dynamic[top]`のときに限りディスパッチで消費する。
  3. **ランタイムトレース→sig-gen** — TracePointハーネスの下でスイートを実行し、*観測された*型（アサーションではなくファクト）を記録し、既存の`rigor sig-gen`レビューチャネル（ADR-14）を通じてRBSを出力する。オプトインのランタイム実行はコアではなく別gemに置き、`rigor check`の「ランタイム依存なし」の主張を保つ。

- **WD4 — READMEの整合**。コミットメント#2の文言（「本当に型注釈を書く必要がありますか？」「生きた型オラクル」）は、スペック→実装の推論を含意していた。出荷された方向（アサーションはスペック本体をナローイングする。実装の型はスペックへ流れる。ファクトリーはプラグインチャネルへ供給する）を述べるよう、本ADRと同じ変更で文言を改める。*フォローアップ（同日、`62bec417`）*: 改稿後のコミットメントはプラグイン機能の紹介ツアーのように読め、設計コミットメントには見えなかったため、このスロットは偽陽性の規律の柱で埋め直した。rspec／factorybotのケイパビリティはプラグインカタログに引き続き文書化され、ROADMAPの「柱2」への言及には、ここを指す`<del>`／`<ins>`の撤回マーカーが付く。

## 棄却・先送りした代替案

| 代替案 | 扱い |
| --- | --- |
| アサーション由来のシグネチャをそのまま採用する | 棄却 — WD1の境界の非対称性により、呼び出し元での偽陽性は偶発ではなく構造的になる。 |
| 確信度で重み付けした採用（アサーションをソフトなシグネチャとし、なお診断をゲートさせる） | 棄却 — 未検証の主張が診断を発火し得る経路はすべて基準を満たさない。「ソフト」にしても、情報が境界のどちら側にあるかは変わらない。 |
| スイートがグリーンであることの証明を採用の条件にする | 棄却 — グリーンであることは静的には知り得ず、健全性をCIの状態に結合させることは、検証不能な外部事前条件の持ち込みになる。 |
| 弱形1〜3（WD3） | 先送り、需要ゲート付き — 本ADRでは実装のコミットメントはしない。 |

## 帰結

- **ポジティブ**: 創設期のアイデアが再浮上を繰り返す代わりに、記録された決着を持つ。READMEコミットメント#2が事実として正確になる。3つの弱形はFP安全性の論証込みの居場所を得るので、将来のスライスは基準の再審理ではなく基準そのものから出発できる。
- **ネガティブ**: 運用上は皆無 — 出荷済みの挙動は何も変わらない。

## 他のADRとの関係

- **ADR-9**は、WD3の各道筋が使うことになるファクトチャネルの基盤である。rigor-factorybotの`:factory_index`が実例としての先行事例。
- **ADR-14**は、WD3の道筋3が出力先とするsig-genレビューチャネルを所有する。
