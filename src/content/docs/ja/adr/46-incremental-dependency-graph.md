**ステータス:** Accepted — 実装済み（ボディティア）。全体実行キャッシュ（[ADR-45](../45-unchanged-project-fast-path/)）は粗粒度（解析済みファイルが1つでも変わると全体を再実行）。このADRはその後継として、ファイル単位のインクリメンタル解析（incremental analysis）を実装する。葉のコントローラーを編集した場合はそのファイルだけを再チェック、モデルを編集した場合はそのモデルと実際に依存しているファイルだけを再チェックする。`rigor check --incremental`はクロスプロセスのボディティア（スライス1+2）を提供する: ファイル単位のクロスファイル依存関係を記録 → `dependents`へ反転 → 実行時に`ΔF ∪ dependents[ΔF]`だけを再解析し、残りをディスクスナップショットから提供。健全性は必須の`--verify-incremental`ゲート（インクリメンタル == `--no-cache`全体解析、バイト同一）によって担保され、CIに組み込まれている。Rigor自身の`lib`（262ファイル）での計測: ウォーム変更なし0.75s vs全体解析7.2s（約9.6倍）、1ファイル葉編集1.15s（約6.3倍）、診断結果は同一。スライス3（構造的ティアのネガティブ依存関係追跡）とスライス4（シンボル粒度）はランディング済みであり、構造的ティアはファイルの*追加・削除*もインクリメンタルにカバーするようになった（スナップショットフィンガープリントはファイルリストではなく解析ルートをキーとしており、追加されたファイルはネガティブエッジ経由でそれが今や定義する名前に依存していた消費者を再チェックし、削除されたファイルはその正のdependentsを再チェックする）。スライス3はウォームセッションの健全性ギャップも2つ解消した: 後続の編集で定義が届く`helper()`の見逃し（`call.unresolved-toplevel`）と、スーパークラスが後から定義されるサブクラス（`def.override-*`、そのチェッカーがレコーダーのチョークポイント外でクラスグラフを読む）の見逃しだ。

ADR-45は*変更なし*のプロジェクトを高速化した（全体実行のキャッシュを記録・検証する手法で、GitLab上では約42倍）。ただし意図的に粗粒度にしている。変更ファイルが1つあればエントリー全体が無効になるため、1ファイルを編集するだけで全体の再スキャンが必要になる（GitLab上では約113秒）。このADRの背景にある直観はまさにユーザーの感覚と一致する。MVCアプリにおいて具体的な`PostsController`は依存グラフ（dependency graph）の**葉**であり、そのアクションを通じて何かが型推論されることはない。したがってその編集はそのファイルだけの再チェックで済むべきだ。一方`Post`モデルを編集した場合は、そのメソッドを呼び出しているサービスやコントローラーのほんの一部だけを再チェックすればよく、2,630ファイル全体を対象にする必要はない。

この設計は、解析済みの各ファイルが*実際に他のファイルから何を読み取ったか*を記録し、それを反転させてdependentsインデックスを構築する。編集時には影響を受ける推移閉包（closure）だけを再解析し、それ以外のすべてのファイルの診断結果はキャッシュから提供する。健全性（soundness、古い診断結果を提供しない）がこの設計の核心課題である。

根拠: [ADR-45](../45-unchanged-project-fast-path/)（全体実行キャッシュと「プラグインが解析中にファイルを読み込む」という問題）および[ADR-24](../24-self-method-call-resolution/)（ファイル横断のself呼び出し解決）。

## あるファイルの診断結果が実際に依存するもの

AのASTと、Aの解析が消費したすべてのファイル横断（cross-file）ファクトを純粋関数として受け取り、Aの診断結果が得られる。エンジンに照らしてファイル横断ファクトの種類を列挙すると次のとおりだ:

1. **クラス・メソッド・定数の宣言**。Aは`Post`、`Post#publish`、`Admin::Report`をプロジェクト全体の`discovered_classes` / `discovered_def_nodes` / `discovered_superclasses` / `discovered_includes`インデックス（`seed_project_scope`によって各ファイルの`Scope`に注入される）に対して解決する。エンジン内の66箇所のファイル横断読み取りは、`user_def_for`、`user_def_site_for`、`superclass_of`、`includes_of`、`top_level_def_for`、及びdiscovered-classルックアップという少数の`Scope`アクセサに集約される。これがこの設計が計装する**チョークポイント（choke point）**だ。
2. **推論された戻り値のボディ（深いエッジ）**。`infer_user_method_return`は`Post#publish`の`DefNode`を解決し、呼び出しの戻り値型を推論するために**そのボディを評価する**。したがってAは`Post#publish`の存在だけでなく、その**ボディ**にも依存する。推論された戻り値型を変える本体編集は、Aの診断結果を変える。
3. **クラス状態の推論**。プロジェクト全体の`class_ivars` / `class_cvars` / `program_globals` / `in_source_constants`は、あるファイルで代入され別のファイルで読み取られる型を持つ。
4. **プラグインの寄与**。プラグインは解析の途中でプロジェクトファイルを読み込み（`rigor-pundit`ポリシー）、診断結果や戻り値型を提供することがある。各プラグインの`io_boundary.cache_descriptor`（ADR-45）を通じてファイルダイジェストとして既に捕捉されている。
5. **グローバル入力**。RBS環境（gems + `sig/`）、解決済みの設定、`Rigor::VERSION` — ADR-45のフィンガープリント（fingerprint）。

主要な種類についてはソースの帰属（attribution）が既に存在する。`discovered_def_sources`は`(class, method) → "path:line"`をマッピングするため、メソッド宣言や推論戻り値の依存関係は現時点でもファイルに帰属させられる。クラス・スーパークラス・includeの帰属はまだ記録されていないが、追加コストは低い。プリパスでは各ファイルをマージしながら`path`を保持しており（`merge_discovered_defs`はすでにメソッドに対してこれを行っている）。

## 設計

### 1. アクセサのチョークポイントでファイル単位の依存関係を記録する

軽量な**依存関係レコーダー**を`analyze_file(A)`に通す。Aの解析が`Scope`アクセサを通じてファイル横断シンボルを解決するたびに、レコーダーは解決されたエントリーの**ソースファイル**（`discovered_def_sources`および今後追加するクラス・スーパークラス・includeのソースマップ経由）を記録する。また**ネガティブ**ルックアップ — Aが問い合わせたが解決*しなかった*シンボル — も記録する。後でそのシンボルが追加された場合にAを再チェックしなければならないためだ（§4参照）。

結果として`deps[A] = { Aが宣言またはボディを読み取ったファイル }`、Aのダイジェスト、グローバルフィンガープリントが得られる。プラグインのファイル読み取り（種類4）はio_boundaryのディスクリプタから折り込まれる。

### 2. dependentsインデックスへの反転

`dependents[X] = { A : X ∈ deps[A] }`。ファイル単位の診断キャッシュと並べて永続化する。これがインクリメンタルステップが辿る逆辺である。

### 3. ファイル単位の診断キャッシュ

`(A.digest, {dep.digest for dep in deps[A]}, global fingerprint)`をキーに**各ファイル**の診断結果をキャッシュする（全体実行ではなく） — ADR-45の記録・検証機構（`fetch_or_validate` + `Descriptor#fresh?`）をファイル単位に適用する。AとAが依存しているすべてのファイルが変更されていなければAはキャッシュヒットとなる。

### 4. インクリメンタルステップ: 宣言構造ティアとボディティア

実行時に、前回実行で記録されたファイルダイジェストとプロジェクトを差分比較し、変更セットΔFを求める。そして:

- **宣言構造フィンガープリント**。プロジェクトの*宣言のシェイプ* — すべてのclass / module / def / constの*名前* + そのソースファイル + 各クラスのスーパークラス・include名 — をダイジェストする。ただしメソッドのボディは含まない。ボディ編集ではこれは変わらず、クラス・メソッド・constの追加・削除・リネーム・移動では変わる。
  - **フィンガープリント不変（よくある開発編集 — メソッドボディ）:**シンボルの生成・削除がないため、ネガティブ依存関係は安定している。`ΔF ∪ ⋃_{X∈ΔF} dependents[X]`（変更ファイルとその推移的dependents）だけを正確に再解析し、その他すべてのファイルはファイル単位キャッシュから提供する。
  - **フィンガープリント変化（構造的編集）:**シンボルが出現・消滅・移動した。保守的に範囲を広げる: ΔF ∪ そのdependents ∪ 宣言デルタ内の名前に対して**ネガティブ**依存関係を持つすべてのファイル（`Post#publish`を参照して見つからなかったが今なら見つかるファイル）を再解析する。最初のカットでは構造的変更が起きると全体再解析にフォールバック（fallback）する場合もある。ネガティブ依存関係の絞り込みは後のスライスで行う。

### 具体例

- **`PostsController#create`のボディを編集**。宣言フィンガープリント不変。`dependents[posts_controller]`は空（コントローラーアクションを通じて型推論するファイルはなく、ルートはrails-routesプラグイン経由で*文字列*名として参照し、`routes.rb`を読み込むためコントローラーの戻り値型ではない）。**1ファイル**を再解析。✔「葉」の直観。
- **`Post#publish`のボディを編集**（12個のサービス・コントローラーがPostを使用）。宣言フィンガープリント不変。`Post` + `dependents[post]` = **約13ファイル**を再解析。2,630ファイルではない。✔「関連ファイルのみ」の直観。
- **`ApplicationController`を編集**（すべてのコントローラーの基底クラス）。サブクラスはそれに依存しているため（継承されたdef・推論戻り値）、`dependents`はコントローラーツリー全体 — 正しく再チェックされる。
- **`Post#archive`メソッドを追加**。宣言フィンガープリント変化 → 構造的ティア: `Post` + そのdependents + `archive`をネガティブにルックアップしたファイルを再解析する。

## 健全性 — 難しい部分

問題となる唯一の失敗モード: **依存関係の記録漏れ**があると古い診断結果が提供される。型チェッカーにおいては意図せず偽陽性・偽陰性を生み出すことになる — このプロジェクトが速度のために妥協しない最上位の規律だ。防御策:

- **1つのチョークポイント、監査済み**。ファイル横断の読み取りは少数の`Scope`アクセサを通じて行われる。66箇所のコールサイトではなく*それらのアクセサ*を計装することで、宣言・ボディ・状態の辺を構造的に捕捉できる。アクセサの集合は完全性の監査に十分なほど小さく、ファイル横断アクセサが記録なしで追加された場合に失敗するテストでガードできる。
- **精度より保守的なフォールバック**。不確実性があれば再解析の範囲は広がるのみで、狭まることはない。構造的変更、読み取りを記述できないプラグイン、キャッシュスキーマのバンプ、設定・RBS・バージョン変更 → 全体再解析。過剰解析は遅いが健全であり、不足解析は非健全だ。
- **`--verify-incremental`クロスチェック**。インクリメンタル解析と`--no-cache`の全体解析を両方実行し、バイト単位で同一の診断結果であることを確認するモード（およびCIジョブ）。これがインクリメンタルコンパイラが信頼を得る方法であり、潜在的な健全性バグを、サイレントな古い結果ではなく大きなテスト失敗へと変換する。すべてのスライスの受け入れゲートだ。
- **ADR-45の不変条件を引き継ぐ**。記録されたファイルを再ダイジェストして検証し、検証ゲートは`--no-cache`で実行し、キャッシュエラーが実行を壊すことはない。

## ステージング

1. **帰属 + 記録**。クラス・スーパークラス・includeのソースマップを追加し、`Scope`アクセサに依存関係レコーダーを通し、`deps` / `dependents`とファイル単位の診断エントリーを永続化する。`--verify-incremental`がMastodon + GitLabでグリーンの状態で、デフォルトオフのフラグの後ろにランディングする。
   - **スライス1aランディング済み** — `Analysis::DependencyRecorder`（スレッドローカルアキュムレータ + 無効化時のファストパスが単純な整数読み取りになるようモジュールレベルの有効化カウント、計装済みアクセサはディスパッチのホットパス上にあるため）は、ファイルごとに解析がメソッド・ボディを読み取ったソースファイルと、未解決（ネガティブ）のクロスクラスメソッドルックアップを記録する。`Scope#user_def_for` — メソッド解決・`infer_user_method_return`のチョークポイント — が計装され、既存の`discovered_def_sources`を通じて帰属する。`Runner.new(record_dependencies: true)`でオプトイン、`runner.file_dependencies`として公開。デフォルトはオフ（診断結果はバイト同一、`make verify`グリーン）。
   - **スライス1bランディング済み** —— 祖先エッジ。ファイルごとの`class_sources`マップ（`class_name → Set<宣言ファイル>`、ある名前に`def` / スーパークラス / `include` / 素の宣言を寄与するすべてのファイルから`accumulate_project_index`で構築）が記録スコープにシードされ、`Scope#superclass_of` / `#includes_of`が祖先エッジを解決するときに宣言ファイルの完全な集合を記録する —— つまり複数ファイルにまたがって再オープンされたクラスは、その祖先を読む消費者を*すべての*再オープンに依存させる（設計上の過剰記録、保守的な方向）。これを修正する過程で潜在的なスライス1aのギャップが表面化した: ADR-44の単一アロケーションボディスコープ（`ExpressionTyper#build_user_method_body_scope`、`StatementEvaluator#build_fresh_body_scope`）が`discovered_def_sources`を落としていたため、メソッドボディエッジ（`user_def_for` → `def_sources`）が暗黙的selfの祖先解決呼び出しに対してサイレントに*記録されていなかった*（スライス1aのスペックは明示的なレシーバーのみをカバーしていた）。両ボディスコープは現在`def_sources` + 新しい`class_sources`を持つ;デフォルトはオフ、`make verify`グリーン、診断結果はバイト同一。
   - **スライス1cランディング済み** — §2の反転。`Runner#file_dependents`は、記録された`sources`セット（凍結済み、デフォルトprocは削除 — missing-keyの読み取りがbuilderを再エントリーするのではなくnilを返すように）から`dependents[X] = { A : AがXの宣言・ボディを読み取った }`をオンデマンドで構築する。これがスライス2が編集時に`{X} ∪ dependents[X]`を再解析するために辿る逆辺だ。ネガティブ（`missing`）エッジはここでは意図的に反転しない — 構造的ティア（スライス3）に渡される。ポジティブな**クラス存在**ルックアップエッジ（素の定数をプロジェクトクラスに解決する）も意図的に計装しないままにする: クラスの出現・消滅・移動は宣言フィンガープリントティア（§4）が検出する*構造的*変更であり、単に`Post`を参照するファイルが依存するのは`Post`の*メソッド*（既に`user_def_for`経由で記録済み）であり、素のクラス存在には依存しない — そのためクラスルックアップエッジはボディティアでは冗長であり、その分散した読み取りサイト（`discovered_classes`は1つのアクセサを通さず直接読まれる）はここでチョークポイントのリファクタリングをする価値がない。
   - **このスライスの残り:**永続化（`deps` / `dependents` + ファイル単位の診断エントリー、ADR-45の`Cache::Store#fetch_or_validate`を再利用）と必須の`--verify-incremental`クロスチェック。
2. **ボディティア**。宣言フィンガープリント不変のパス（ΔF ∪ dependentsを再解析）を繋ぎ込む。これだけでMVCの恩恵が得られる。
   - **健全性コアランディング済み** — `Analysis::Incremental`は副作用のない集合演算を担う: `affected(changed, dependents)`（ボディティアが再解析する推移閉包 = changed ∪ そのdependents）と`changed_files(before, after)`（`Diagnostic#to_h`経由の構造的なファイル単位診断差分）。スペックは実際の実行を動かして健全性*プロパティー*をエンドツーエンドで確認する: 宣言フィンガープリントが変わらない葉のボディ編集はすべての診断変更を`affected`に閉じ込め、無関係なファイルのキャッシュ済み診断には手を付けない（葉編集の恩恵）。Runnerと独立しているため、不変条件はキャッシュ・サブセット機構なしでユニットテストできる。
   - **実証的知見（ティアへの示唆）**。Rigorの偽陽性の<ruby>規律<rp>（</rp><rt>discipline</rt><rp>）</rp></ruby>により、推論されたクロスファイル戻り値型がdependentの診断を引き起こすことはほぼない: `String#+ Integer`、推論されたレシーバーへの`Integer#upcase`、`if <inferred-bool>`はいずれもファイル境界を越えてサイレントのままだ。したがって実際には、ボディ編集では`changed ≈ ΔF` — dependentsの再解析は保守的な*保険*（健全性を保つための再チェック）であり、新しい診断の頻繁な発生源ではない。これが葉のコントローラー → 1ファイル勝利がよくあるケースになる理由だ。
   - **サブセット解析フックランディング済み** — `Runner.new(analyze_only:)`はパスのコレクションを受け取る。プロジェクト全体のプリパスは全ファイルに対して走り続け（クロスファイルインデックスを完全に保つ）、`target_files`が解析対象セットを渡されたパスに絞り込む。サブセット実行における特定ファイルの診断結果は全体実行のものとバイト同一（スペック`runner_subset_analysis_spec.rb`）なので、ボディティアは影響を受けるクロージャを再解析してその結果が全体解析と一致すると信頼できる。プールモードはフィルタを無償で引き継ぐ（`target_files`をディスパッチする）。
   - **インメモリオーケストレーターランディング済み** — `Analysis::IncrementalSession`がすべての部品を組み合わせる: `#baseline`は記録ありで全体解析を実行し、解析された各ファイルの診断結果（キャッシュ）・コンテンツダイジェスト・ソースセットを保持する。`#recheck`はファイルを再ダイジェストし、ΔFを計算し、`analyze_only:`経由で`ΔF ∪ dependents[ΔF]`だけを再解析し、残りはキャッシュから提供する。スペックは受け入れプロパティーをエンドツーエンドで確認する: 実際のオンディスク葉編集後、`#recheck`がマージした診断結果は、編集済みツリーの全体再解析のものとバイト同一（ソート済み）であり、変更されたファイル1つだけを再チェックする — さらに連続する編集をまたいで正しく動作し続ける（キャッシュ + dependentsインデックスは各ラウンドで`Incremental.invert`経由でリフレッシュされ、現在は`Runner#file_dependents`と共有される）。`Runner#analyzed_files`はマージが影響を受けるクロージャを差し引く解析済みセットを公開する。実行レベルの診断ストリーム（gem-RBSの`info`、`.rigor.yml`をキーとする）は各実行で新たに再計算され、ファイル単位キャッシュから除外される。
   - **`--verify-incremental` CLIゲートランディング済み** — `rigor check --verify-incremental [paths]`はベースライン（記録あり）を実行し、1ファイルおきにサブセットとして再解析してそれ以外をファイル単位キャッシュから提供し、マージされた診断結果が`--no-cache`全体実行とバイト同一であることを確認する。PASSの1行か、差異のある診断結果を表示して0/1で終了する。これはサブセット解析とキャッシュ提供の両パスを実際のプロジェクト上で全体実行のオラクルに照らして検証する（ソース編集不要）。Rigor自身の`lib`（261ファイル中131を再解析、全体と一致）で検証済み。`IncrementalSession`がそのエンジンであり、編集駆動の*dependents健全性*（実際の編集がその診断変更を影響を受けるクロージャに閉じ込める）はスペックスイートでカバーされ、CLIゲートがユーザーのツリーでできない編集を作り出せる。
   - **CIゲート組み込み済み** — `make check-incremental`はRigor自身の`lib`（261ファイル中131を再解析）と`plugins/*/lib` + `examples/*/lib`ツリー（141ファイル中71を再解析）に対して`--verify-incremental`を実行し、CIのセルフチェックジョブが`check` / `check-plugins`と並んで（コールドバリアント）実行する。意図的にローカルの`make verify`ファストパスには含まれない — ターゲットごとに約3回の解析（lib + pluginsで約39秒）を実行し、毎回のローカル検証には遅すぎる。そのため機構を構築するにつれ、健全性ゲートはすべてのCI実行で維持される。
   - **このスライスの残り:**インプロセスキャッシュをディスクへ移行（ADR-45の`Cache::Store#fetch_or_validate`、プロセスやCIをまたいでセッションを維持）+ ユーザー向けの`--incremental`*スピードアップ*フラグを繋ぎ込む（ゲートが機構の健全性を証明し、スピードアップフラグがその報酬）+ CIゲートをMastodon + GitLab調査ツリーに拡張する。
3. **構造的ティア — ネガティブ依存関係追跡（ランディング済み）**。編集で*出現した*シンボルは、そのシンボルをルックアップして見逃した消費者を再チェックしなければならない。ほとんどの見逃しはすでにカバーされている。見逃した*クラスメソッド*の解決はレシーバーの祖先（`superclass_of` / `includes_of`）を辿り、チェーン上のすべてのクラスに正の祖先エッジを記録するため、そのうちどれかにメソッドが追加されると消費者が再チェックされる。見逃した*定数*は診断を生じないため、古くなるものがない。ギャップは**トップレベル呼び出し**（`call.unresolved-toplevel`、ADR-34）だった: `helper()`にはクラス祖先がないためミスはエッジを記録せず、他の場所で`helper`を定義しても呼び出し元の診断が古いまま残った（プローブが捕捉した意図せず生じた偽陽性で、スペックスイートではまだ検出されていなかった）。`Scope#top_level_def_for`を計装し、解決時に**正の**シンボル粒度エッジ（`<toplevel>#name`、ボディ・削除編集が呼び出し元を再チェック）とミス時に**ネガティブな**`toplevel:name`エッジを記録することで解消した。セッションは`missing`を吸収して`negative_dependents`へ反転し、再チェック時に変更ファイルで*出現した*すべてのシンボルのネガティブdependentsで影響クロージャを広げる（`Incremental.{appeared_symbols,negative_closure}`）。メソッドのネガティブはクラス修飾の完全一致（`method:C#m`）でマッチングし、多層防御とする。祖先エッジが主要カバーとして残る。

   2つ目のギャップが表面化した。**`def.override-*`チェッカーがクラスグラフを直接読む**（`CheckRules#known_user_class?`）が、レコーダーの`Scope`アクセサを経由せず、祖先名がプロジェクトクラスに解決しないと*短絡する* — したがって`NewBase`が未定義の`class ASub < NewBase`はエッジを記録せず、後から`NewBase`（またはそれが`include`するモジュール）を定義しても`ASub`の欠けているオーバーライド診断が古いまま残った（同ファイルでのクラス追加編集と下記のファイル追加ティアの両方で発生する）。`resolve_override_ancestor_name`が失敗した箇所で**ネガティブクラスエッジ**（`class:Name`、非修飾）を記録し、編集で出現したすべての**クラス**のネガティブdependentsで再チェックを広げることで解消した — ファイルごとの宣言クラスセット（`Runner#class_declarations`）に対する`Incremental.appeared_classes`で、シンプルな名前一致。継承メソッドへの橋渡しとなる*空の*クラスもカバーされる。なぜなら出現したクラスがそれ自体が欠けているリンクだからだ（その後の再解析がそれ以降を辿る）。`--verify-incremental`はバイト同一を維持し、`IncrementalSnapshot`ペイロードは`missing` + `class_decls`を持つ（SCHEMA 4）。

   **ファイルの追加・削除（ランディング済み）**。スナップショットフィンガープリントは展開されたファイルリストではなく解析*ルート*（パス引数、例: `["lib"]`）をキーとするため、ルート以下でのファイルの追加・削除ではスナップショットが無効にならない。`#recheck`は現在のファイルセットと前回実行のものを照合する。**追加された**ファイルは解析されてそれ以前の状態を持たないため、そのシンボル・クラスすべてを*出現したもの*として寄与し、上記のネガティブエッジがそれが今や定義する名前に依存していたものを再チェックする。**削除された**ファイルはファイルごとのすべてのマップ（キャッシュ・ダイジェスト・ソース・フィンガープリント）から削除され、その正のdependents（`@dependents[removed]`、シンボルと祖先の消費者の和集合を捕捉）が再チェックされるため、トップレベルメソッドを失った呼び出し元やスーパークラスを失ったサブクラスが正しく再発火する。added-toplevel / added-superclass / removed-toplevel / removed-superclass / added-unrelated編集（インプロセスとクロスプロセス）で全体実行に対してバイト同一を検証し、**実際のOSSライブラリで大規模検証済み**（`liquid/lib` 64ファイル、`mail/lib` 111ファイル） — すべてのremove + re-addがバイト同一を維持し、設計が目指す葉vsハブの精度を示した（`Liquid::Block`の削除はその9dependentsを再チェック、葉フィルタの削除は0を再チェック）。注: [`docs/notes/20260605-adr46-file-add-remove-scale-validation.md`](../../notes/20260605-adr46-file-add-remove-scale-validation/)。

   **残り（需要ゲート）:**ボディ + 構造的ティアについては残りなし。唯一のオープンレバーは推論戻り値ファンアウトを抑制する戻り値型*サマリー*（§ リスク）— **ROIが低い**: 上記の実証的知見では推論されたクロスファイル戻り値がdependentの診断を引き起こすことはほぼないため、スライス4のボディフィンガープリントトリガーはすでに安価な保険となっている。また単純に実装すると非健全（ボディ編集は戻り値型だけでなくクロスファイルクラス状態を通じて消費者に影響しうる）。ファンアウトが実際に問題になると示す実測ケースが得られるまで延期。
4. **シンボル粒度（オプション）**。ファイルレベルの依存関係を`(file, symbol)`に精緻化し、1つのモデルメソッドを編集したときにモデルの全呼び出し元ではなく、*そのメソッド*の呼び出し元だけを再チェックするようにする。

## 却下・延期された代替案

- **全体実行キャッシュのみ（ADR-45）**。粗粒度の前身。このADRはその置き換えではなく改良版であり、変更なしのケースでは引き続きADR-45が1回のルックアップで勝る。
- **純粋なタイムスタンプ・mtime無効化**。ファイル横断辺にわたって非健全（mtimeが変わらないボディ編集、またはディスク上で触れられていないdependentを見逃す）であり、推論戻り値の依存関係も無視する。
- **Salsaスタイルのファクトレベルメモ化**（個々のファクトの値を記録し、ファクト単位で再検証）。精密な理想だが、クエリレイヤーを*エンジン全体*（66箇所すべての読み取り点 + ディスパッチャーティア）に通す必要があり、はるかに大規模な書き直しになる。アクセサチョークポイントでのファイルレベル依存関係は、少ないサーフェスでほとんどの恩恵を得られる。シンボル粒度（ステージ4）がそこへの漸進的なステップだ。
- **宣言ティアなしで、全体の逆到達可能セットを編集ごとに再解析**。健全ではあるが、推論戻り値を変えないボディ編集でdependentsを不必要に再チェックする。宣言ティアと（後の）戻り値型サマリーがそれを剪定する。

## リスク

- **記録の完全性は正確さの不変条件であり、パフォーマンスのノブではない**。`--verify-incremental`ゲートは必須であり、オプションではない。
- **推論戻り値の揮発性**。メソッドの推論戻り値を変えるボディ編集はすべての呼び出し元に波及する。深いコールグラフは「影響を受けるクロージャ」を大きくしうる。戻り値型**サマリー**（依存先の推論サマリーが変わった場合のみdependentを再チェックする）がこれを抑えるが、それ自体が独立したスライスだ。
- **永続化コスト**。`deps` / `dependents` / ファイル単位エントリーはプロジェクトとともに肥大する。これらはADR-6の退避なしバックエンドを共有しており、サイズに関する方針が必要だ（ファイル単位エントリーは安定したパスをキーとしているため、ADR-45のスロットと同様に蓄積ではなく上書きになる）。
