ステータス: **Accepted、2026-06-11。スライス1（定数引数の有界アンロール）とスライス2（不動点の戻り値型サマリー）はともに2026-06-11に実装**。どちらも精度加算的であり、新しいdiagnosticもなく、偽陽性のサーフェス（surface）も生まない。すべての枯渇パスは現在の挙動へとデグレードする。スライス1はフューエル有界の値キー付きガード（`RECURSION_UNROLL_FUEL = 32`、64ノードの値サイズ上限）、`BudgetTrace::RECURSION_UNROLL_FUEL`カウンタ、そして畳み込まれた定数フレームをメソッド本体の中で顕在化させる値ピン留めself呼び出しの採用（`factorial(5) → Constant[120]`）を出荷する。スライス2はKleene不動点の戻り値型サマリー（`RECURSION_FIXPOINT_CAP = 3`、最終反復での値ピン留めの広げ、`BudgetTrace::RECURSION_FIXPOINT_CAP`カウンタ）を出荷する。サイクル内の再入は`Dynamic[top]`の代わりに想定サマリー（`bot`を種とする）を返すので、非定数の引数で呼ばれた再帰的なStringビルダーは`String`を返し、`factorial(x : Integer)`は`1 | Integer`を返す。どちらも`Dynamic[top]`の寄与を持たない。Mastodonの`app/models`もhamlの`lib`も、スライス2以前のベースラインとバイト単位で同一である。

アーキタイプ: deliberative。ステークス: mid（エンジンの戻り値型推論コア。各機構は構成上FPニュートラルだが、ADR-41 WD4により譲歩不可とされる再帰停止機構に触れる）。

## コンテキスト

再帰の再入ガード（`ExpressionTyper#infer_user_method_return`、`lib/rigor/inference/expression_typer.rb`のおよそL1508）は`(receiver, method)`のみをキーとし、実効深さ1で発火してサイクル内呼び出しに`Type::Combinator.untyped`を返す。ADR-24 WD5はこのキーを意図的に選んだ──引数の*型*でキーを取ると、`module_function`モジュールを介した相互再帰が無限に再帰してしまった（`SystemStackError`）。戻り値型の精度に対する帰結はこうだ。

```ruby
def factorial(n)
  n <= 1 ? 1 : n * factorial(n - 1)
end
factorial(5)  # Integer       (the in-cycle factorial(4) is Dynamic[top];
              #                Integer#* happens to absorb it here)
factorial(0)  # Constant[1]   (constant-folded condition skips the cycle)
```

`Integer`が生き残るのは、`Integer#*`のRBSが`Dynamic`オペランドに対して全域的（total）だからにすぎない。結果を*構築する*再帰メソッド（`String`を返すレンダラー、`Array[T]`を返す木の走査）は、`Dynamic[top]`と結合された戻り値を得る──再帰的な寄与は未解析であり、呼び出し側はシグネチャが存在しないまま`untyped`汚染を見ることになる。

ADR-41 WD5はすでに`recursion_depth`を、ハードな**停止フロア**（配線済み、深さ1）と省略可能な**精度アンロール深さ**（先送り、需要駆動）に分割していた。本ADRはその需要の到来であり、加えてWD5が想定していなかった第2の機構──再帰的戻り値そのものに対する不動点──を伴う。本当の精度はそこに宿る。定数引数の帰結（`factorial(5) → Constant[120]`）は依頼者によって明示的に**低価値**とされている。スライス1はガードキーの安全な一般化として、またスライス2が再利用する`BudgetTrace`配線として地位を得るのであって、定数畳み込み機能としてではない。

## 決定

両機構を順に、一つの基準のもとで採用する。

> **再帰戻り値の精度機構は、その枯渇パスが現在の広げとバイト単位で同一である場合にのみ許容される**──フューエル切れ、反復上限到達、値の膨張、非収束のいずれでも、すべての出口は深さ1ガードが現在返すもの（`untyped`）を返す。精度はハードかつ設定不可の停止エンベロープ（ADR-41 WD4）の内側でのみ厳密に得られる。いかなるノブも非停止をオプトインできない。

### WD1 — スライス1: フューエル付き定数引数アンロール

再入した呼び出しの束縛引数が**すべて**値ピン留め（`Constant[…]`、またはそれらの`Tuple`）であるとき、ガードキーは`(receiver, method, argument values)`へ拡張され、相異なる定数フレームが再帰できるようになる。ADR-41 WD4の意味でガードを数える、2つのハード上限がある。

- **フューエル** — 最外の入りあたりのアンロール総フレーム数、デフォルト32。
- **値サイズ上限** — ピン留め値が小さな構造的サイズ（例: 64ノード）を超えるフレームは拡張の対象外となる。

枯渇または対象外化はプレーンな`(receiver, method)`ガード──すなわち現在の結果──にフォールバックする。非定数の引数はこのパスを決して取らない。新しい`BudgetTrace`カウンタ（`RECURSION_UNROLL_FUEL`）。デフォルトON: これらの上限は停止ガードであって、計測ゲート付きの精度予算ではない（ADR-41 WD3は適用されない）。

**支配的なクランプ規則（省略不可、必須）**。定数引数アンロールは*完全に値ピン留めされた*結果しか顕在化させてはならない。それ以外の帰結はプレーンガードの`untyped`とバイト単位で同一でMUST。2つのクランプがこれを強制し、いずれも精度をアンロール自身のエンベロープに閉じ込める。

1. 拡張された（値キー付きの）パスを取ったが、そのプレーンな`(receiver, method)`シグネチャがすでにガードスタックに載っている──プレーン形で、あるいは拡張フレームのプレーン部分として──フレームは、*スライス1以前ならガードされていた*はずである。その本体が評価された後、結果が完全に値ピン留めでなければ`untyped`へクランプし戻す（`RECURSION_GUARD`ヒットを1回計上）。ガードされなかったはずのフレームは、スライス1以前の結果パスをそのまま保つ。
2. 値ピン留めself呼び出しの*採用*（メソッド本体の中で畳み込まれた定数が顕在化すること）は、アンロールが進行中である間に限り許容される──すなわちガードスタックがすでに拡張フレームを担っている場合だ。メイン検査の走査ではスタックが空なので、採用はちょうど`self_type.nil? || Bot`に戻る。

> **コーパスFAILの注記（2026-06-11）**。 WD3のコーパスゲートは、最初のスライス1ランディングが漏らした3つの新しいdiagnosticを捕捉した。mastodonの`account.rb:496`の`undefined method 'in?' for :denied`（self呼び出しの戻り値で、以前は`Dynamic[top]`だったが、具体的なSymbolユニオンへ解決され、ActiveSupportのギャップが誤ディスパッチした）、mastodonの`base_action.rb:32`の常に真、hamlの`children_compiler.rb:112`の常に偽（`find_else_index`が常にnilへ誤畳み込みされた──本体評価器は反復ブロック内の非局所`return`をモデル化せず、無差別な採用がその誤ったピン留め値を顕在化させた）。根本原因: 無差別な`adoptable_self_call_result?`の拡幅がプロジェクト全体で*あらゆる*完全に値ピン留めされた結果を採用し、`Dynamic[top]`ゲートが覆い隠していた本体評価器の既知の盲点を露呈させた。上の2つのクランプが修正であり、その後は両コーパスともスライス1以前のベースラインとバイト単位で同一である。

### WD2 — スライス2: 不動点の戻り値型サマリー

サイクル結果の`untyped`を、既存のガードスタックと並べて、`(receiver, method)`シグネチャごとに**`bot`からのKleene反復**で置き換える。

1. 最外の入りは、`INFERENCE_GUARD_KEY`と同じようにキー付けされたスレッドローカルのテーブルに、想定サマリー`bot`を種として置く。
2. サイクル内の再入は、（`untyped`の代わりに）現在の想定サマリーを返す。
3. 本体が評価された後、**その間にガード/サマリーが参照された場合に限り**: 計算された戻り値が想定と整合する（想定に包摂される）なら、不動点に到達している。そうでなければ想定を`join(assumption, computed)`へ更新し、本体を再評価する。
4. 反復上限3。許される最後の反復では、収束を強制するためにjoinが**値ピン留め構成要素をその名前的基底へ広げる**（`Constant[1]` → `Integer`）。なお不安定なら、サマリーは`untyped`へ崩れる──現在の挙動だ。

`factorial(x : Integer)`の場合: ラウンド1は`bot`を想定し、本体は`1 | (x * bot) = Constant[1]`を生む。ラウンド2は`1 | Integer = Integer`を生む。ラウンド3が確認する。結果は`Dynamic`寄与のない`Integer`であり、`String`を構築するレンダラーは`String | Dynamic[top]`ではなく`String`を返すようになる。

`bot`は正しい種であり、健全性のリスクではない。*再帰しかしない*メソッドは本当に決して返らない（サマリーは`bot`のまま留まり、常に発散する形を`adoptable_self_call_result?`はすでに安全として扱う）。相互再帰は不変のガードスタックによって深さ有界のままである。サマリーはサイクル内の結果を*より*精密にするだけで、決してより深くはしない。

**実装ノート（2026-06-11）**。スライス2のランディング中に2つの設計上の論点が定まった。

- *サマリーテーブルはプレーンな`(receiver, method)`シグネチャでキー付けされる*。スライス1の値拡張版ではない。そのため定数引数アンロールとプレーンな再帰は一つのサマリーを共有する。プレーンシグネチャの最外フレームがテーブルエントリーを所有して反復を回し、ネストした拡張フレームは本体を一度評価して所有者に反復させる。エントリーはガードスタックが空に排出されたときに落とされる（入りごとのフューエルリセットを鏡映する）。シグネチャごとの*参照済み*フラグが不動点をゲートする: 単に`infer_user_method_return`を共有するだけの非再帰本体は、サマリーを決して参照しないので、追加の反復なしに計算した型をそのまま返す。
- *WD4の合成は狭められた。* WD4のステップ5は、スライス1のフューエル枯渇**と**クランプのフォールバックがともにサマリーへルーティングされると想定していた。サイクル内のガードヒットとフューエル枯渇したプレーン再入は、（`consult_summary`を介して）確かにサマリーへルーティングされる。しかしスライス1の**クランプ**（`clamp_unroll_result`）は素の`untyped`を返し続ける: それは*信頼できないアンロール値*に対する健全性のバックストップであるのに対し、進行中のサマリーは反復途中のKleene*下*界だからだ──クランプをそこへルーティングすると、`recursive_unroll_clamp`フィクスチャで実際の基底ケース構成要素を落としてしまった（`1 | "s"`が`"s"`へ崩れる）。したがってクランプは保守的な`untyped`上界に留まり、サマリーを参照するのはガード/フューエルのパスだけである。
- 相異なる2つのシグネチャにまたがる相互再帰は停止するが、一般には精密なサマリーへ畳み込まれない（各シグネチャが自分のKleene反復を所有する）ので、`Dynamic[top]`へ健全にデグレードする──現在の挙動であり、荷重を担う性質は停止である。

**健全性修正 — `bot`崩れ（2026-06-11）**。スライス2のランディング（コミット`36c0cfaa`）には健全性バグがあった: 到達可能な非再帰の出口を持つパススルー再帰が、本当の戻り値型ではなく`bot`（「決して返らない」）と型付けされうる、というものだ。2つの機構が組み合わさっていた。

1. *サイクル内サマリーが採用されない。* Kleene想定が`bot`の種を越えて値ピン留め型（例: `Constant[:done]`）まで育つと、その想定を返すサイクル内self呼び出しの結果が`adoptable_self_call_result?`（`Bot`またはアンロール内ピン留め値しか採用しない）に失敗し、`Dynamic[top]`へ漏れ戻った。そのため反復は精密な型へ決して収束せず、`untyped`へと流れた。修正: `adoptable_self_call_result?`は、アクティブな不動点サマリーの生きた想定オブジェクトである型（`active_fixpoint_summary?`）も採用するようになり、想定が反復をまたいで本体へ伝播し戻れるようにした。

2. *`bot`の種での自明な収束。*呼び出し地点の引数が狭い値集合（`some_int : 1 | 2 | 3`）を担うとき、`n <= 0 ? :done : passthrough(n - 1)`のような基底ケースガードが常に偽へ定数畳み込みされ、`:done`分岐が枝刈りされ、本体は`bot`を計算した（再帰分岐は`bot`の種を返し、基底ケースは消えている）。すると収束チェック`joined == assumption`が種の地点で自明に成立し（`bot == bot`）、`bot`を返した──不健全であり、ADR-47の到達可能性/常に偽のdiagnosticに偽陽性の源を供給する。修正（`resolve_bot_collapse`）: ある反復が参照済み本体に対して`bot`を計算したら、パラメータを広げた本体スコープ（`1 | 2 | 3` → `Integer`）で不動点を**一度**回し直す。これで末尾の基底ケースの枝刈りが解かれる（`passthrough → :done`）。広げた本体が*なお*`bot`を計算するが、本体が到達可能な明示`return`──その値を末尾専用の本体評価器が決して結果に畳み込まない（`pick`の`return nil`）──を含む場合は、`bot`ではなく保守的な`Dynamic[top]`（スライス2以前の観測可能値）へフロアする。非再帰の出口を本当に持たない本体（`spin`）は`bot`を保つ。

修正後、Mastodonの`app/models`もhamlの`lib`も、スライス2以前（`2ba608a0`）のベースラインとバイト単位で同一のまま留まる。新しいリグレッションアサート（`passthrough → :done`、`pick → Dynamic[top]`）が`recursive_fixpoint_summary.rb`に着地する。フィクスチャのFactorial / Builderのアサートは、不動点の弁別子ではなくRBS吸収のアンカーであることを示すよう訂正される（再帰分岐は`Integer#*` / `String#+`の式であり、その戻り値はサイクル内の型にかかわらずRBSで固定される）──弁別するのは*素の*self呼び出し分岐だけである。

### WD3 — コストエンベロープとゲート

本体の再評価は高々3回であり、実際にサイクルに参加するメソッドに対してのみ行われる。ストレスコーパスは既知である: hamlはガードを421回、jbuilderは126回発火した（ADR-41の調査）。スライスごとのゲート: `make verify`＋`make bench-perf`がニュートラル以上 ＋ コーパスのdiagnostic比較（Mastodonの`app/models`および/またはRedmine）で、diagnosticの差分が**ゼロか厳密に除去のみ**でなければならない──新しいdiagnosticはどれも判断の余地なくブロッカーである。

### WD4 — 順序と相互作用

スライス1が先に着地する（より小さく、独立にテスト可能）。スライス2が着地すると、スライス1のフューエル枯渇パスは`untyped`へ直行する代わりに不動点サマリーへフォールバックする──両者は合成され、サマリーがより良いフロアとなる。

## 却下／先送りした代替案

- **すべての呼び出しでガードを引数の*型*でキー付けする**。却下──ADR-24 WD5の`SystemStackError`を蒸し返す。スライス1は引数の*値*のみでキー付けし、これはフューエルで有界化される。
- **枯渇時にいまエラー／diagnoseする**。 ADR-41 WD2の`static.*`サーフェスへ先送り。これらのスライスはdiagnosticを一切出さない（FPの規律: 精度加算的のみ）。
- **フューエル／反復上限をいまユーザー設定可能にする**。先送り──ADR-41 WD4は、あるプロジェクトが束縛していると示すまで停止ガードをハードに保つ。`RIGOR_BUDGET_TRACE`カウンタがエビデンスのチャネルを提供する。
- **メモ化されたクロスコールのサマリー（引数型シグネチャごとのキャッシュ）**。先送り──性能のリファインメントであり、WD3のベンチゲートが実コーパスで再評価コストを示した場合に限り、その無効化の複雑さに見合う。

## 帰結

- 再帰メソッドが`Dynamic`汚染のないシグネチャ不要の戻り値型を得る──キャンプ（b）の「無注釈のRubyで動く」という約束（ADR-41 WD1）が再帰へ拡張される。これは調査したキャンプ（b）のどのツールも精度の面では行っていない（TypeProfは深さ5で`untyped`へ広げる）。
- `docs/type-specification/inference-budgets.md`の §「実装ステータス」にある配線済みガードのテーブルは、各スライスで更新されなければならない（深さ1の行にアンロールとサマリーの限定子が加わる）。ADR-41 WD5の「精度アンロール: 先送り」ステータスは「具現化済み（ADR-55）」へ反転する。
- 新たな保守サーフェス: 不動点ループが停止のストーリーの一部になる。その反復上限は荷重を担っており、ハードに保たれなければならない。

## 他のADRとの関係

- **ADR-41** — これはWD5の精度アンロール需要の到来である。WD4のハード停止規則が本ADRの許容基準であり、`BudgetTrace`チャネルが再利用される。
- **ADR-24** — WD5のガードキー決定は保たれる。スライス1はフューエルのもとで値ピン留めフレームに対してのみそれを狭める。
- **ADR-5 / ADR-48** — これらのスライスが内側に留まらなければならない、精度加算的でゼロFPのエンベロープ。
