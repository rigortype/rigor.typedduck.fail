**ステータス:** Accepted — ボディスコープチェーンの折り畳みとアロケーション衛生（allocation hygiene）が着地済み。ミュータブルプーリングは却下。フィールド再グループ化は優先度引き下げ（オブジェクトシェイプのベンチマーク（benchmark）により、サイズ削減にはなるがアロケーション数削減にはならないことが判明）。

ADR-37でプラグインコントリビューションパスの過剰アロケーション（churn）が解消された（コレクター: `dynamic_returns`スナップショットメモ、遅延累積、`Registry#contribution_index`）後、プラグインを多用する実アプリ（GitLab FOSS、`app/{controllers,mailers,workers,services}` = 2,630ファイル、11プラグイン、`cwd=gitlab`による忠実な実行）で`rigor check`をプロファイルした結果、解析は依然として**アロケーションバウンド**であることが判明した。残存する最大の2つのアロケーションサイトは*構造的*なものだ: ナローイング（narrowing）のたびに再構築されるイミュータブルな`Scope`と、ディスパッチ（dispatch）のたびに構築される`CallContext` `Data`である。

本ADRは、魅力的な構造的リライト（ミュータブルかつプール型のスコープ／コンテキスト）が偽陽性（false positive）の規律（discipline）上の理由から**却下**された経緯、段階的なアロケーション衛生として**着地したもの**、そして**承認済み**（ただし先送り）のフィールド再グループ化の方向性を記録する。

根拠となる資料: [`docs/notes/20260604-gitlab-plugin-contribution-allocation.md`](../../notes/20260604-gitlab-plugin-contribution-allocation/)および[`docs/notes/20260604-mastodon-allocation-profile.md`](../../notes/20260604-mastodon-allocation-profile/)。

## コンテキスト

ADR-37のプラグイン過剰アロケーション解消後の、忠実なGitLab `:object`プロファイル（合計約2億2200万オブジェクト、`--workers 0 --no-cache`）。アロケーション上位サイト:

| alloc % | サイト | 性質 |
|--:|---|---|
| 約7.5 % | `CallContext.build` + `.new` + `Data#initialize` | ディスパッチごとに`Data`を1個 |
| 約4.3 % | `Scope#rebuild` | ナローイングごとに`Scope`を1個 |
| 約2.6 % | `StaticReturnRefinements.owners_for` | リファインメント対象外のメソッドごとに`[]`を新規作成 |
| 約2.5 % | `MethodDispatcher.discovered_method_lookup` | ディスパッチごと |
| 約2.3 % | `RbsDispatch.receiver_descriptor` | ディスパッチごと |

意思決定を方向付ける2つのファクト（fact）:

- **`Scope`は契約（contract）によりイミュータブルである**。制御フロー解析（[`docs/type-specification/control-flow-analysis.md`](../../type-specification/control-flow-analysis/)）は、ナローイングされた`Scope`が*値*であることに依存している: 分岐はスナップショットを保持でき、合流点は2つのスコープを比較でき、ある分岐で確立されたファクトが別の分岐に漏れることはない。`with_*`はすべてフリーズされたコピーを返す。22個のフィールドのほとんどは共有済みフリーズハッシュなので、コストはラッパーオブジェクトであってデータそのものではない。
- **ディスパッチは再入可能（re-entrant）である**。`MethodDispatcher.dispatch`は再帰する: ブロックの畳み込みは要素ごとに再ディスパッチし、ユーザーメソッドの戻り値推論はメソッド本体を評価し（そこで再びディスパッチが起きる）、ユーザークラスのフォールバックは`public_only`で再ディスパッチする。`CallContext`は複数のスタック深さで同時にスタック上に存在する。

## 決定

### 却下 — ミュータブルかつプール型のScope / CallContext

自明な「修正」はアロケーションを止めることだ: スレッドローカルなミュータブル作業用`Scope` / `CallContext`を保持し、ナローイング／ディスパッチのたびにリセットして再利用する。**却下**。ディスパッチとナローイングは*再入可能*であるため、再利用される単一のミュータブルオブジェクトが複数のスタック深さで同時に生存してしまう — 内側の再ディスパッチが外側フレームのコンテキストを上書きし、「スナップショット」スコープをキャプチャした分岐は、合流点でそれが書き換えられるのを目の当たりにすることになる。障害モードはクラッシュではなく**サイレントな誤ったナローイング**だ — 正常なコードに対して偽陽性を生み出すバグの典型であり、このプロジェクトが最悪ケースの静的読み取りよりも高く評価する価値観（偽陽性の<ruby>規律<rp>（</rp><rt>discipline</rt><rp>）</rp></ruby>の最上位）に反する。スタック深さをキーとする適切なサイズのプールなら安全性を回復できるが、取り除こうとしたブックキーピング（とアロケーション）のほとんどを再導入する。アロケーションはイミュータブルかつ再入可能な設計の代償であり、その設計はナローイングの健全性にとって不可欠（load-bearing）だ。採用する価値はない。

### 承認（着地済み）— 段階的なアロケーション衛生

イミュータビリティ契約には触れず、各サイト周辺の*回避可能な*アロケーションを削減する:

- **`StaticReturnRefinements.owners_for`**は、（圧倒的に多い）ミスのケースで`[]`を新規作成する代わりに、共有済みフリーズ定数`NO_OWNERS`を返す — これはディスパッチのたびに参照される。
- **`CallContext.build`**は`new(receiver:, …)`のキーワード形式ではなく、*位置引数*による`new`（`Data.define`のフィールド順）で構築することで、キーワード形式がディスパッチごとに確保するキーワードハッシュを回避する。

忠実なA/Bテスト（cwd=gitlab、11プラグイン、`--no-cache`）: 2億2850万→2億2240万オブジェクト（−2.7 %）、実時間122.5 s → 120.5 s、診断結果バイト同一（2,323 / 38）。`make verify`グリーン。小さな改善だが、無償かつリスクゼロだ — 同じ「共有空コレクション／位置引数構築」の衛生は、ホットパスが使い捨ての空コレクションやキーワードハッシュをアロケートしている箇所すべてに適用できる。

### 承認（着地済み）— ボディスコープの`with_*`チェーンを折り畳む

`Scope#rebuild`の呼び出し元内訳は意外だった: 制御フローのナローイングが**支配的ではなく**、*ボディスコープの構築*が支配的だったのだ。`ExpressionTyper#build_user_method_body_scope`（ユーザーメソッド呼び出し推論ごと）と`StatementEvaluator#build_fresh_body_scope`（クラス／メソッド本体ごと）はそれぞれ、**約12〜13回の`with_*`呼び出しをチェーンする**ことで新鮮なスコープを構築していた。毎回、中間のフリーズ済み`Scope`をアロケートして`rebuild`のフィールドコピーを再実行する — 1個のスコープを構築するために十数個の使い捨てスコープが生まれていた。

両メソッドとも、すべてのフィールドを一度に設定する**単一の`Scope.new`**でスコープを構築するようになった。ボディスコープは空のファクトストア／ナローイングセットから始まるため、チェーンされた`with_local`の無効化はno-opであり、`project`/`self_type`セッターは単なるフィールド書き込みだった — したがって1回の`new`はチェーンとバイト同一だ。忠実なA/Bテスト: 2億2240万→2億1330万オブジェクト（−4 %）、**GC実行回数362→256（−29 %）**、診断結果バイト同一（2,323 / 38）。実時間の改善がアロケーション減少を上回るのは、チェーンの折り畳みによってボディスコープごとに約12回の`rebuild`メソッド呼び出し（とキーワード処理）も除去されるためだ。

### 調査済み、低優先度 — `Scope`のイミュータブルなプロジェクトフィールドを再グループ化する

当初承認されていたアイデアは、`Scope`の約15個のイミュータブルなプロジェクト全体フィールド（`discovered_*`、`run_project_pre_passes`内で一度だけ構築）を1つの共有`ProjectScope`の背後にまとめ、`Scope`を約22フィールドから約8フィールドに縮小するというものだった。**対象Ruby（4.0.5）でのマイクロベンチマークにより、これがアロケーションを削減するという前提は否定された:**オブジェクトシェイプ／可変幅アロケーションにより、3、8、16、22、または24個のivarを持つフリーズ済みオブジェクトはすべて**ちょうど1個**のオブジェクトをアロケートする — 第2のivarバッファへのスピルは発生しない。したがって再グループ化は、遅延スイープコストを左右するアロケーション*数*ではなく、`Scope`ごとのヒープスロット*サイズ*のみを削減する。`Scope#rebuild`のサンプルシェアは変わらない。アロケーション数削減の真の成果は上述の構築チェーンの折り畳みだった。再グループ化は*メモリフットプリント*のレバー（スロット縮小→ヒープの充填が遅くなる→GCサイクルが減る）として残るが、大規模な変更（`scope.discovered_*`リーダーすべてに影響）であり、見返りは今や限定的であることが判明しており、仮定するのではなく計測してから投資する必要がある。

## 結果

- イミュータブルな`Scope`／再入可能ディスパッチの契約が、ナローイングの健全性にとって不可欠であることが*文書化*された。これにより、将来の「ミュータブルにすればよい」という提案はここでの却下された代替案の記録から始まることになる。
- アロケーションはプラグインを多用するプロジェクトにおける支配的コストであり続ける。残存するレバーは段階的な`ProjectScope`再グループ化とサイトごとの継続的な衛生改善であり、アーキテクチャの変更ではない。
- 挙動変化なし: 本ADRのすべてのスライス（slice）は診断結果バイト同一として検証済みだ。

## 却下された代替案（要約）

- **ミュータブルプール型Scope/CallContext** — 再入可能性により単一の再利用オブジェクトは安全でない。サイレントなナローイング破壊→偽陽性。
- **精密ティアのみのパスで`CallContext`をスキップする** — 各ティアは1つの`try_dispatch(CallContext)`インターフェースを共有しており、共通パスを特殊化するとディスパッチコストのごく一部のためにそのインターフェースが分岐する。`ProjectScope`再グループ化（より大規模でクリーン）を優先して先送り。
