ステータス: **Accepted、2026-06-26**。 Rigorはコア／stdlibのRBSを、プロジェクトがgemspecの
`rbs >= 3.0, < 5.0`の範囲内で解決する`rbs` gemから読み込み続けます。バージョン起因の診断の
変動は決定性の欠陥ではなく、構造的に正しいものとして扱われます。決定性は戦術的に
（rbs-compat CIマトリクス＋`core_overlay`／`gem_overlay`のエスケープハッチによって）買い取られ、
Rigorが保持するバンドル済みコアRBSは**先送りされたオプトイン**の選択肢として記録されます——
採用はされません。

根拠: `Gemfile.rbs-compat`という、`~> 3.10`と`~> 4.0`の両方に対して環境specを実行するCIジョブを
追加した、upstreamの[rigortype/rigor#22](https://github.com/rigortype/rigor/pull/22)（「Fix RBS
3.x compatibility」、#21を修正）。[2026-06-16のliquid v5.xスイープノート](../../notes/20260616-liquid-v5.x-regression-sweep/)
（`StringScanner#peek_byte`のバージョン遅延に起因する偽陽性）。rigor-rsポートの設計監査
（[2026-06-26のノート](../../notes/20260626-rigor-rs-design-audit/)、リスクR2）。

## コンテキスト

Rigorは、gemspecの範囲`rbs >= 3.0, < 5.0`（`rigor.gemspec:50`）のどこであれ、**自身が解決した
`rbs` gem**が出荷するRBSに対して型検査を行います。この幅は意図的なものです——ユーザーは
自分のプロジェクトに合わせてRubyと`rbs`をピン留めするのであって、Rigorはそれを指図するのでは
なく追従します。PR #22はローダー（`rbs_loader.rb`）を範囲全体で機能するよう堅牢化し、範囲を
正直に保つCIマトリクスを追加しました。

rigor-rsポートが指摘した帰結: コアクラスのシグネチャが`rbs`のリリース間で異なる（例:
`String#bytes`、`StringScanner#peek_byte`）ため、*同じソース*が`rbs 3.10`と`4.0.2`とで異なる
診断集合を生成しうる——1台の開発者マシンとCIの間で、あるいは2人の開発者の間で。rigor-rsは
自らの側でこれを、ビルド時にコアRBSを**ベンダリングし埋め込む**（そのADR-0007）ことで解決し、
`embedded == runtime`のバイト決定性を実現しました。Ruby版のリファレンス実装に対する未解決の
問い——同じことをすべきか？

## 決定

**Rigorはプロジェクト自身のツールチェーンが出荷するRBSに対して検査します。`rbs`バージョン間の
診断の変動は、プロジェクトが実際に実行しているものへの忠実性であって、工学的に除去すべき
バグではありません。**

判別基準——再利用可能な規則:

> **インストール済みツールチェーンへの忠実性は、クロス環境のビット決定性に優先します**。
> 決定性が買い取られるのは*戦術的に*、バージョンの差が**誤った**結果（ランタイムには存在する
> メソッドに対する偽陽性）を生む場合に限られ、プロジェクトの実際の`rbs`からRBSソースを
> 構造的に切り離して凍結することでは決してありません。

具体的には:

- **`rbs >= 3.0, < 5.0`を維持する**。 PR #22の`Gemfile.rbs-compat` CIマトリクス（`~> 3.10`∧
  `~> 4.0`）が、サポート範囲を正直に保つガードレールです。
- **バージョン遅延による*偽陽性*はバージョンではなくソースで修正する**——Rigor所有の無条件の
  `data/core_overlay/`再オープン（`rbs_loader.rb`の`core_overlay_sig_paths`、例えば`rbs 4.0.2`が
  省く`StringScanner`のメソッド群）、そしてADR-72のGemfile.lock条件付き`data/gem_overlay/`
  （`:missing`のRBSを持つgem向け）。したがってRigorは*すでに*自身のコアRBSの小さく外科的な
  スライスを保持しています——ソースを置き換えるためではなく、誤りを修正するために。

## 先送りされた代替案 —— Rigorが保持するバンドル済みコアRBS（不採用）

完全なコア＋stdlibのRBSをRigorの信頼できる情報源としてベンダリング／埋め込みする
（rigor-rsの経路）とクロス環境の決定性が得られますが、**既定としては**次の理由で却下されます:

1. **診断をプロジェクトの実際の`rbs`／Rubyから切り離してしまう**——Rigorはプロジェクトが実行
   していないかもしれないバージョンに対して報告することになり、幅広い範囲サポートが存在する
   目的である忠実性の隙間をまさに再導入してしまいます。
2. 各upstream `rbs`リリースを追跡する恒常的な**メンテナンス負担**を追加します（rigor-rsの監査が
   埋め込みアプローチのリスクR2として記録したものと同じリリース周期のコストです）。
3. `core_overlay`がすでに戦術的かつ均整の取れた形で行っていることを**重複させます**。

**再評価トリガー:** クロス環境の*非決定性*——修正可能なオーバーレイの隙間に帰せられない、CIと
開発環境の間の診断のドリフト——が実証されたサポートコストになった場合、見直します。範囲は
**オプトインのピン留めRBSモード**（設定で選択するベンダリング済みコア）とし、インストール済み
への忠実性が既定のままであり、決定性は各プロジェクトの意図的な選択となるようにします。
rigor-rsのベンダリング＋埋め込みは、このモードが実現する場合のリファレンス設計です。

## 帰結

- **ポジティブ:** 診断はプロジェクトが実行しているツールチェーンを反映します。幅広いRuby／
  `rbs`サポートが保たれます。新たなベンダリングやバージョン追跡のサーフェスはありません。
- **ネガティブ:** クロス環境の決定性はベストエフォートのままです——新しい`rbs`リリースは
  `core_overlay`のパッチが当たるまで診断を変えうります。compat CIマトリクス（破損は検出される）
  と2つのオーバーレイのエスケープハッチ（誤りは`rbs`をピン留めするRigorリリースなしにパッチ
  可能）によって範囲は限定されます。
- **持ち越し:** オプトインのピン留めRBSモードは、上記の再評価トリガーの背後にある、生きた
  未着手の選択肢として残ります。

## 他のADRとの関係

- **ADR-72** —— Gemfile.lock条件付きの`gem_overlay`は、このADRが戦術的決定性のために頼る
  無条件の`core_overlay`と対をなす条件付きの片割れです。
- **ADR-54 / ADR-7** —— キャッシュ＋RBSの出荷。オーバーレイディレクトリは、env-cacheの
  記述子がすでにダイジェストしているローダーのシグネチャパスに乗ります。
- **rigor-rs ADR-0007**（兄弟のRustポート）—— 単一バイナリ・Ruby非依存の配布目標のため
  *正反対*の既定（ベンダリング＋埋め込み）を採用しています。このADRはリファレンス実装の
  意図的な相違と、それが収束しうる条件を記録します。
