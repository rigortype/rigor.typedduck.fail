ステータス: **Accepted — 発効中（生きているルーブリック（rubric）），2026-06-05**。 RigorのADRを執筆・評価するための、アーキタイプ対応の共有ルーブリックを確立する。目的は実務的なものだ: **必要十分**な情報を持つADRの執筆（半自動下書きを含む）を推進すること——次の実装者がコードを全読みするほど薄くもなく、決定の重要度が正当化する以上に水増しされてもいない。このルーブリックは生きた文書であり、コーパス（corpus）が新たな失敗パターンを教えてくれるたびに評価軸を洗練させていく。

根拠: リポジトリ内のADRコーパス自体（ADR-0 … ADR-48）を、このルーブリックで採点した[`docs/notes/20260605-adr-corpus-rubric-audit.md`](../../notes/20260605-adr-corpus-rubric-audit/)の監査（audit）——ルーブリックを検証した監査（rubric audit）だ。ハイステークス×thin-intentの警告は正しく沈黙を保ち、機械的/基盤的ADRを免除しつつ誤検知も出さず、エコノミーがコーパスの唯一の系統的なドリフトとして浮かび上がった。以下に引用するハイウォーターマーク事例——[ADR-43](../43-rbs-complete-ancestor-resolution/)、[ADR-46](../46-incremental-dependency-graph/)——および適切に簡潔な[ADR-40](../40-config-schema-defaults/)がキャリブレーションセットとなる。

## コンテキスト

コーパスは性格の大きく異なる48本のADRにまで成長した。中には価値が推論そのものにある深い熟慮の記録もある——ADR-43（偽陽性をゼロに保つallow-listスコープ）、ADR-45/46（キャッシュの健全性ストーリー）。一方、メンテナーが流れの中で行った単純な選択をほぼ機械的に記録したものもある——ADR-40（`{kind:, default:}`のデフォルト設定）、ADR-38（追加イニシャライザー）、ADR-7（v0.1.0スライスバンドル）。機械的なものの多くは1行のメンテナー選択から半自動で下書きされたため、深い「意図」の叙述を持ちも必要ともしない。

必要なルーブリックは2つの役割を担う: 既存のADRの情報充足度を**評価**すること、そして次のADR（プロンプトから下書きする人間またはエージェント）を**ガイド**すること。避けるべき罠は均一なチェックリストだ: 一律に適用すると、機械的なADRが必要としなかった熟慮の欠如を「問題」として罰し、逆の欠陥——ちっぽけな決定が10画面に膨らむこと——を指摘する術もない。目標は*比例性*であるため、ルーブリックは**アーキタイプ対応**で**ステークス加重**でなければならず、エコノミーを**双方向**で評価しなければならない。

これはREADMEの「Adding a New ADR」ステップ（構造上の契約: Status / Context / Decision / Consequences＋ インデックスエントリー）と並んで位置する——それは*フォーマット*であり、このADRはその上に重ねた*品質基準*だ。

## 決定

ADRを3つのパスで採点する: **アーキタイプ**をタグ付けし、**ステークス**をタグ付けし、次に**8つの評価軸**を採点する——アーキタイプが免除する評価軸はゼロ点にするのではなく除外する。

### 1. アーキタイプタグ（最初に必ずひとつ割り当てる）

| アーキタイプ | 内容 | コーパス事例 | 評価軸の扱い |
| --- | --- | --- | --- |
| **Deliberative（熟慮型）** | 実際の設計上の選択であり、生きたトレードオフを持つ;推論がペイロードだ。 | 43, 45, 46, 15, 35 | フルルーブリック。評価軸1〜2（意図、基準）がスコアの*核*。 |
| **Mechanical / policy（機械的/方針型）** | 小さな選択やほぼ強制された選択、または定常的な方針の記録。半自動下書きが多い。 | 40, 38, 7, 31 | 評価軸1〜2は**寛大に**採点（1行の根拠で満点）するか、選択が本当に強制されている場合は**N/A**。評価軸8（エコノミー）は**ウェイトを上げる**——これらは短くなければならない。 |
| **Evaluation / proposal（評価/提案型）** | まだ実装されていないオプションを評価する（条件付き採択/保留/却下）。 | 21, 30, 41, 42 | 評価軸5の*エビデンス*サブパートは**N/A**（測定対象となる実装がない）; *受理基準*サブパートは引き続き適用。 |

### 2. ステークスタグ（評価軸ではなく読み方のモディファイア）

各ADRを3つの観点で`low`/`mid`/`high`にタグ付けする: **可逆性**（元に戻しにくさ）、**ブラストラジアス**（エンジン全体vs 1つのプラグインフィールド）、**偽陽性/健全性エンベロープへの影響**（プロジェクトのトップティア規律——`feedback_false_positive_discipline`参照）。ステークスはスコアに加算しない; 2つの評価軸の*読み方を条件づける*:

- **意図/基準（評価軸1〜2）:** `high`ステークス × thin intentは**本当のギャップ**——このルーブリックが最も価値ある発見をする場面だ（根拠なしに記録された重要な決定）。`low`ステークス × thin intentは**正しいエコノミー**であり欠陥ではない。これがまさに半自動ADRへの過剰ペナルティを避ける方法だ。
- **エコノミー（評価軸8）:**ステークスが長さの予算を設定する。`high`ステークスはADR-46の長さを正当化し; `low`ステークスはADR-40のように読めるべきで、`low`ステークストピックでの同じ長さは評価軸8が発見する*過剰*情報欠陥だ。

### 3. 8つの評価軸（各0 / 1 / 2、または**N/A**で採点）

N/Aはゼロではない: 免除された評価軸は**分母から除外される**（§4）。

1. **意図と目的**。目標——*何を求め、なぜ重要か*——が冒頭に述べられ、できればプロジェクトの定常的な価値に結びついているか？* 2*: 価値に固定された明快な動機付けの問い/目標（ADR-43: "Can Rigor itself, Steep-free, warn on plugin-contract misuse?"）。*1*: 目標は述べられているが汎用的。*0*: "なぜ今か"なしにメカニズムへ直行。*（deliberative-core; mechanical: 寛大に/N/A）*
2. **決定基準**。 *どの*オプションかだけでなく、識別に用いた**再利用可能なルール**も記述されているか？* 2*: 将来のケースに適用できる一般化可能な基準（ADR-43: "このクラスのRBSは権威があり完全——省略された呼び出しは本当に間違い"、そして`Plugin::Base` = true、`ActionController::Base` = falseへ適用）。*1*: 選択肢が個別の理由付きで列挙されているが一般ルールなし（ADR-40の却下代替案テーブル）。*0*: 比較なしに主張された選択。*（deliberative-core）*
3. **実行可能性とコードアンカリング**。実装者はすぐにコードに行けるか？* 2*: ファイル＋おおよその行番号と周辺の実際のシンボルを伴う名付きのインジェクションポイント（ADR-43 WD2: `RbsDispatch.lookup_method`、`rbs_dispatch.rb` ~L270、"after the nil lookup, before fall-through"; ADR-46: `Scope`アクセサのチョークポイント）。*1*: 正しいファイル/クラスは指名されているが正確なサイトなし。*0*: 抽象的なメカニズムのみ。*（全アーキタイプ）*
4. **制約とガードレール**。 "これを壊すな"のエンベロープ: 保持すべき不変条件（invariant）、明示的なスコープ内/外の区切り線、FP/健全性境界。*2*: 重要な結合と越えてはならない線を明記（ADR-43: "precisionと`undefined-method`-firing flowは同じパスを通る"; ADR-46: "依存関係の記録漏れは偽陽性を生む → 保守的フォールバック、絞り込みは禁止"）。*1*: スコープは限定されているが制約は暗黙。*0*: 破ってはならないものについて沈黙。*（全アーキタイプ）*
5. **検証可能性とエビデンス**。 *受理基準*（完了をどう知るか——ゲート、テスト、測定によるクリーンバー）**かつ***根拠*（数値、スパイクノートのリンク）。*2*: 両方、具体的に（ADR-46: `--verify-incremental`バイト同一ゲート + "262ファイル、0.75s vs 7.2s、約9.6×"; ADR-43: `make check-plugins`非ゼロ終了 + 正味FPゼロの測定）。*1*: エビデンスなしの基準、またはその逆。*0*: どちらもなし。*（evaluation: エビデンスサブパートN/A——基準のみで採点）*
6. **ステータスと進捗の忠実度**。 Statusの行はスライス粒度で真実を語り、本文と一致しているか？* 2*: 正確なステータス、どのバージョンで何がランドし、何が残っているか（ADR-46のステータスブロックが再開のブックマークとして機能）。*1*: 粗いステータス、本文は大まかには一貫。*0*: ステータスと本文が一致しない（例: まだ提案として書かれた本文に"landed"）。*（全アーキタイプ）*
7. **コネクテッドネス**。 ADRを設計ウェブに位置づけるクロスリファレンス——隣接するADR、デュアル、バインディング仕様。*2*: 隣接関係と関係性を名付ける（ADR-43 ↔ ADR-26が逆のノブとして; → ADR-24の`discovered_superclasses`）。*1*: 素のリンクがひとつふたつ。*0*: 孤立している。*（全アーキタイプ）*
8. **エコノミー/比例性（目標軸——双方向）**。長さはステークスに対して*必要十分*か？ 過少情報（コード読みを強いる）と過剰情報（儀式的な記述、繰り返しのコンテキスト、`low`ステークスの決定が正当化する以上の水増し）の**両方**にペナルティを課す。*2*: 長さがステークスに追従——ADR-40は小さな変更に対して短く、ADR-46は健全性上重要なものに対して長く、どちらも正しく感じられる。*1*: やや過剰/過少。*0*: どちらの方向にも著しく不均衡。*（全アーキタイプ; mechanicalではウェイトを上げる）*

### 4. 採点と過剰ペナルティ回避ルール

最終スコア = `sum（採点済み評価軸） / （2 × count（採点済み評価軸））`、パーセント表示。**N/Aの評価軸は分母から除外される**——評価軸1〜2が免除された機械的ADRは16ではなく12（6つの評価軸）で採点されるため、正しく省略した熟慮によって引き下げられることはない。`high`ステークスのADRは1〜2を**決して**免除されない;そこでの薄い意図は本物の0としてカウントされ、ヘッドラインの発見となる。

パーセントは**スコアではなく会話の出発点**として扱う: その役割はADRを引き下げている1〜2つの評価軸を特定し、著者がそれらを修正できるようにすることであり、コーパスをランク付けすることではない。1〜2がN/Aというだけで"低"スコアになっている半自動ADRは、健全な状態であり失敗ではない。

## 却下/保留の代替案

| 候補 | ステータス | 理由 |
| --- | --- | --- |
| アーキタイプなしの均一なチェックリスト | 却下 | 機械的/半自動ADRが必要としなかった熟慮の欠如を罰し、過剰情報欠陥を指摘できない。比例性こそが全要点だ。 |
| ステークスを独立した採点評価軸にする | 却下 | ステークスは*決定*の属性であり、*文書の品質*の属性ではない;二重計上はスコアを歪める。評価軸1〜2と8の読み方モディファイアとして位置づくべきだ。 |
| 絶対的な長さ制限（例: "≤ N語"） | 却下 | エコノミーは比例的であり絶対的ではない——ADR-46の長さは正しく、ADR-40の簡潔さも正しい。固定上限は正しく長いADRを罰し、中身のない短いADRを報奨する。 |
| 明示的な意図がないADRを自動ペナルティ | 却下 | 目標と真っ向から矛盾する: 多くのADRは正しく薄い。薄い意図は`high`ステークスのときだけ欠陥だ（§2）、これはモディファイアが一律ペナルティなしに捉える。 |
| 意図を既存の"Context"期待に折り込む | 保留 | 意図（目標/なぜ）とコンテキスト（問題/背景）は別物であり、別々に採点する価値がある;合体させると重要なADRで最も一般的なギャップが隠れる。2つの評価軸が実際に連動することが判明した場合のみ再検討する。 |

## 結果

ポジティブ:

- 下書きエージェント（またはメンテナー）は具体的なターゲットを得る: アーキタイプを選び、ステークスを判断し、適用される評価軸を満足させて——止まる。これが薄い下書きと水増しされた下書きの両方を抑制するものだ。
- N/Aメカニズムにより、ルーブリックはコーパス全体に**安全に実行**できる: 半自動ADRはその性質に対して罰せられないため、低スコアはアーキタイプの不一致ではなく本物のギャップを確実に意味する。
- ハイステークスで薄いADRが発見可能になる——最も有用な監査の成果のひとつだ（推論が欠落した重要な決定）。

ネガティブ:

- アーキタイプとステークスのタグ付けは人間の判断; deliberative/mechanicalの境界で2人のレビュアーが異なるタグを付けることがある。作業済み事例（43/46はdeliberative、40はmechanical）をアンカーとすること、およびスコアが助言的であることで軽減される。
- 8つの評価軸はクイックな勘チェックより多い。明らかに些細な機械的ADRでは、評価軸3/6/8だけが合理的な高速パスになる; 8つ全部はdeliberativeなケースとコーパス監査のためにある。

## 他のADRとの関係

- **README "Adding a New ADR"** — 構造上の契約（Status / Context / Decision / Consequences＋ インデックスエントリー）を供給する;このADRはそのフォーマットの上の品質基準だ。
- **ADR-31**（コントリビューション＆サプライチェーンポリシー）——姉妹的な"policy in force" ADR; ADR-31がコントリビューションを統治するように、このADRはADR執筆を統治する。
- **`feedback_false_positive_discipline`**（およびADR-43/45/46）——評価軸4と`high`ステークスタグが鍵とする偽陽性/健全性エンベロープ; ADRの意図を記録する際に最も価値のある判断基準だ。
- **AGENTS.md** ——下書きエージェントが最初に読む開発上の契約;このルーブリックはそれに対するADR固有のコンパニオンだ。
- **`.claude/skills/rigor-adr-author`** ——このルーブリックを運用化する手続き的ラッパー: アーキタイプ＋ステークスをタグ付けし、マッチするスケルトンを選択し、機械的な配線（ナンバリング、2つのインデックス更新、検証）を行い、品質基準はここに委ねて再記述しない。
