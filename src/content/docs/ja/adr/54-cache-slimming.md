`5f53db09`がWD1、`0c671e04`がWD2、`d2465fe1`がWD3、`5ced88f1`がWD4）。
[2026-06-10のキャッシュディスク + ウォームロード監査](../../notes/20260610-cache-disk-runtime-audit/)
に根拠を置く（以下の数値はすべてそのノート由来。Mastodonキャッシュ、v0.1.17の作業
ツリー）。着地後の観測: 圧縮された`rbs.environment`エントリーは1.76 MB（11.0 MBの生
blobの16%）と計測された。リポジトリ自身の`.rigor/cache` — 長命なチェックアウト —
は約2 MB / 14エントリーのアクティブ集合に対して約180 MB / 47エントリーを保持しており、
WD3のオーファン（孤児）の筋書きをライブで確認した（キャップ付き実行がまさに古い
エントリーを刈り取り、次の実行はウォームのままだった）。すべてのスライスは診断同一の
self-check実行（`--no-cache` / コールド / ウォーム）+ キャッシュ / 環境 / 設定の
スペックでゲートした。コーパス横断のMastodon等価性実行は監査ノートのフォローアップに
記録されている。

## コンテキスト

すべてのプロジェクトの`.rigor/cache`は、プロジェクトサイズに関わらずほぼ一様な
**約32 MB**を量る。これは3つの単一blob RBSプロデューサーが支配的である:
`rbs.instance_definitions`（14.5 MB）、`rbs.environment`（10.6 MB）、
`rbs.singleton_definitions`（9.0 MB）。自身の`signature_paths:`を持たない
プロジェクトは**バイト単位同一**のエントリーを生成するので、約30プロジェクトの調査
コーパスを持つマシンは1 GB超の重複データを抱える。ウォームパスでは、3つのblobの
ロードに**約760 ms / 220万アロケーション**かかり、そのうち`Marshal.load`が約700 ms
である — ディスク読み取り + SHA-256エンベロープ検証は合計約15 msで注意を要しない。

監査の決定的な計測: env blobがすでにキャッシュされている状態で、**そこから492個の
インスタンス定義をすべて構築するのに137 ms / 50万アロケーション — 対して、
instance-definitions blobを`Marshal.load`するのに366 ms / 106万アロケーション**
（singleton側: 全構築178 ms対ロード180 ms — 拮抗。そして実行が実際に触れるものの
クラスごとの遅延構築はほぼゼロである: 一般的な12クラス = 0.0 ms）。定義blobは
プロジェクトあたり23.4 MBを費やして、ウォーム実行を*遅く*している。

ADR-7の§「Slice 6-D」が単一blobレイアウトを選んだのは、**クラスごとのディスク
エントリー**が`--no-cache`より遅かったからである。それは*blob対キャッシュ済みenvからの
再計算*を一度も計測しなかった。その比較が今や存在し、blobが負ける。

## 決定

次の基準を採用する: **キャッシュ階層がそのバイト数を稼ぐのは、ウォームパスにおいて
次に安い階層からの再計算を上回る場合に限る** — コールド再構築に対してではなく、自身の
上流キャッシュに対して計測する。適用:

1. `rbs.instance_definitions` / `rbs.singleton_definitions`のディスクプロデューサーを
   廃止する（WD1）、
2. すべてのエントリーの値ペイロードをdeflate圧縮する（WD2）、
3. エビクションにnilでないデフォルトキャップを与える（WD3）。

期待されるエンベロープ: ディスク上で**33.7 MB → 約1.7 MB / プロジェクト（−95%）**。
定義に触れるウォーム実行は最大で約550 ms / 160万アロケーションを節約し、コールド実行は
先行的な全構築 + 23 MBの書き込みを脱ぎ捨てる。精度 / 診断は手付かずである — すべての
スライスはバイト単位同一の診断 + `make bench-perf`でゲートする。

## ワーキング・デシジョン

**WD1 — 定義blobを廃止し、キャッシュ済みenvから構築する**。
`cached_instance_definition` / `cached_singleton_definition`
（`rbs_loader.rb`約L979/L990）は、テーブルルックアップからオンデマンドのクラスごとの
`build_*_definition`へ切り替わる。既存のプロセスごとのメモ
（`@instance_definition_cache` / `@singleton_definition_cache`）がすでに繰り返しを
短絡する。`lib/rigor/cache/rbs_instance_definitions.rb`（両プロデューサークラス）を
削除する。**制約 — ADR-15のprewarm / Reflectionの消費者はeagerなままにする:**
`RbsLoader#prewarm`（約L765）と`#reflection`（約L809）は完全なテーブルを必要とする
（プールワーカーは`RBS::EnvironmentLoader.new`を決して呼んではならない。forkバックエンドは
親でウォーム化されたテーブルをCOW共有する）。それらのために、
`instance_definitions_table` / `singleton_definitions_table`は完全なテーブルを構築し
続ける — いまやMarshalロード（546 ms）ではなくキャッシュ済みenvから計算される（両側で
315 ms）。すでに構築されたenv上の`RBS::DefinitionBuilder`は`EnvironmentLoader`定数
チェーンに触れないので、Ractor隔離の論拠は保たれる。

**WD2 — 値ペイロードをdeflateする**。 `Store#write_entry`は`value_bytes`をdeflateし
（zlib、標準ライブラリ — 新しい依存なし）、`#read_entry`はinflateする。フォーマット
バイトをbumpする（`HEADER` `RIGOR\x00\x01` → `\x02`、`store.rb:30`）— 旧フォーマットの
エントリーはマジックチェックに失敗し静かなミスとして読まれるので、**移行コードは不要**:
次の実行が再計算して上書きする（既存のフォールトトレランス契約）。SHA-256トレイラーは
格納された（圧縮済みの）バイトをカバーし続ける。計測: blobは13〜16%へ圧縮される。
inflateは、それが隣り合う約700 msの`Marshal.load`に対して合計約50 msかかる —
ノイズの範囲で実行時中立である。

*追記（同日に着地したフォローアップ）:*フォーマットのbumpだけでは旧エントリーを読めなく
するが決して削除しない — そして約32 MBではWD3のキャップを永遠に下回って居座る。
そこで`schema_version.txt`マーカーはいまや`"<SCHEMA_VERSION>.<FORMAT_VERSION>"`
（`Store.schema_marker_value`）を運ぶので、フォーマットbump後の最初の書き込み可能な
実行が、確立されたマーカー不一致パスを通じてルートをクリアしバイトを回収する。ADR-46の
インクリメンタルスナップショット — `Store`を迂回する唯一のキャッシュ成果物 — は同じ
deflate処理を受ける（その`SCHEMA`を4 → 5へbump。生の5未満のblobはinflateに失敗し
nilとしてロードされる、いつものコールド実行パス）。

**WD3 — デフォルトのエビクションキャップ**。 `cache.max_bytes`はnilをデフォルトとし
（`configuration.rb:78`）、`Store#evict!`（すでに`cli.rb:107`で配線済み）を恒久的な
no-opにしていた。今日エントリー数がプロデューサーあたり1に留まっているのは、
`Descriptor::SCHEMA_VERSION`のbumpがルートをクリアしてきたからにすぎない。スキーマが
安定すれば、`rbs` gemのbumpや`signature_paths:`の変更が新しいキーを書き、古い
エントリーを永遠にオーファンにする。キャップを寛大な**256 MB**にデフォルト設定する
（WD1/WD2後はプロジェクトあたりの完全な集合が約2 MBなので、キャップはオーファンのみを
刈り取る）。明示的な`max_bytes:`設定は依然として上書きする。

**WD4（マイナー）— ローダーごとに`RbsDescriptor.build`をメモ化する**。これは
プロデューサーフェッチごとに一度走る（実行あたり7回、結果は同一）。今日1.3 ms × 7と
計測される — ノイズ — が、`signature_paths:`のサイズ（`gem_rbs_collection`の
チェックアウト）に対して線形にスケールし、メモは1行である。

## 棄却・延期した代替案

| 代替案 | 判定 | 理由 |
| --- | --- | --- |
| 定義blobを残しつつ圧縮する | 棄却 | WD1に支配される — 圧縮blobでもキャッシュ済みenvからの再計算より`Marshal.load`が遅く、依然としてディスクを支払う。 |
| プロジェクト横断の共有キャッシュルート（`rbs.*`プロデューサーのためのXDG `~/.cache/rigor`） | 延期 | 安全（エントリーはコンテンツをキーとする。ADR-6が延期したのはマシン*横断*の共有のみ）だが、WD1/WD2後は重複が約1.7 MB × Nへ縮む — 第2のルート、そのロックの筋書き、読み取り専用 / エディタモードの相互作用に見合わない。env blobが桁違いに大きくなれば再評価する。 |
| ADR-45の`fresh?`再ダイジェストのためのmtime高速パス | 棄却 | Mastodon規模の依存集合でウォーム約50〜150 msを節約するが、ADR-45が拠って立つダイジェスト厳格の健全性を犠牲にする。 |
| gem依存を介したzstd | 棄却 | zlibは標準ライブラリですでに13〜16%の範囲内である。圧縮gemは、ADR-6が独自フォーマットバックエンドを選んだ新しい依存なしの姿勢に反する。 |
| クラスごとのディスクエントリー（ADR-7のslice 6-Dの再審） | 棄却のまま | ADR-7の計測は立つ: エントリーごとのディスクオープン + ロードがウォーム実行を`--no-cache`より遅くした。WD1はその階層を再シャーディングするのではなく丸ごと除去する。 |
| シークインデックス付きの遅延blob（オフセットテーブル + 部分読み取り） | 棄却 | WD1と同じ先行ロードの問題を、厳密により多くのフォーマット機構で解く。 |

## 帰結

- **ポジティブ:**プロジェクトあたりのキャッシュが約95%小さくなる。調査コーパスの
  マシンで1 GB超を回収する。定義に触れるウォーム実行が最大で約550 msと160万
  アロケーションを節約する。コールド実行は先行的な983定義の構築と23 MBの書き込みを
  スキップする。プールモードのprewarmが速くなる（計算315 ms対ロード546 ms）。
- **ネガティブ / コスト:** WD2のフォーマットbumpでの一度限りの完全なキャッシュ無効化
  （静かなミス + 再構築 — 標準的なパス）。親がprewarmしなかった定義を遅延構築する
  プールワーカーは、その小さな構築をワーカーごとに重複する（全構築の315 msで上限が
  あり、COWがそれを稀にする）。
- **ステータスの波及:** ADR-7の§ Slice 6-Dは**部分的に置き換えられる** — 定義blobの
  半分が廃止される。env blobとクラスごとのディスクエントリーの棄却は立つ。ADR-6の
  フォーマットバージョンポリシーがWD2のヘッダーbumpをカバーする。

## 他のADRとの関係

- **ADR-6** — ストレージバックエンド。WD2がbumpするバージョンバイトを持つオンディスク
  フォーマットを所有する。その延期された「マシン横断の共有」の行は影響を受けない。
- **ADR-7** — slice 6-DはWD1によって部分的に置き換えられる（上記を参照）。
- **ADR-15** — prewarm / Reflectionのeagerテーブル契約（フェーズ2b / 4b.x）がWD1の
  遅延 / eager分割における拘束的な制約である。
- **ADR-44/45/46** — 同じ性能プログラム。このADRはそれらが手付かずに残したキャッシュ
  層のディスク / ロードの軸を閉じる。
