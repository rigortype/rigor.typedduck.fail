**ステータス:** Accepted — record-and-validateの実行キャッシュがランディングした。単純な解析前フィンガープリント（fingerprint）は却下された（`pundit_plugin_spec`のクロスプロセス退行テストによって不健全（unsound）であることが実証された）。健全（sound）なrecord-and-validateの設計が実装・検証済みであり、変更のないMastodonの`app/models`（248ファイル）は11.6秒→1.8秒（約6倍）に短縮し、診断（diagnostics）はバイト単位で同一、`make verify`はグリーン。

`rigor check`は、プロジェクトに何も変更がない場合でも、呼び出しのたびに**ファイル単位の推論全体**を再実行する。永続キャッシュ（`.rigor/cache`、ADR-6）は中間成果物のみ——RBS環境とプロデューサーごとのプラグインテーブル——をコンテンツアドレス指定の`Cache::Descriptor`をキーとして保持するが、解析の**結果**はキャッシュしない。そのため、全ファイルに対する`CheckRules.diagnose`——ADR-44のプロファイリングで計測したアロケーションバウンドな処理で、ウォームな実行では壁時計時間の~90%以上を占める——は、バイト単位で同一の結果を再現するためにフルコストを支払う。GitLabの設定済みサブセット（2,630ファイル、11プラグイン）では「何も変わっていない」と知るだけに~100～150秒かかる。

このADRは、変更のないプロジェクトの全結果をキャッシュから提供するファストパスを設計し、自明な実装が健全でない理由を記録する。

背景：[`docs/notes/20260604-gitlab-plugin-contribution-allocation.md`](../../notes/20260604-gitlab-plugin-contribution-allocation/)のプロファイルと[ADR-6](../6-cache-persistence-backend/)のキャッシュアーキテクチャ。

## コンテキスト——実行の診断が依存するもの

1. **解析された各ファイルの内容**。
2. **他の解析済みファイルの内容**——クロスファイルのユーザーメソッド戻り値推論（`infer_user_method_return`）：`B#foo`を呼び出すファイルAは`B#foo`の*推論された*戻り値の型を採用し、それは`B#foo`の**本体**に依存する。解析済みファイルの変更はどれも別のファイルの診断を変更しうる。
3. **プラグインが読み込むファイル**——そして*いつ*読み込むか。`prepare`中に読み込まれるものもある（rails-routesの`config/routes.rb`、actionpackのコントローラー）。**他のものはファイル単位の解析中にオンデマンドで読み込まれる**。`rigor-pundit`は`authorize`呼び出しを初めて見たときにポリシーファイルを読み込み、ポリシーがそのアクションを定義しているか確認する——つまりプリパス後、`analyze_files`の途中で読み込まれる。
4. **RBS環境**——`Gemfile.lock`のgemセット、`sig/`ファイル、`rbs`のバージョン、`target_ruby`。
5. **設定**——`severity_profile`、`disabled_rules`、`plugins`、`paths`、`exclude`、`pre_eval`、`--explain`など。
6. **エンジン**——`Rigor::VERSION` / キャッシュスキーマ。

## 決定

### 却下——解析前フィンガープリントをキーとする実行全体キャッシュ

最初の実装（プロトタイプ後に差し戻し）は、診断の計算を`Cache::Store#fetch_or_compute`でラップし、**プリパス後に**以下から構成した`Cache::Descriptor`をキーとした：解析済み全ファイルのダイジェスト、RBSディスクリプタ（gems + sig）、各プラグインの`io_boundary.cache_descriptor`、`Configuration#to_h`のダイジェスト、および`Rigor::VERSION`。

基本的な健全性チェックは通過した（変更なし→バイト同一の診断でヒット；エラーを修正→ミス→新鮮な結果）。しかし、**`spec/integration/plugins/pundit_plugin_spec.rb`のクロスプロセスのキャッシュ無効化退行テスト**に失敗した。このテストはまさにこのハザードに対する標準的なガードである：

> `archive`**なし**でポリシーを書く→`rigor check`が`authorize :archive`呼び出しをフラグする；`archive`**あり**でポリシーを書き直す→2回目の`rigor check`（新しいプロセス、同じキャッシュ）はフラグを**立てていけない**。

フィンガープリントはプリパス直後に構築されるが、Punditはポリシーファイルを`analyze_files`の**最中に**（`authorize`呼び出しごとに）読み込む。そのため、フィンガープリント構築時点ではポリシーの読み込みはまだ発生しておらず、解析対象ファイル（`demo.rb`）は変更されておらず、フィンガープリントは両方の実行で同一になる→2回目の実行は古いヒットになり、修正済みの呼び出しを再報告する。**解析前に計算されたフィンガープリントは、解析自体が発見する入力を捕捉できない**。これを出荷すると編集をまたいで偽陽性（false positive）が生まれる——正確性ツールとして最悪の障害モードである——ため却下する。

### 採用済み（ランディング）——依存関係のrecord-and-validate

健全なモデルは順序を逆転させる：**実行し、実際に読み込んだすべての入力を記録する；その依存集合（dependency set）とともに結果をキャッシュする；次回の実行では、記録された依存関係を再読み込みして検証する**。

- **キー：**実行前に安定して判明している——解析パス*セット* + 設定ダイジェスト（`Configuration#to_h`）+ gems（`RbsDescriptor`のgem/configエントリー）+ `Rigor::VERSION` + `--explain`。ファイルの追加/削除はパスセットを変更→新しいキー；ファイルの編集はキーを保持し検証によって捕捉される（つまりコンテンツの編集は同じスロットを再利用——キャッシュが肥大化しない）。
- **保存値：**`[diagnostics, dependency_descriptor]`。ディスクリプタの`files`は実行**後に**収集される——解析済みファイル + RBSの`sig`ファイル + 各プラグインの`io_boundary.cache_descriptor`（Punditポリシーのような解析時読み込みを含む完全な実行後の情報）。`Diagnostic`はフラットな値オブジェクトなので、ペアは`Marshal`クリーンである。
- **ルックアップ：**`Cache::Store#fetch_or_validate`は安定したキーのエントリーを読み込み、`Descriptor#fresh?`を呼び出す。これは記録された全`FileEntry`を再ダイジェストし、すべてが一致する場合のみ**ヒット**（不一致なら→ミス→再実行→同じスロットを上書き）。最初の実行後にポリシーが依存集合に含まれるため、Punditポリシーを捕捉できる。

実装が強制するロバストネスの2つのルール：**キャッシュが実行を絶対に壊してはならない**——書き込み時のシリアライズ/ディスク障害は飲み込まれ（キャッシュをスキップ）、キャッシュパスエラーはいずれもキャッシュなしの直接解析にフォールバックする；そして実行後の依存関係収集は各プラグインの`@io_boundary`を遅延`||=`初期化子をトリガーせずに読み込む（プラグインインスタンスは実行後にフリーズされており、境界を構築しなかったプラグインはそれを通じてファイルを読み込んでいない）。ファストパスは順次実行・書き込み可能キャッシュ・非エディタ・非プリビルドの実行にのみ適用される；キャッシュヒット時は`nil`のstatsを返す（要約するはずだった解析は実行されなかった）。

検証済み：変更のないMastodonの`app/models`（248ファイル）は**11.6秒→1.8秒（約6倍）**に短縮；`fix-an-error → 0 errors`（古いものなし）；`pundit_plugin_spec`のクロスプロセス退行テストとプロデューサーごとのキャッシュテスト（rigor-routes、rigor-rbs-inline）はパス；`make verify`グリーン。

### 付随機能（ランディング）——検証ゲートはキャッシュプルーフ

結果キャッシュは、プロジェクト自身のゲートが古い結果を読み込むことを絶対に許してはならない。フィンガープリントはエンジンコードを意図的に除外している（`Rigor::VERSION`のみ）ため、バージョンを変えずにエンジンを編集するとヒットによってマスクされる可能性がある。そのため`make check` / `check-plugins`は`rigor check --no-cache`を実行する——ゲートは常に解析を再実行する。`lib/`を編集した後に実際のプロジェクトで`rigor check`を実行する開発者も同様に`--clear-cache`（または`--no-cache`）すべきである。

## 影響

- **健全性がすべて**。単純な設計は解析時のプラグイン読み込みに対して無効化が不十分であり；record-and-validateの設計は、実行前の推測からではなく実行が実際に読んだものから依存集合を導出することでこれを解消する。pundit退行テストが実装の受け入れゲートである。
- **変更なし時のフロアはプリパスであり、ゼロではない**。健全なキャッシュでも、検証してサーブする前に`run_project_pre_passes`（発見済みシンボルインデックス + プラグイン`prepare`のために解析済み全ファイルをパース）を実行する。後続のスライスで同じファイルダイジェストセットをキーとしてプリパスの成果物をキャッシュし、ほぼ即座に近づけることができる；先送り。
- **ファイル単位の粒度はスコープ外**。AのDiagnosticsがBの*本体*（項目2）に依存するため、単一ファイルの編集を生き延びるファイル単位のキャッシュにはRigorが構築しないクロスファイル依存グラフが必要である；実行全体のエントリーが適切な最初の粒度である。

## 却下した代替案（まとめ）

- **解析前の実行全体フィンガープリント**——解析時のプラグイン読み込み（Pundit）に対して不健全；既存の退行テストによって実証済み。
- **解析セットのみからのフィンガープリント**——`paths:`外のプラグイン読み込みファイルも見逃す；上記に包含される。
- **手動選択した設定フィールドのフィンガープリント**——脆弱；`Configuration#to_h`全体をダイジェストする。
- **単一ファイル編集用のファイル単位キャッシュ**——クロスファイル依存グラフが必要；先送り。
