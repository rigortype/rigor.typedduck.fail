Status: **Proposed — 未実装。需要ゲート付き**。多相なメンバーフィールド
（例: `kramdown::Element#value`）を、その兄弟である**判別タグ**
（discriminant tag）フィールド（`Element#type`）でナローイング（narrowing）する。
これにより、ペイロードの型がタグの関数になっているAST／ビジター／コンバーター
コードが、ペイロードの読み取りを`Dynamic`に落とさず型付けできる。段階的で
偽陽性に安全な（FP-safe）設計である。安価なティアは限定的で、実コードで効く
ティアが最も難しい。

根拠: 2026-06-16の保護引き上げパイロット
（[`docs/design/20260616-act-on-coverage-skill.md`](../../design/20260616-act-on-coverage-skill/)
§「Multi-repo pilot」）。ここで本件は**最大の手に負えない残存保護ホール**
であった（kramdownの第2クラスタ、121〜165サイト。あらゆるパーサ／
テンプレート／リンターで再発する）。

## コンテキスト

Rubyに広く見られるシェイプ（shape）が**タグ付きユニオン**（tagged union、判別共用体とも）で
ある。1つのクラスが判別タグと、タグの値に型が依存するペイロードとを持つ。

```ruby
class Element                    # kramdown's AST node
  attr_accessor :type            # :text | :a | :ul | :smart_quote | :entity | …
  attr_accessor :value           # String if :text, Symbol if :smart_quote, nil if :ul, …
end
```

`value : String | Symbol | EntityObj | nil`である。`el.value.upcase`のような
ペイロード読み取りには、どちらのフラットな形式も役立たない。

- `value: untyped` → 保護なし（今日の挙動。レシーバーは`Dynamic`のまま）。
- `value: String | Symbol | nil` → Rigorは`upcase`を欠くアームに対して
  `possible nil`／undefined-methodを発火する。**コードが
  `case el.type; when :text`でガードしていてもである** ── 動作するコードに
  対する偽陽性だ。

そこで`untyped`が*安全な*下限として選ばれ、保護がその代償になる。これは
AST／ビジター／インタプリタのジャンルである ── パーサ（Prism、parser、Ripper
ラッパー）、テンプレート（haml／slim／kramdown／erb）、シリアライザ、
`AST::Node`を巡回するrubocop、ステートマシン ── そのすべてが
`case node.type; when …; node.<payload>`を共有する。Rigorはすでに**修正の
両半分を所有していながら、それらをつないでいない**。コンシューマー側は`==` /
`case`/`when`で`el.type`を`Constant[:text]`にナローイングし（コントロール
フロー解析。ADR-47の`eval_case_when_branches` / `falsey_scope`）、プロデューサー
側はインスタンスごとのメンバーレイアウトを持つ（ADR-48の`Type::DataInstance`
＋`Scope#data_member_layouts`）。欠けている接続詞が、**（tag ⇒ payload）の
マップ**と、判別子がナローイングされたときにペイロードをナローイングするルール
である。

## 決定

タグキー付きメンバーナローイングを、1つの要となる基準のもとで提案する。偽陽性に
安全な部分を研究水準の部分から独立して提供できるよう段階化する。

**基準（偽陽性の許容範囲。交渉の余地なし）:**多相なメンバーフィールドが兄弟の
判別子でナローイングされるのは、**（tag ⇒ payload）マップが健全な過大近似で
あるときに限る**。信頼できるマップがなければ、フィールドはフラットな／
`untyped`型を保ち、ナローイングは**何もしない**（no-op）。この機能は保護を
*追加する*だけであり、動作するコードの沈黙を診断へ変えることは決してない。これは
ADR-58の「宣言由来の型は診断の燃料ではない」を兄弟フィールドに適用したもので
ある。

実装される際、規範的なナローイングルールは
[`control-flow-analysis.md`](../../type-specification/control-flow-analysis/)
に着地し（スペックが拘束する）、宣言マップの文法は
[`rbs-extended.md`](../../type-specification/rbs-extended/)に着地する。この
ADRは根拠と段階化を記録する。

## ワーキングデシジョン

- **WD1 — 加算のみのコンシューマーフック（下限）**。ナローイングは、既存の
  判別子ナローイングの上に重ねられる純粋な*リファインメント*である。兄弟の
  判別子がすでにタグ定数にピン留めされたペイロード読み取りでは、マップされた
  アームに置き換える。マップがなければ何もしない。どんなマップも発火サーフェスを
  *広げる*ことはできない。このガードレールこそが、下位ティアを安全に提供できる
  理由のすべてである。
- **WD2 — Tier 1: 宣言されたバリアントマップ**。両側で記述されたRigor拡張 ──
  クラス上の
  `%a{rigor:v1:variant(tag: type){ text: {value: String}, smart_quote: {value: Symbol}, … }}`
  ── が、新しい**タグインデックス付きメンバーシェイプキャリア**（carrier。
  ADR-48のフラットな`DataInstance`レイアウトをタグごとのテーブルに一般化
  したもの）に供給する。コンシューマーフックは既存の`case`/`when`＋等価
  ナローイング（ADR-47）をそのまま再利用する。両側記述である ⇒ 構造上偽陽性に
  安全（ADR-35 / `conforms-to`の縁戚関係）。明示的な`case`イディオムを
  ただちに助ける。
- **WD3 — Tier 2: 推論されたマップ（研究水準、予算ゲート付き）**。判別子
  （クラス全体で`case`/`==`の対象になるSymbolフィールド）を自動検出し、構築
  サイトを相関させてタグごとのペイロードを推論する（`Element.new(:text, "x")`
  ⇒ `text ⇒ {value: String}`）。プログラム全体の相関は高コストであり、**動的な
  構築のもとでは健全でない**（`Element.new(t, v)`）ため、予算ゲートを付けて
  **きれいな過大近似のみを採用**し、そうでなければ（基準どおり）WD1の
  no-opにフォールバックする。
- **WD4 — Tier 3: ビジターディスパッチのタグ束縛（最難。価値があるのはここ）**。
  支配的な実イディオムはリテラルのチェックではなく、メソッド*名*がタグになって
  いる動的ディスパッチである。`send("convert_#{el.type}", el)` → `convert_text`
  の内側では、`el.value`は本体に`==`のない`String`である。ディスパッチ
  されたメソッド名からタグを束縛するには、ADR-16のマクロ／ディスパッチ基盤が
  必要だ。これこそ、Tier 1〜2がこのADRを動機づけたまさにそのリポジトリで期待を
  下回る理由である（kramdownのコンバーターは`send`ディスパッチされている）。
- **WD5 — ミューテーションの健全性**。 `value`/`type`は`attr_accessor`で
  ある。判別子またはナローイングされたペイロードフィールドへの書き込みは
  ナローイングを無効化する（コントロールフローのミューテーション効果モデル／
  ADR-56のライトバック）ので、`value`を更新しない`el.type = …`が古い
  ナローイング済みペイロードを残すことはありえない。

## 却下／先送りされた代替案

| 代替案 | 判定 |
| --- | --- |
| フラットなユニオン`value: String \| Symbol \| nil`を出力し、ナローイングされていないサイトで発火させる | **却下** ── ガードされたコードに偽陽性をでっち上げる。`untyped`が安全な下限であり続ける（基準）。 |
| ガードなしのプログラム全体マップ推論 | **予算の背後でWD3へ先送り** ── 動的な構築のもとでは健全でない。きれいな過大近似のみを採用する。 |
| 汎用の依存型／GADTシステム | **スコープ外** ── これは狭いタグキー付きの特殊ケース（1つの判別子 → 兄弟ペイロード）であり、完全な依存型付けではない。 |
| Tier 1を提供してパターンは解決したとする | **却下** ── 一般的なコンシューマーイディオムはディスパッチである（WD4）。Tier 1単独では明示的な`case`という少数派を助けるだけ。非対称性は述べよ、隠すな。 |

## 再評価のトリガー

需要ゲート付き。次の**いずれか**が起きたら見直す。（a）[ADR-58](../58-ivar-field-typing/)
が着地する ── ほとんどのペイロードフィールドはインスタンス変数なので、インスタンス
変数フィールドの型付けがナローイングに意味を持たせる前提条件である。（b）コーパスに
よって、明示的な`case`イディオムがWD2単独で見合うほど一般的だと示される。（c）
[ADR-63](../63-type-protection-coverage/)の保護カバレッジのプロダクト化により、この
ホールがユーザープロジェクト全体で頻繁に表面化する`add_a_type_here`ターゲットになる。

## 帰結

- **ポジティブ** ── Rubyのまるごと1ジャンル（AST／ビジター／コンバーター／
  インタプリタ）について、支配的な残存保護ホールを塞ぐ。基準により精度に対して
  加算的。
- **ネガティブ** ── Tier 1は作者によるアノテーション**かつ**スペック拡張を必要と
  する。一般的なイディオムは最難のティア（WD4）を必要とする。安価なティアは少数派の
  イディオムを助ける ── この非対称性が、本件をADR-58（アノテーションも新しい
  スペックサーフェスもなしに大きなクラスタを自動で塞ぐ）や動的クラスビルダーの
  フォールディングよりも**下位**にランクづける。
- **持ち越し** ── ADR-58（インスタンス変数ペイロード）の後に並べ、WD4ではADR-16
  （ディスパッチ）に頼る。新しいキャリアはADR-48のメンバーシェイプファミリーを
  拡張する。

## 他のADRとの関係

- **ADR-48** ── プロデューサー基盤。タグインデックス付きキャリアは、そのフラットな
  `DataInstance`レイアウト＋`data_member_layouts`サイドテーブルを一般化する。
- **ADR-47** ── ペイロードフックがそのまま乗る、コンシューマーナローイング
  （`case`/`when`の到達可能性）。
- **ADR-58** ── 前提条件: ナローイングが意味を持つ前に、インスタンス変数に裏打ち
  されたペイロードフィールドが型付けされていなければならない。
- **ADR-16** ── WD4（ビジター名のタグ束縛）が必要とするディスパッチ基盤。
- **ADR-20** ── 直和型（sum type）キャリアの縁戚関係（タグでパラメータ化された値）。
- **ADR-63** ── 本件を最上位の手に負えないホールとして表面化させた、保護カバレッジ
  パイロット。
