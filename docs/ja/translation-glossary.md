## 用語紹介の表記ルール

専門用語は**漢語・カタカナを問わず**、各ページで**初めて登場するとき**に
英語原語を一度併記する。これは読者が原語で検索・参照できるようにするため
のサイト全体ポリシーである。別訳語がある場合は併せて添える。

形式: `日本語訳（english、別訳語とも）`

例:
- 名前的型（nominal type、公称型とも）
- 漸進的型付け（gradual typing）
- 部分型（subtype）
- キャリア（carrier）

2回目以降は括弧注記なしで日本語訳（またはカタカナ）のみ使う。併記は
ページ単位の初出に付ける（同一ページで繰り返さない）。

### ruby による併記（discipline と gradual ファミリー）

一部の用語は、パーレン併記の代わりに ruby で原語を併記する。形式は
`<ruby>規律<rp>（</rp><rt>discipline</rt><rp>）</rp></ruby>`（`<rp>` はruby
非対応環境での `規律（discipline）` フォールバック）。Astro標準のMarkdownが
生HTMLをそのまま通すので `.md` のまま描画できる（config変更や `.mdx` 化は不要）。

対象（閉じたリスト。なし崩しで増やさない）:

- **discipline → 規律**（単一語ルビ）
- **gradual ファミリーの複合語**（グループルビ）── `漸進的型付け`(gradual
  typing)・`漸進的一貫性`(gradual consistency)・`漸進的保証`(gradual
  guarantee)・`漸進的境界`(gradual boundary)。複合語まるごとをベースにする。

適用ルール:

1. **ページ初出の地の文1回だけ**に付け、2回目以降は素の日本語。gradual は
   複合語ごとに別々の用語として初出判定する（語数が増えるので密度に注意）。
2. **語根 `漸進的`（gradual）単独はパーレン据え置き**（`漸進的（gradual）`）。
   ルビは複合語にだけ。語根と複合語で表記が割れるのは許容する。
3. **リンクテキスト・見出し・コード・フェンス・注記同居はルビ対象外**（素の
   日本語かパーレンのまま）。ruby を `[...]` リンク内に入れない。`gradual
   consistency、`consistent(A, B)`と書く` のような**注記付きパーレンは rt に
   載らないのでパーレン維持**。
4. **太字にしたいときは `**…**` でなく `<strong>…</strong>` を使う**。
   CommonMark の emphasis は左右フランキング規則に従い、`</ruby>` の `>` や
   `<ruby>` の `<` が `**` に隣接すると規則違反で**太字が成立せず `**` がリテラル
   で残る**。生HTMLの `<strong>` なら回避できる（既に ruby で生HTMLを使って
   いるので一貫）。
5. ruby は生HTMLなので、`normalize-ja-typography.mjs` を**ファイル全体には
   当てない**（テーブルセル内の `\|` エスケープを外す等、無関係な巻き込み変更が
   混ざる）。タイポグラフィ調整が要るときは手で当該箇所のみ直す。

**typography 実測メモ**: 「短い漢語＋長い英単語は間延びする」のは*単語ルビ*の
話（`規律`32px に `discipline` 中身42px ＝ rt がはみ出し約31%伸長）。複合語の
*グループルビ*なら、英語句の自然幅がベース幅に収まり**伸長0%**（`漸進的型付け`
96px ＞ `gradual typing` 60px、`漸進的一貫性` 96px ＞ `gradual consistency`
81px、`漸進的保証`/`漸進的境界` 80px ＞ 74/73px）。つまり複合語ルビはむしろ
単語ルビより綺麗で、ベース幅 ≥ rt幅 が成り立つ限り安全。

リストを上記2件（discipline・gradual ファミリー）に**閉じておく**。`normative`
ほか他の用語へは広げない（理由は normative 行・本節の密度規律を参照）。

#### 何に付けるか ── 正の warrant と負のフィルタ

**正の warrant（これがあるとルビ／英語併記が earn される）:**

1. **対比的弁別** ── 紛らわしい漢語が別々の英原語に対応するとき、英単語の
   別れ目を可視化する。例: `規律`(discipline) と `規範的`(normative) は
   ともに `規〜`／「規範・秩序」系で漢字面からは英語の別れ目が見えないため、
   `規律` に discipline をルビで貼って別物性を示す。**条件: 対比は両側を最低
   一度ずつ原語に pin して初めて効く**（現状 normative は未gloss で片側のみ＝
   弁別が未成立。完成させるなら `規範的（normative）` の paren 併記を主たる
   初出に1回足す。ただし normative は偏在する状態語なのでルビにはせず paren で
   十分＝非対称が妥当）。
2. **文献アンカー** ── 特定箇所で説明・参照される specialist 概念で、英語名が
   後続の検索・文献参照の足場になる（`漸進的型付け`←gradual typing）。

**負のフィルタ／決定手順。判定軸は文字種（カタカナ/漢字）でも音訳/意訳でも
なく、対象読者（プログラマー）にとっての定着度・明快さ・誤解リスク。「漢訳が
辞書的に正しいか」ではなく「読者に自然・明快か、誤解を生まないか」で決める:**

1. その語の**カタカナ（loanword）形が対象読者に定着・自然**か？
   - **はい** → **その形のまま。漢訳もルビもしない**。漢語へ寄せると不自然で、
     誤解も生みうる。定着済みなので英語併記も既習の反復＝ノイズ。
     - 例: メソッド／クラス／オブジェクト／モジュール／ミックスイン ── 日常語
       では未定着でも、プログラミング／ビジネス語として定着＝この読者には自然
       （`<ruby>方法<rt>メソッド</rt></ruby>` は不可。rt がカタカナで日本語↔
       日本語、かつ「方法」は領域違い）。
     - 例: ディレクティブ ── 指令も一般には未定着だが、この読者にはディレクティブ
       が自然。加えて「指令」は「命令」の一般語と**混同される誤解リスク**を持つ
       （`<ruby>指令<rt>directive</rt></ruby>` は「前処理指令」がJIS定訳でも不可。
       これは「主表記でない」とは独立した、形そのものの欠点）。
   - **いいえ** → 2へ。
2. **漢訳のほうが明快**か？
   - **はい** → 漢訳を主表記にする。さらに**英単語がニュアンス追跡／対比弁別の
     価値**を持つなら、ルビで原語を併記。価値が無ければ漢訳のみ（ルビ不要）。
     - 例: 規律 ── ディシプリンは英語形でもカタカナ形でもこの読者に未定着、規律
       のほうが明快。英語 discipline はニュアンス追跡＋normative との対比弁別の
       価値を持つのでルビ併記（`<ruby>規律<rt>discipline</rt></ruby>`）。
   - **いいえ** → 漢訳のみ。

定着度はコーパス／読者相対（前処理指令はJIS定訳だが、プログラマー読者には
ディレクティブが自然で、指令は誤解リスクを伴う）。rt は常に英語原語（カタカナ
読み・コードを入れない）。

#### 密度規律 ── 「ルビ＝悪」ではなく「無差別密度＝悪」

ルビは多用すると*シグナル*をやめて*装飾*になる。参考に docs.docker.jp は
用語集ページ（`glossary.html`）で `<ruby>` を約108個（見出し語ほぼ全部＋本文
の再出にも）使い濫用気味だが、同サイトでも地の文中心の `overview.html` では
4個しか使っていない。**走る文章にはルビを撒かない**のが彼ら自身の規律であり、
高密度が許されるのは「語そのものが内容」である専用の用語集ページに限る。

当サイトの方針はこの overview 側（控えめ）に倣う。具体的には:

- **全出現に振らない**（first-occurrence で十分）。これが濫用の核。
- 高密度ルビが正当化されるのは、将来**読者向けの専用用語集ページ**を作る場合
  のみ。`docs/ja/translation-glossary.md`（このファイル）はエージェント用の
  内部資料で読者には出ないため、その出番は今はない。

#### 静的展開 vs トグル ── 変曲点

「Markdownで生 `<ruby>` 直書き（静的・常時表示）」と「属性に埋めてボタンで
出し入れ（読者トグル）」は別々の2軸（ソース表現／実行時挙動）の話。判断:

- **読者トグルが報われるのは高密度のときだけ**（デフォルト非表示で地の文を
  汚さず、学習したい読者だけ ON）。低密度（first-occurrence）の現状ではトグル
  は隠す対象がほぼ無く過剰設計。Rigorの読者は技術者で、ルビは*読みの補助*では
  なく**原語の検索アンカー**なので出し入れ需要も小さい。
- したがって**今は静的 `<ruby>` 直書きのまま**（5箇所のためにパイプラインを
  建てない＝YAGNI）。
- **避けるべきは「生 `<ruby>` ＋ トグル」**（脆いソース＊かつ＊実行時機構の
  最悪の組み合わせ）。
- **変曲点**: 将来「多くの用語に振りたい」「読者制御が欲しい」となったら、まず
  ソース表現だけを動かす ── 生HTMLを意味マーカー（`<Term ja="規律"
  en="discipline"/>` 等）に置換し、ビルド時に静的 `<ruby>` へ展開する。これで
  生HTMLの脆さ・normalizer巻き込み・`<rt>` のtypo放置リスクが消え、用語表から
  初出を自動ルビ化する余地が生まれ、トグル（実行時軸）は後から分岐1つで足せる。
  順序は「**意味マーカー化（常時表示・JSなし）→ density を上げる決断と同時に
  トグルJS**」。トグルOFF時も英語が静的HTMLに残る（`data-en` 属性等）よう実装
  しないと「原語で検索」の利点が消える点に注意。

## カタカナ長音符（ー）の表記規則

外来語末尾の長音符「ー」の有無は、**対象読者（プログラマー）にとっての定着度・
明快さ・誤解リスク**で決める（文字数でも音訳/意訳でもない）。

1. **スコープ**：落とす対象は語尾 -er/-or/-ar/-ure 由来の `ー` のみ。-ow
   （ワークフロー）・-ough（ウォークスルー）等と語中の `ー` は常に伸ばす。
2. **デフォルトは伸ばす**。落とすのは**短縮形が工学慣用として定着している語だけ**
   （パラメータ・ハンドラ・クラスタ・ポインタ・フィルタ・コンパイラ 等）。
3. **-ry/-ly/-ory は全部伸ばす**（エントリー・サマリー・カテゴリー 等）。
4. 複合語は run 内の全要素に適用。コード・フェンス内は対象外。

全72ペアの確定リスト（DROP/KEEP）と漢訳判断（評価器・言語サーバー）は
[カタカナ長音符 表記決定台帳](katakana-longvowel-ledger.md)に記録。一括適用は
1570箇所/125ファイル（2026-06 実施）。新語はこの規則で判定し、迷う -er/-or/-ar は
台帳に追記する。

**強制**：この長音規則と確定済みの1:1漢訳は `scripts/normalize-ja-terms.mjs`
（冪等・コード保護・ja-nativeスキップ）が機械的に適用する。翻訳後に
`normalize-ja-typography.mjs` と並べて実行（sync-upstream SKILL 参照）。新たに
確定した語幹/訳語はスクリプトの `DROP`/`KEEP`/`TERMS` リストと台帳の両方に追記。
ルビ／パーレン併記の配置と意味ズレ語の判断はスクリプト対象外（翻訳時に手当て）。

## 翻訳対象外（upstream所有のja-nativeページ）

フロントマターに `sourceLanguage: "ja"` を持つページ（`notes/` 配下の
一部など）は、upstream/rigorがja-nativeで所有し、`pnpm sync:docs`
（`prebuild`で自動実行）がupstreamから上書き再生成する。当リポジトリの
翻訳対象ではないため、**用語統一・英語併記・表記正規化の対象にしない**。
編集してもビルド時に巻き戻る。該当ファイルは
`grep -rl 'sourceLanguage: "ja"' src/content/docs/ja` で一覧できる
（2026-05時点で `notes/` の6ファイル）。
